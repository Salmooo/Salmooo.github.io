14.1 推理系统概述

14.1.1 推理系统 vs 推理引擎

在深入探讨之前，首先需要厘清两个核心概念：

概念 定义 类比
推理系统 专门部署 AI 模型、执行推理预测任务的完整系统，类似于传统 Web 服务 整套 Web 服务架构
推理引擎 推理系统中的核心组件，负责 AI 模型的加载与执行 Web 服务中的 Runtime 引擎

推理 是指利用训练好的模型结构和参数，对新的小批量数据进行一次前向传播，得到预测输出的过程。与训练不同，推理不涉及梯度的反向传播和权重更新。

推理系统支撑推理任务的三大支柱：

  1. 模型小型化（第 15 章）：设计更高效的网络计算方式，在减少参数量的同时保持精度。重点关注 CNN 轻量化模型（MobileNets、SqueezeNet 等）和 Transformer 小型化策略。
  2. 离线优化压缩（第 16 章）：通过低比特量化、二值化网络、模型剪枝、知识蒸馏等手段对模型"瘦身"，使其更轻便、更利于执行。
  3. 在线部署和优化：模型格式转换 → 计算图优化 → Kernel 算子优化，三层递进，确保模型在不同硬件上高效运行。

14.1.2 推理应用场景

  • 人脸 Landmark 识别：快速捕捉脸部轮廓、五官位置等关键面部信息，应用于美妆试妆、虚拟形象创建等
  • 人脸检测与手势识别：如华为 HMS Core 实现的手势控制游戏
  • 智能客服与对话：智能客服理解用户问题、对话引导、意图识别

14.1.3 推理系统思考点

实际维护推理系统需要全面考虑：API 接口设计、数据质量保障、低时延网络传输优化、硬件加速资源利用、负载均衡与弹性扩容、新模型平滑上线与 AB 测试等。

14.2 推理系统介绍

14.2.1 推理场景的重点

训练 vs 推理的核心差异：

维度 训练 推理
目标 收敛到指定准确度 实时响应用户请求
批量大小 大批量（追求吞吐量） 小批量（追求低时延）
运行时长 数小时到数天 7×24h 长期服务
资源约束 数据中心充裕资源 更严格（内存、功耗）
部署设备 服务器集群为主 云/边/端多样化设备

推理系统的典型架构如图 14.2.1 所示：训练后的模型存储于文件系统 → 通过服务系统部署上线 → 模型加载到内存 → 版本管理 → 对外暴露 HTTP/gRPC 接口。

14.2.2 优化目标和约束

推理系统面临 AI 框架多样性和硬件多样化的双重挑战。六大优化目标如下：

优化目标 原因 相关策略
灵活性 支持多种框架 模型转换工具（ONNX）、多语言接口
时延 减少用户等待时间 模型压缩/剪枝/量化、预测性加载
吞吐量 应对大量服务请求 多线程/多进程、异步处理、缓存
高效率 降低服务成本 DVFS、低功耗模式、高效算法
可扩展性 应对增长需求 Kubernetes、弹性扩展、负载均衡
可靠性 保证 SLA 达标 多副本跨地域部署、故障转移、限流降级

14.2.3 推理系统与推理引擎

推理系统组件与架构（图 14.2.3）：涵盖请求/响应序列化、请求调度、推理引擎调用、模型版本管理、健康监控、边缘推理芯片与代码编译等模块。推理系统需要高效处理图像、文本、音频等非结构化数据。

推理引擎架构（图 14.2.4）：整体分为四层 —— API 接口层（多语言统一接口）→ 模型压缩与优化层（格式转换、模型压缩、端侧学习、图优化）→ Runtime 调度层（动态 Batch、异构执行、大小核调度）→ Kernel 硬件优化层（CANN、NEON、CUDA、Vulkan）。

14.3 推理流程全景

14.3.1 云侧部署 vs 边缘侧部署

维度 云侧部署 边缘侧部署
算力 强大（TFLOPS 可水平扩展） 有限，水平扩展性差
时延 主要来自网络传输 + 计算 本地计算，实时响应
网络依赖 强依赖 弱依赖
能耗 百瓦以上 几十瓦，能耗比高
系统架构 开放，高度集中 封闭，架构分散
多样性 CPU/GPU/NPU 标准化 多样性芯片架构（SoC）

14.3.2 五种边缘部署和推理方式

  1. 纯边缘设备计算：模型完全在手机、耳机、手环上推理。依赖轻量化模型（MobileNets）、模型压缩和专用 AI 芯片
  2. 安全计算（卸载到云侧）：模型部署于数据中心，边缘侧通过安全通信发送请求，适合端侧无法部署的大模型
  3. 边缘设备 + 云服务器（模型切片）：将部分层在设备端计算，部分放在云侧。前几层计算后中间结果大幅减少，可降低网络开销
  4. 分布式计算：将 AI 任务分解，在多个辅助边缘设备上并行处理（如 DeepThings、MoDNN）
  5. 跨设备卸载：不同层在不同设备/服务器上计算，需注意中间结果传递的时延

14.4 推理系统架构

14.4.1 推理、部署、服务化三层

推理系统的三个核心环节：

  1. 推理（Inference）：模型对新数据执行前向计算生成输出。优化方向：模型压缩（量化/剪枝/蒸馏）+ 硬件加速（GPU/NPU）
  2. 部署（Deployment）：将模型从开发环境迁移到生产环境。关键步骤：移植 → 压缩 → 加速 → 监控维护
  3. 服务化（Serving）：将模型封装为可调用服务。三种方式：SDK 封装 → 应用集成 → Web 服务（HTTP/gRPC）

14.4.2 英伟达 Triton 推理服务

Triton Inference Server 是英伟达推出的高性能开源推理框架，具有六大特点：高性能、可扩展性（水平/垂直扩展）、多框架支持（TensorFlow/PyTorch/ONNX）、模型优化（集成 TensorRT）、灵活部署（云/私有云/边缘）、安全性（安全传输与访问控制）。

14.4.3 模型生命周期管理

金丝雀策略（Canary Deployment）：逐步将一小部分流量导向新版本，保持大部分流量在稳定旧版本上。通过路由系统（用户 ID 哈希、地理位置或随机分配）分发请求，实时监控新旧版本的表现指标（精度、响应时间、资源消耗），逐步增加新模型流量直至完全替换。

回滚策略：一旦检测到新版本严重缺陷，立即触发回滚流程。关键是维护清晰的模型版本历史记录、快速定位和部署旧版本、平滑切换（最小化服务中断）。

14.5 推理引擎架构

14.5.1 推理引擎四大特点

特点 核心要求 具体体现
轻量 资源占用少、体积小、启动快 零依赖设计、FP16/INT8 压缩量化
通用 多模型/多平台支持 兼容 TF/PyTorch/MindSpore/ONNX、跨平台（iOS/Android/Linux）
易用 简化部署、降低门槛 丰富算子库（Numpy 映射）、CV/NLP 专用模块、跨平台训练
高效 高性能推理 深度适配硬件、SIMD/手写汇编、多精度计算（FP32/FP16/INT8）

14.5.2 三大技术挑战

  1. 需求复杂性与程序大小：如何在有限的程序体积内支持成千上万种算子和多样化的模型结构？

    • 策略：模块化与插件化设计、算子优化与融合、动态编译与代码生成

  2. 算力需求与资源碎片化：从高性能服务器到低功耗移动设备，算力差异巨大

    • 策略：模型分层与多级缓存、自适应推理、异构计算整合、边缘计算与分布式推理

  3. 执行效率与模型精度：量化/剪枝等压缩手段可能以精度损失为代价

    • 策略：知识蒸馏、量化微调、模型剪枝与稀疏化、缓存与预计算

14.5.3 整体架构

推理引擎整体架构分为两个阶段：

优化阶段（离线）：

  • 模型转换工具：模型格式转换 + 计算图优化（算子融合、布局转换、算子替换、内存优化）
  • 模型压缩：量化、知识蒸馏、剪枝、二值化
  • 端侧学习：增量学习和联邦学习，支持模型在部署后持续适应新数据
  • IR（中间表示）：统一的模型表达，提供丰富的优化空间，“一次转换、处处运行”

运行阶段（在线）：

  • 调度层（Runtime）：动态 Batch、异构执行、内存管理、大小核调度、多副本并行
  • 执行层（Kernel）：算子优化（融合/量化/稀疏）、算子执行（CPU SIMD / GPU CUDA）、算子调度（异构调度 / 流水线调度）

14.5.4 推理流程

完整推理流程：不同 AI 框架（MindSpore/TF/PyTorch/PaddlePaddle）的模型 → 模型转换工具进行格式转换 → 模型压缩 → 环境准备（大小核调度、模型文档获取）→ 编译生成推理进程 → Runtime 在线执行推理。

14.5.5 开发推理程序

开发推理程序的标准六步：

  1. 模型转换：将训练模型转换为推理引擎统一格式
  2. 推理配置管理：设置模型路径、运行设备（CPU/GPU）、图优化开关、算子融合、内存优化等
  3. 创建推理引擎对象：传递配置，管理整个推理过程
  4. 准备输入数据：预处理原始数据（减均值/缩放）→ 获取输入 Tensor → 复制数据
  5. 执行推理：调用 Run 方法启动模型推理
  6. 获取结果并后处理：获取输出 Tensor → 后处理（如目标检测中裁剪检测框）

14.6 昇腾推理引擎 MindIE

14.6.1 MindIE 基本介绍

MindIE（Mind Inference Engine） 是华为昇腾针对 AI 全场景业务的推理加速套件，通过分层开放 AI 能力，使能百模千态，释放昇腾硬件设备算力。

关键能力：

  • 兼容 Triton/TGI/vLLM 等业界标准 RPC 接口，实现小时级应用部署
  • 针对 LLM（Transformer）和文生图（SD 模型）提供加速参考代码与预置模型
  • 少量代码实现训练向推理的平滑迁移
  • 支持请求并发调度和模型多实例并发调度

MindIE 包含三大组件：

组件 定位 核心能力
MindIE-Service 服务化部署 对接主流推理框架接口（兼容 Triton/OpenAI），支持 Continuous Batching 和 PagedAttention
MindIE-Torch PyTorch 推理加速 支持 TorchScript 和 ExportedProgram 编译优化，混合精度推理，异步推理
MindIE-RT 推理运行时 集成 Transformer 加速库 ATB，Parser 直接导入 ONNX 模型，丰富编译/运行时优化

14.6.2 关键功能特性

  • 服务化部署：大模型服务化快速部署，标准化昇腾服务化接口，兼容 Triton/OpenAI/TGI/vLLM
  • 大模型推理：PyTorch 模型迁移（全流程逐级能力开放）、TorchScript 支持（动态输入/静态输入）、ExportedProgram 支持
  • 推理运行时：集成 Transformer 高性能算子加速库 ATB，支持动态输入推理和框架模型解析

14.7 昇腾计算语言 AscendCL

14.7.1 AscendCL 基本介绍

AscendCL（Ascend Computing Language） 是一套用于在昇腾平台上开发深度神经网络应用的 API 库，提供运行资源管理、内存管理、模型加载与执行、算子加载与执行、媒体数据处理等 API。

三大优势：

  1. 高度抽象：算子编译/加载/执行的 API 归一，大幅减少 API 数量，降低复杂度
  2. 向后兼容：基于旧版本编译的程序可在新版本上运行
  3. 零感知芯片：一套接口实现应用代码统一，多款昇腾 AI 处理器无差异

三种应用场景：

  • 直接开发应用：调用 AscendCL 接口开发图片分类、目标识别等应用
  • 供第三方框架调用：通过第三方框架调用 AscendCL，使用昇腾计算能力
  • 供第三方开发 lib 库：封装 AscendCL 实现运行管理和资源管理能力

14.7.2 核心概念：Device、Context、Stream

概念 描述
Device 安装了昇腾 AI 处理器的硬件设备，通过 PCIe 与 Host 连接，提供 NN 计算能力。多个 Device 之间内存资源不共享
Context 管理所有对象（Stream/Event/设备内存等）生命周期的容器。可隐式创建（默认 Context）或显式创建（推荐）
Stream Device 上的执行流，维护异步操作的执行顺序，确保按代码顺序在 Device 上执行。支持默认 Stream 和显式 Stream
Event 用于同步不同 Stream 之间的任务依赖关系
AIPP AI 图像预处理，在 AI Core 上完成色域转换、图像归一化和抠图
动态 Batch/动态分辨率 支持推理时 Batch Size 或分辨率动态可变
EP 模式 vs RC 模式 PCIe 从模式（EP）和主模式（RC），决定处理器的工作方式

关键关系：

  • 一个用户线程绑定一个 Context，所有 Device 资源调度必须基于 Context
  • 一个线程中可以创建多个 Stream，不同 Stream 上的计算任务可以并行执行
  • 显式创建 Context 和 Stream 是推荐做法，默认 Context/Stream 适用于简单应用
  • 多线程 + 单 Stream vs 单线程 + 多 Stream，一般后者性能略好（少了线程调度开销）

14.7.3 AscendCL 基本开发流程

AscendCL 开发 AI 应用的五大步骤：

  1. AscendCL 初始化：调用 aclInit 接口初始化 AscendCL
  2. 运行管理资源申请：申请 Device、Context、Stream 等运行管理资源
  3. 具体计算（三种组合）：
    • 模型推理：模型加载 →（可选）媒体数据处理 → 模型执行 →（可选）数据后处理 → 模型卸载
    • 算子调用：通过 AscendCL 接口直接加载并执行单个算子，省去模型构建/训练，相对轻量级
    • 媒体数据处理：JPEG 编解码、视频解码、抠图/缩放/格式转换等
  4. 运行管理资源释放：依次释放 Stream、Context、Device
  5. AscendCL 去初始化：调用 aclFinalize 接口

小结

第 14 章构建了推理系统的完整知识体系：

  • 宏观视角：推理系统是 AI 从训练到落地的"最后一公里"，涉及推理/部署/服务化三个环节
  • 架构设计：推理系统需平衡灵活性、时延、吞吐量、效率、可扩展性、可靠性六大优化目标；推理引擎以轻量/通用/易用/高效为设计哲学
  • 部署形态：云侧强算力高吞吐 vs 边缘侧低时延弱依赖，5 种边缘部署方式灵活组合
  • 模型生命周期：金丝雀策略、回滚策略保障模型迭代的稳定性和可靠性
  • 引擎实战：MindIE 提供了 MindIE-Service/Torch/RT 三大组件覆盖服务化部署、训练迁移、推理运行时全流程；AscendCL 通过统一的 API 框架简化了 Device/Context/Stream 资源管理和模型推理开发

核心启示：一个高效的推理系统不仅仅是"把模型跑起来"，而是需要在资源管理、性能优化、服务可靠性等多个维度进行系统性设计，才能真正释放 AI 的商业价值。
