AI系统：第17章 模型转换

《AI系统：原理与架构》学习笔记 — 深入理解模型转换：从推理引擎架构、模型序列化与Protobuf/FlatBuffers，到自定义计算图IR设计，再到ONNX规范式转换与模型转换通用流程

5 minute read

本章概览

前面几章我们聊了 AI 编译器（前端优化、后端优化），但都有一个前提：模型已经在一个框架里了。现实是，你用 PyTorch 训练好了一个图像识别模型，但生产环境用的是 MindIE 做推理——两个框架的模型格式完全不同。谁来打通这个"巴别塔"？ 答案就是本章的主角——模型转换。

模型转换的核心任务是实现模型在不同框架之间的流转。随着深度学习技术发展，训练框架（PyTorch、TensorFlow 等）和推理框架（TensorRT、MindIE 等）的功能逐渐分化：前者侧重易用性和算法创新，后者侧重极致硬件优化。由于各框架内部的模型表示方式各不相同，模型转换成为连接两者的必不可少的环节。

本章将从推理引擎架构出发，依次讲解模型序列化格式（Protobuf / FlatBuffers）、自定义计算图 IR 的设计方法、ONNX 规范式转换，最后梳理模型转换的通用流程。

17.1 引言

17.1.1 推理引擎

推理引擎是推理系统中用来完成推理功能的模块，分为 2 个主要阶段：

阶段 职责 组成
优化阶段 模型转换工具，由模型转换和图优化构成 模型压缩工具、端侧学习等
运行阶段 负责 AI 模型的加载与执行 调度层 + 执行层

模型转换工具模块又分为两个部分：

  1. 模型格式转换：将不同框架的模型转换为推理引擎所需的中间表示（IR）或其他格式
  2. 计算图优化：通过图的等价变换化简计算图，降低计算复杂度或内存开销

转换模块的四大挑战

① AI 框架算子的统一

神经网络模型包含众多算子，重合度高但不完全相同。推理引擎需要用有限的算子去实现不同框架的算子。例如 PyTorch 的 Padding 和 TensorFlow 的 Padding，执行方式和方向不同；PyTorch 的 Conv 可任意指定 Padding 步长，而 TensorFlow 的 Conv 不可指定。

解决方案：让推理引擎定义属于自己的算子定义和格式，来对接不同 AI 框架的算子层。

② 支持不同框架的模型文件格式

主流框架导出的模型文件格式各不相同：

AI框架 模型文件格式
PyTorch .pt, .pth
MindSpore .ckpt, .mindir, .air, .onnx
PaddlePaddle .pdparams, .pdopt, .pdmodel
TensorFlow .pb (Protocol Buffers), .h5 (HDF5)
Keras .h5, .keras

需要推理引擎支持自定义计算图 IR，将不同框架训练出的模型文件转换成统一的中间表示。

③ 支持主流网络结构

CNN、RNN、Transformer 等不同网络结构各有擅长领域。推理引擎需要通过丰富的 Demo 和 Benchmark，保证对主流网络结构的支持性和可用性。

④ 支持各类输入输出

神经网络中存在多输入多输出、任意维度输入输出、动态输入（运行时形状可变）、带控制流模型等复杂情况。推理引擎需要可扩展性和对动态形状（Dynamic Shape）的支持。

例如，下面是一个动态输入 ONNX 模型的测试结果：

优化模块的四大冗余

图优化的目标是消除以下四类冗余：

冗余类型 含义 优化手段
结构冗余 无效计算节点、重复计算子图、相同结构模块 算子融合、公共子表达式消除
精度冗余 使用 FP32 但实际 FP16/INT8 就够 降低数值精度（量化）
算法冗余 Kernel 层面的实现算法本身存在计算冗余 算法简化、算子替换
读写冗余 不必要的内存读写、低效的内存访问模式 内存布局优化、数据格式转换

17.1.2 转换模块架构

转换模块（Converter）由格式转换（Graph Converter） 和图优化（Graph Optimize） 两部分构成。

格式转换

将不同 AI 框架的模型转换成统一的中间表示（IR），后续优化都基于这种统一的 IR 进行。针对 MindSpore 有 MindSpore Converter，针对 PyTorch 有 ONNX Converter——通过不同的转换模块，把不同的 AI 框架统一转换成推理引擎的 IR。

图优化

四大核心技术：

  • 算子融合：将连续算子合并为一个，减少计算和内存访问开销。如 z = ReLU(Conv(x, w))
  • 算子替换：用对在线部署更友好的算子替换。如标准卷积 → 深度可分离卷积
  • 布局调整：重新组织张量存储方式。如 NHWC → NCHW，适配 GPU 高效计算
  • 内存分配：分析计算图峰值内存使用，必要时插入 CPU-GPU 内存交换

离线模块流程（三段式优化）

优化模块分成三段：

阶段 主要工作 典型方法
预优化 语法检查和初步优化 公共子表达式消除、死代码消除、代数简化
优化 算子级优化 算子融合、算子替换、常量折叠
后优化 内存和数据访问优化 数据格式转换、内存布局计算、重复算子合并

17.2 推理文件格式

17.2.1 模型序列化

训练好的模型在内存中是暂时的，需要将其从内存移动到硬盘上永久存储——这个过程叫做序列化与反序列化。

  • 模型序列化：模型部署第一步，把训练好的模型保存下来
  • 模型反序列化：将硬盘中的二进制数据恢复到内存中，得到模型对象

序列化分类

分类 示例 特点
跨平台跨语言通用 XML, JSON, Protobuf, FlatBuffers Protobuf 使用最广泛，ONNX 即基于此
模型自定义 TensorFlow Checkpoint, PyTorch 模型文件 文本或二进制格式，针对特定框架优化
语言级通用 Python pickle (.pkl) 仅限特定语言环境
用户自定义 自定义二进制格式 灵活但维护成本高，优先使用通用方案

ONNX 使用 Protobuf 存储神经网络权重信息；CoreML（苹果）同样使用 Protobuf 在 iOS/macOS 上实现高效推理。

PyTorch 模型序列化的两种方式

① PyTorch 内部格式（torch.save / torch.load）

# 保存模型状态字典
torch.save(model.state_dict(), PATH)
# 加载

state_dict 是一个 Python 字典对象，将每层映射到参数张量。只有具有可学习参数的层（卷积层、线性层等）和注册缓冲区（BatchNorm 的 running_mean）才在 state_dict 中有条目。此方法只存储参数/权重，不存储计算图结构。

② ONNX 导出（torch.onnx.export）

1
2

torch.onnx.export(model, dummy_input, "alexnet.onnx",
    verbose=True, input_names=input_names, output_names=output_names)

这种方法不仅保存模型参数，还包括完整计算图信息，支持跨框架、跨平台部署。

17.2.2 目标文件格式

Protobuf（Protocol Buffers）

谷歌开发的高效、语言无关的序列化机制，特点是语言无关、平台无关，比 XML 更小、更快、更简单。

TLV 编码模式：Tag-Length-Value

1

Tag = (Field Number << 3) | Wire Type
  • 编码过程：构建消息结构 → 逐个字段编码（tag + length + value）→ 组合成二进制流
  • 解码过程：Root Message 由多个 TLV field 组成，逐个解析。通过 field length 偏移找到下一个 field；嵌套 message 递归解析

Protobuf 字段规则：

  • required：必须含有一个该字段
  • optional：可有 0 或 1 个值
  • repeated：可重复任意多次（相当于 List）

FlatBuffers

开源、跨平台、高效的序列化工具库。核心优势：不需要解析，直接通过序列化后的二进制 buffer 即可完成数据访问。 特点：内存高效、速度快、生成代码量小、支持强类型检测。

17.3 自定义计算图

17.3.1 计算图定义回顾

转换模块架构 = 格式转换 + 图优化，中间通过 IR（中间表示） 承载，将不同 AI 框架对接到同一个 IR。有了 IR，就可以做各种图优化工作。

计算图 = 张量（Tensor） + 算子（Operator）

  • 张量（Tensor）：多维数据。标量（零秩）、向量（一秩）、RGB 图像（三秩）。元素类型可为 INT、FLOAT、STRING 等。

  • 算子（Operator）：神经网络的基本计算单元。N 个输入张量 → M 个输出张量。包括数学运算、数据变换、条件控制等。

AI 框架计算图 vs 推理引擎计算图

维度 AI 框架计算图 推理引擎计算图
计算图组成 算子 + 张量 + 控制流 算子 + 张量 + 控制流
前向和反向 前向 + 反向 仅前向
动静态 动态图 + 静态图（可动静统一） 以静态图为主
分布式并行 支持数据并行、张量并行、流水并行 以单卡推理服务为主
使用场景 训练场景：科研创新、模型训练和微调 推理场景：模型工业级部署、对外服务

核心差异：推理引擎聚焦前向传播，使用静态图（更易优化），以单卡高性能推理为目标。

// 张量数据类型
enum DataType : int {
    DT_INVALID = 0, DT_FLOAT = 1, DT_DOUBLE = 2,
    DT_INT32 = 3, DT_UINT8 = 4, DT_INT16 = 5, DT_INT8 = 6,

// 张量数据排布格式
enum DATA_FORMAT : byte {
    ND, NCHW, NHWC, NC4HW4, NC1HWC0, UNKNOWN,
}

// 张量定义
table Tensor {
    dims: [int];
    dataFormat: DATA_FORMAT;
    dataType: DataType = DT_FLOAT;
}

常用数据排布格式：

  • NCHW：批次-Batch → 通道-Channel → 高度-Height → 宽度-Width（GPU 更高效）
  • NHWC：批次 → 高度 → 宽度 → 通道
  • NC4HW4：通道和宽度扩展为 4 的倍数（GPU 加速器使用）
  • NC1HWC0：Ascend 芯片专用排布格式

17.4 模型转换流程

17.4.1 模型转换设计思路

模型转换主要有两种方式：

方式 流程 举例
直接转换 AI 框架 → 推理引擎 IR（一步到位） MindSpore Converter 直接将 MindSpore 格式转为推理引擎 IR
规范式转换 AI 框架 → 中间开放格式 → 推理引擎 IR（两步） PyTorch → ONNX → 推理引擎 IR

直接转换的三步流程：

  1. 内容读取：识别模型文件中的张量类型/格式、算子类型和参数、计算图结构
  2. 格式转换：在代码层面翻译成推理引擎支持的格式，复杂算子封装专用转换函数
  3. 模型保存：在推理引擎下保存模型文件

技术难点：算子实现差异、张量格式适配（NCHW/NHWC）、参数命名/含义差异、计算图前置优化。

17.4.2 规范式转换：ONNX

ONNX（Open Neural Network Exchange） 是一种针对机器学习设计的开放式文件格式，由微软、亚马逊、Meta 和 IBM 等公司共同开发。它定义了一种可扩展的计算图模型、内置运算单元（OP）和标准数据类型。

ONNX 相当于一种"通用计算图"——不同 AI 框架构建的计算图都能转化为它。

ONNX 模型可视化（Netron）

使用 Netron 可视化 ONNX 模型，每个算子记录三类信息：

  1. 算子属性信息（Attributes）：如卷积核大小、步长等，最终用来生成具体算子
  2. 图结构信息：算子名称、邻边关系。如 /fc1/Gemm，输入 input，输出 /fc1/Gemm_output_0
  3. 权重信息：训练后的参数，如 fc1.weight 和 fc1.bias

17.4.3 模型转换通用流程

通用流程可分为四步：

  1. AI 框架生成计算图

    • 基于 AST 转换：词法/语法分析 → 抽象语法树转写，将动态图代码映射为静态图代码
    • 基于 Trace：动态图模式下记录调度顺序 → 构建静态图模型并保存

  2. 对接主流通用算子：确保通用算子在目标框架有对应实现；自定义算子需编写专用转换逻辑或替换为等效通用算子组合

  3. 目标格式转换：将模型转换到推理引擎的自定义 IR（包含计算图、算子、参数等全部信息）

  4. 导出保存模型文件：根据 IR 导出最终可执行推理的模型文件

转换过程中需注意：算子兼容性、张量格式差异、计算图优化，以及确保模型精度不受损失。

小结与思考

  • 模型转换：将不同 AI 框架训练得到的模型统一转换为推理引擎能够理解和执行的中间表示（IR），以实现跨框架的模型部署。

  • 推理引擎架构：包含优化阶段和运行阶段。优化阶段负责模型转换和图优化；运行阶段涉及模型的实际加载与执行，包括调度与执行两层。

  • 转换模块四大挑战：AI 框架算子的统一、不同框架模型文件格式的支持、主流网络结构的适配、各类输入输出的兼容。

  • 优化模块四大冗余：结构冗余、精度冗余、算法冗余、读写冗余——通过图优化消除它们以提高模型效率和性能。

  • 模型序列化：Protobuf（TLV 编码、需解析）和 FlatBuffers（零解析、直接访问）是两种主流的目标文件格式，各有适用场景。

  • 自定义计算图 IR：通过 Protobuf/FlatBuffers 构建张量、算子、网络的完整定义，实现对不同 AI 框架的统一对接。

  • 模型转换两种方式：直接转换（框架→IR 一步到位）和规范式转换（框架→ONNX→IR 两步走），后者以 ONNX 为代表实现了良好的跨框架兼容性。

核心感悟：模型转换是 AI 工程化的"翻译官"——它让 PyTorch 训练的模型能在 TensorRT 上高效推理，让 MindSpore 的模型能跑在手机端。理解 IR 的设计思想和 Protobuf/FlatBuffers 的序列化机制，是深入 AI Infra 的关键一步。
