FPGA学习笔记

第2章: What Can FPGA-Based Prototyping Do For Us?

Jun 04, 2026
3 minute read

章节概览

第1章建立了"为什么需要原型验证"的商业论证；第2章则聚焦于"FPGA 原型具体能做什么（以及不能做什么）"。

本章的核心信息是：FPGA 原型是唯一能同时提供 RTL 精度和接近实时速度的方法。这打破了其他建模方法中精度与速度的反比关系。通过三个详实的案例（Freescale 3G 协议测试、DS2 电力线通信、BBC DVB-T2 数字电视），本章展示了 FPGA 原型在真实项目中的关键价值。同时，作者也坦诚地讨论了 FPGA 原型的局限性——它不是仿真器，不是 ESL 工具，不同验证方法之间需要保持连续性。

核心知识点

  • Table 1 项目目标分组：三大类原因驱动 FPGA 原型——高性能与精度（含实时数据流、软硬件集成、软件验证）、真实世界接口（真实数据测试、人机交互、罕见数据依赖）、实验室外使用（可行性验证、算法测试、展览/投资者演示）
  • 精度-速度的反比关系：虚拟平台速度快但精度低，RTL 仿真精度高但速度慢——只有 FPGA 原型同时打破二者
  • 软件栈的分层精度需求：底层（BSP/驱动）需要周期精度和完全硬件可见性，顶层应用只需"刚好够骗过应用"的模型精度
  • Freescale 案例：FPGA 原型使 3G 协议测试从仿真器的"几周"或纯 RTL 仿真的"几年"缩短到一天，产品周期加速 6 个月以上
  • DS2 案例：FPGA 原型是唯一能拿到真实家庭/工业电力线环境中进行现场测试的方案——仿真器太贵且不可搬运
  • BBC 案例：DVB-T2 标准发布同日即实现世界首次直播传输，三个月后完成端到端演示——FPGA 的灵活性和独立运行能力是关键
  • 三个"不是"：FPGA 原型不是 RTL 仿真器（调试可见性远不如），不是 ESL 工具（不能在架构层面探索），需要在 RTL 稳定后才能移交

详细笔记

2.1 FPGA 原型的不同目标

Table 1：项目目标分类

本书作者通过多年与原型工程师的交流，将使用 FPGA 原型的理由归纳为三类：

项目目标 为什么用原型？
测试实时数据流 高性能与精度
早期软硬件集成
早期软件验证
测试真实世界数据效应 真实世界接口
测试实时人机接口
调试罕见数据依赖
可行性验证（概念证明） 实验室内使用
算法测试
公开展览 实验室外演示
吸引投资者
扩展 RTL 测试与调试 其他目标

2.1.1 高性能与精度

这是所有 FPGA 原型项目最底层的共同原因。

即使团队对外声称的目标是"验证嵌入式软件"，真正驱使他们选择 FPGA 而非虚拟平台的根本原因仍然是：他们需要 RTL 精度的同时还需要接近实时的速度。

2.1.2 实时数据流：HDMI 原型案例

Synopsys IP 团队（葡萄牙波尔图）的 HDMI 原型项目展示了实时数据流的价值：

  • 高清媒体数据流从外部接收器 → 原型处理器核心 → HDMI PHY → 外部显示器
  • 包含 MicroBlaze CPU + 外设 + 存储器，是一个完整的 SoC 结构
  • 只有 FPGA 原型能做到：立即看到和听到不同高清数据对设计的影响

这个案例的关键洞察是：SoC 的状态取决于当前状态 + 输入序列 + 系统反馈——只有实时运行才能观察到这些动态交互。

2.1.3 软硬件集成

软件开发已主导 SoC 开发工作量，处于项目关键路径上。软件开发和验证决定了 SoC 能否按时进入量产。

2.1.4 为软件开发建模 SoC

以 Android 软件栈为例（图15）：

栈层 精度需求 开发者关注点
应用层（Contacts, Browser, Phone） 最低——“刚好够骗过应用” 不直接访问硬件
中间件 / OS Kernel / 驱动 需要寄存器精度 完整的硬件映射
BSP / BIOS 层 最高——地址和时钟周期精度 硬件行为的软件描述

关键原则：

在每个软件栈层，开发者只需要一个精度"刚好够骗过自己的代码觉得在目标 SoC 上运行"的模型。更多精度只会让模型跑得更慢。

  • 应用层 → SDK 或虚拟平台即可，速度快
  • 底层驱动/BSP → 需要周期精确模型，FPGA 原型
  • 当整个软件栈需要一起运行、或需要处理真实世界数据时，FPGA 原型的价值最大化

2.1.5 案例一：Freescale 3G 协议测试

这是本章最精彩的案例。

背景：Freescale 蜂窝产品集团（德州奥斯汀），手机市场产品生命周期只有 1-2 年。

挑战：3G 协议测试即使在实时的速度下也需要一整天才能完成一轮：

  • RTL 仿真 → 需要几年
  • 仿真器（Emulator）→ 需要几周
  • 两者都不实用

解决方案：

  • 硬件：Synopsys HAPS-54 原型板 + 4 颗 Xilinx Virtex-5 FPGA
  • 设计规模：基带处理器超过 500 万 ASIC 门
  • 架构：双核 MXC2（StarCore DSP + ARM926）+ 60+ 外设 + RF 收发器数字部分
  • 工具：Synopsys Certify 将基带处理器分区到 3 颗 FPGA，RF 设计放第 4 颗
  • 模拟部分不原型化，用 Anritsu 协议测试仪直接提供数字蜂窝网络数据

遇到的困难：

  • 老核心使用了一些 ASIC 友好但 FPGA 不友好的设计技巧
  • 自动生成的 RTL 有过于复杂的时钟网络
  • 需要修改 RTL 使其 FPGA 兼容——但回报显著

成果：收到初版芯片时已经完成：

  • 调试器软件发布（硅后无大改）
  • 驱动软件开发完成
  • SoC 启动到 OS 提示符
  • 调制解调器完成网络注册和驻留
  • 收到芯片仅 1 个月后实现系统级通话
  • 产品进度加速超过 6 个月

项目负责人 Scott Constable 的原话（非常重要）：

“除了我们既定的协议测试目标之外，FPGA 系统原型在诸多其他领域也带来了项目进度加速，多次证明了它的价值。也许最重要的是不可量化的人力收益——让工程师更早地参与项目，让设计、软件、验证、应用所有团队在芯片到来之前 6 个月就非常熟悉产品。这种加速的产品专业素养很难在甘特图上衡量，但可能是最有价值的。”

“鉴于这些成就，使用 FPGA 原型方案加速 ASIC 进度是一个’不用动脑’的决定。我们已将这套方法论推广到 Freescale 网络和微控制器部门。”

2.2 真实世界接口：测试真实数据效应

为什么需要真实数据测试？

验证的困境：

  • 对于单个门或小 RTL 块，完整测试是可能的
  • 随着复杂度增长，统计上不可能穷尽所有输入数据和初始条件的组合
  • 约束随机验证虽然极大提升了覆盖度，但永远不是完全穷尽的

FPGA 原型的独特贡献：将 SoC 放入原型中，以接近真实芯片的速度和精度运行，在最终的真实环境数据中进行"浸泡"测试——就像最终芯片一样。

2.2.1 案例二：DS2 电力线通信（BPL）

背景：西班牙瓦伦西亚的 DS2 公司，宽带电力线通信技术——用家庭电力线传输高清视频。

核心技术挑战：

  • 复杂的硬件算法和嵌入式软件负责在电力线中编码和提取高速信号
  • 电力线是非常嘈杂的电气环境
  • 必须在各种真实世界条件下验证算法

为什么只能用 FPGA 原型（DS2 的 ASIC 设计经理 Javier Jimenez 的话）：

“开发可靠的高速通信需要使用鲁棒的验证技术。需要非常多次的试验，使用不同的信道和噪声模型。只有 FPGA 原型能让我们充分测试算法并在原型上运行嵌入式软件。此外，我们可以把原型带出实验室进行广泛的现场测试，将多个原型放置在真实的家庭和工作场所——其中一些是非常恶劣的电气环境。仿真器系统无法用于此目的：它们太贵且不便于携带。”

关键启发：让平台可靠和便携是成功的关键——这将在第5章和第12章深入讨论。

2.3 可行性实验

FPGA 原型在项目初期的可行性研究中也有价值：

  • 芯片拓扑、性能、功耗、片上通信结构等基本决策阶段
  • 一旦有 RTL（即使是算法工具自动生成的），就可以在 FPGA 中运行
  • 比纯算法/数学工具的额外好处：更精确的模型 + 实时速度 + 真实世界输入交互
  • 用途：在完整的 SoC 项目之间利用已有的 FPGA 硬件和工具，提高投资回报

2.4 原型走出实验室

FPGA 原型的独特能力：独立运行。

因为 FPGA 可以从 Flash EEPROM 等自包含介质配置（无需宿主机），原型可以在仿真器或虚拟平台无法企及的环境中运行。例如：

  • 将原型安装在移动车辆中，探索运动、外部噪声、天线场强变化对设计的影响
  • 通过公共 GSM 网络在行驶车辆中进行移动通话测试
  • 贸易展会上进行预生产演示

2.4.1 案例三：BBC DVB-T2 数字电视

背景：BBC 研发部门（伦敦），DVB-T2 是新的地面广播高清电视国际标准，规范制定耗时 30,000 工程师小时。

挑战：国际标准在最终冻结前会经历多次变更，需要灵活的硬件平台。

为什么选 FPGA：只有 FPGA 能提供在标准变更过程中所需的高度灵活性。

时间线：

  • 2008年3月：标准冻结
  • 2008年6月26日：作为 DVB 蓝皮书发布
  • 同日：BBC 使用 FPGA 原型实现世界首次 DVB-T2 直播传输
  • 2008年9月：IBC 阿姆斯特丹展览会上完成端到端调制解调演示——标准发布后仅 3 个月

硬件方案：

  • 调制器：Synopsys HAPS-51 卡 + Virtex-5 FPGA + BBC 自研子卡（ASI 接口 + 上变频到 UHF）
  • 解调器：同样基于 HAPS 的 FPGA 原型

BBC 项目负责人 Justin Mitchell：

“FPGA 最大的优势之一是能够跟踪规范在发射日前的晚期变更。能够在规范变更时快速修改调制器是非常重要的。很难想到有其他技术能在如此短的时间内实现调制器和解调器的快速开发，并且具备便携性，使调制器和解调器能够独立用于现场发射机和公开展览。”

后续：2009年3月都灵 PlugFest —— 5 个不同调制器和 6 个不同解调器互操作成功。

2.5 FPGA 原型做不到什么

作者明确列出了三个局限性，体现平衡视角：

2.5.1 FPGA 原型不是 RTL 仿真器

将仿真器比喻为 引擎 + 仪表盘：

组件 仿真器 FPGA 原型
引擎（激励模型 + 记录结果） 灵活、快速迭代 快得多，但设置代价大
仪表盘（检查结果） 完全可见和控制 即使最充分的插桩也仅提供仿真器的一部分信息

“仿真器是反复编写和评估 RTL 代码的更好环境。应该等到仿真基本完成、RTL 相对成熟后，再移交给 FPGA 原型团队。"（第4章会详细讨论交接点）

2.5.2 FPGA 原型不是 ESL

ESL（电子系统级）工具允许在 SystemC 中使用预定义模型库进行设计和仿真：

  • FPGA 原型需要 RTL，不适合算法或架构探索
  • FPGA 原型在 RTL 成熟后最有价值——此时软件可以在更精确、更真实的环境中运行
  • 有些团队会写少量 RTL 在 FPGA 中做可行性研究——这是一个小而重要的用途，但不同于全 SoC 系统级探索

2.5.3 连续性是关键

优秀工程师选择正确工具，但必须确保工作可以在工具间传递：

  • ESL 仿真 → FPGA 原型 → 需要尽可能少的重复工作
  • 高层综合（HLS）可以生成 RTL 供 FPGA 原型使用
  • 但同样，应该等到硬件-软件分区和架构探索完成、RTL 稳定后再上 FPGA
  • 第13章将讨论如何将 FPGA 原型与 ESL 和 RTL 仿真连接

2.6 总结：为什么要用 FPGA 原型？

三个视角：

团队 FPGA 原型的价值
硬件团队 为需要长运行时间的 RTL 交互提供高速平台（如整个 OS boot 可在接近实时完成，节省数天仿真时间）
软件团队 独一无二的芯片前软件调试平台——快速且精确，接近最终芯片条件
整个团队 最关键的阶段是软硬件首次集成。如果在芯片流片后才做这件事——发现新 bug 不是好消息。FPGA 原型让软件尽早接触精确快速的硬件模型

SoC 团队经常告诉我们：FPGA 原型最大的好处是——当初版芯片到达时，系统和软件在一天之内就能跑起来。

个人总结

第2章通过三个高水平的真实案例，把第1章的抽象论证变成了具体故事。

对我触动最大的是 Freescale 案例中的"不可量化的人力收益”——让所有团队在芯片到来前 6 个月就熟悉产品。这个价值不在甘特图上，但任何一个管理过复杂硬件项目的工程师都能理解它的分量。当硬件、软件、验证、应用四个团队同时熟悉同一个平台时，chip bring-up 的效率提升是倍数级的。

另一个重要收获是：FPGA 原型不是来替代仿真器或 ESL 的，而是填补它们之间的空白。一个成熟的验证策略应该是三者的组合：

  1. ESL/虚拟平台 → 最早期的架构探索和早期软件开发
  2. RTL 仿真器 → 反复迭代的 RTL 开发与验证
  3. FPGA 原型 → RTL 成熟后的高速、高精度系统级验证

“连续性"是本章埋下的伏笔——第13章将详细讨论这三种方法如何桥接。

实践思考

  • 如何判断 RTL 是否"足够成熟"可以上 FPGA？ 这是第4章的核心话题，但本章暗示了标准：仿真大部分完成、RTL 基本稳定、软硬件开始需要集成验证
  • 自主运行能力（standalone）是 FPGA 原型的一个被低估的优势。不需要宿主机的独立运行意味着原型可以放在仿真器永远去不了的地方——行驶的车辆、真实的家庭、贸易展览会
  • BBC 案例的标准-实现并行策略值得关注：在标准制定过程中就使用 FPGA 原型进行开发和测试，标准发布当天就完成首次直播——这种敏捷性在标准驱动的行业中极具竞争力
  • DS2 案例的启示：如果被测环境本身就是"嘈杂"和"不可预测"的，那么无论多么先进的仿真都无法替代真实环境中的浸泡测试
