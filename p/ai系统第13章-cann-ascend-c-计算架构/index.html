AI系统：第13章 CANN & Ascend C 计算架构

《AI系统：原理与架构》学习笔记 — 深入华为CANN异构计算架构与Ascend C算子编程语言，从五层架构设计到向量编程范式

8 minute read

本章概览

第6章我们讲的是昇腾 NPU 的硬件（达芬奇架构、AI Core、Cube 单元），这一章则聚焦于它上面的软件栈。如果说 AI Core 是昇腾的"肌肉"，那么 CANN 和 Ascend C 就是支配肌肉的"神经系统"。

本章的核心内容可以拆成两个大块：

  • CANN（昇腾 AI 异构计算架构）：从顶到底的五层软件栈 —— AscendCL → 计算服务层 → 编译层 → 执行层 → 基础层。每一层都有清晰的分工：向上对接 PyTorch / TensorFlow / MindSpore 等框架，向下驱动昇腾 AI 处理器。
  • Ascend C 编程语言：面向算子开发而生的 C/C++ 扩展语言，通过 SPMD 并行模型、流水编程范式、TQue/TPipe/TBuf 等机制，让 AI 开发者能以"写 C++“的方式开发高性能算子。

一句话总结：CANN 是昇腾的"操作系统”，Ascend C 是 CANN 的"编程语言"。理解这两者，你就理解了如何从零开始让一段 PyTorch 代码在昇腾芯片上高效运行。

13.1 昇腾异构计算架构 CANN

13.1.1 总体架构

CANN（Compute Architecture for Neural Networks）是一套为深度神经网络计算需求专门设计的架构。针对达芬奇架构提供软件层面的支撑，包括管理网络模型、计算流及数据流的软件栈。

在软件层面，CANN 支持业界多种主流 AI 框架（MindSpore、PyTorch、Jittor 等），并提供：

组件 作用
Ascend C 开放全量低阶 API（高性能自定义算子）+ 高阶 API（快速开发复杂算子）
GE（图引擎） 图优化、图编译、图执行，优化整网性能
HCCL（集合通信库） 改善网络拥塞，提升网络资源利用率和运维效率
AOL（算子加速库） 基础算子 + 大模型融合算子 API，使能大模型极致性能优化
Runtime 运行时 开放硬件资源 API（计算、通信、内存管理等）

CANN 自顶向下分为 5 层：

层级 名称 核心职责
L1 AscendCL（昇腾计算语言） 昇腾开放编程框架，封装底层服务接口，提供 Device/Context/Stream/内存/模型加载等 API
L2 昇腾计算服务层 AOL 算子库（NN/BLAS 等）+ AOE 调优引擎 + Framework Adaptor（适配 PyTorch/TF 等）
L3 昇腾计算编译层 图编译器（Graph Compiler）+ TBE 张量加速引擎，将 IR 编译为昇腾硬件可执行模型
L4 昇腾计算执行层 Runtime 库 + Graph Executor + DVPP/AIPP + HCCL 集合通信
L5 昇腾计算基础层 SVM 共享虚拟内存、设备虚拟化（VM）、主机-设备通信（HDC）等基础服务

13.1.2 关键功能特性

CANN 围绕四大核心场景提供能力：

1. 推理应用开发：通过 AscendCL 统一 API 框架，实现对昇腾硬件计算资源、深度学习推理、图像预处理、单算子加速等能力的调用。

2. 模型训练：提供 PyTorch/TensorFlow 模型迁移工具 + 自动化调测工具（数据异常检测、融合异常检测、整网数据比对）。

3. 算子开发：提供 1400+ 硬件亲和的高性能算子，同时开放自定义算子开发能力。

4. 四大核心优势：

  • 统一 APP 编程语言：AscendCL 屏蔽底层多种芯片差异
  • 统一网络构图接口：标准 AIR 中间表示
  • 高性能计算引擎及算子库：预置高性能算子库，降低部署成本
  • 基础业务：驱动、虚拟化、媒体、集合通信等

13.1.3 CANN 各层面能力详解

1. 昇腾计算语言（AscendCL）

包含三部分接口：

接口类型 用途
应用开发接口 深度学习推理、图像预处理、单算子调用及加速，通常用于离线推理应用
图开发接口 统一网络构图，支持多框架调用，可基于算子原型构图或用 Parser 解析神经网络输出 IR
算子开发接口（Ascend C） C&C++ 原生支持，结构化核函数编程，自动流水并行调度，CPU/NPU 孪生调试

2. 昇腾计算服务层

  • AOL 算子库：NN 算子库、BLAS 算子库、DVPP 算子库、AIPP 算子库、HCCL 算子库、融合算子库
  • AOE 调优引擎：算子调优（OPAT）、子图调优（SGAT）、梯度调优（GDAT）、模型压缩（AMCT）

调优是门槛很高的开发活动——开发者需要了解片内缓存大小、数据搬运逻辑、调度策略等。AOE 将常见调优手段固化为工具，让开发者只需通过工具分析模型即可获得性能提升。

3. 昇腾计算编译层

分为图编译和算子编译两个阶段：

  • 图编译：形状推导 → 常量折叠 → 死边消除 → 图融合 → 图切分 → 流水执行 → 缓存复用 → 算子引擎选择 → Cost Model → 整图内存复用 → 连续内存分配 → Task 生成
  • 算子编译：UB（Unified Buffer）融合、CCE-C 代码生成

4. 昇腾计算执行层

核心组件：

  • Runtime：执行流管理、上下文管理、事件管理、任务管理、资源申请/管理
  • Graph Executor：计算图加载和执行
  • HCCL：子通信域管理、Rank 管理、梯度切分、集合通信
  • DVPP/AIPP：两个维度上的数据预处理

5. 昇腾计算基础层

组件 职责
RMS（资源管理服务） 管理与调度昇腾设备的计算、Device 内存等关键资源
CMS（通信管理服务） 提供片内、片间高效通信
DMS（设备管理服务） 设备配置、切分、升级、故障检测
DRV（驱动） 使能硬件
UTILITY 基础库和系统维测能力

基础层的四大竞争力：高性能（微秒级确定性调度 + 数据零拷贝）、高可信（五道安全防线）、归一化（一套架构-接口-代码支持多芯、多板、多场景）、弹性（端/边/云灵活适应，细粒度算力切分）。

13.2 CANN 与算子

13.2.1 算子基本介绍

算子是编程和数学中的重要概念：

分类 示例 说明
一元算子 取反、递增 对单个操作数操作
二元算子 加法、赋值 对两个操作数操作
关系算子 >, <, == 比较值之间关系
逻辑算子 AND, OR 逻辑表达式中组合条件
位运算符 &, |, ^ 操作二进制位

在神经网络中，算子就是计算图中的张量处理节点——卷积算子、全连接算子、激活算子等，无一例外地接受张量输入并输出处理结果。

关键认知：

  • 并非所有算子都有数学含义——数据排布变换、类型转换、广播与归纳等属于纯编程操作
  • 同一算子在不同硬件上实现千差万别（CPU/GPU/NPU/TPU 上的卷积完全不同）
  • 不同算子适合运行在不同硬件上：大量矩阵/向量计算的放专用加速器，分支/上下文切换多的放 CPU

13.2.2 CANN 算子体系

CANN 算子分为两类：

算子类型 运行位置 适用场景
AI Core 算子 AI Core（矩阵/向量/标量计算密集） 大多数计算密集型算子
AI CPU 算子 AI CPU（负责 CPU 类算子） 非矩阵类复杂计算、控制算子、标量和向量通用计算

AI CPU 涉及的组件：GE（图引擎）→ AI CPU Engine（子图编译引擎）→ AI CPU Scheduler（模型调度器）→ AI CPU Processor（Task 执行器）→ Data Processor

三种使用 AI CPU 的场景：

  1. 场景一：不适合跑在 AI Core 上的算子——非矩阵类复杂计算、分支密集型算子、离散数据/资源管理/随机数类计算
  2. 场景二：AI Core 不支持的数据类型——如 Complex32、Complex64
  3. 场景三：快速打通模型流程——先用 AI CPU 调通功能，后续再转换为 AI Core 算子做性能优化

AI CPU 开发三步要求：

  1. 自定义算子类必须是 CpuKernel 的派生类，在命名空间 aicpu 中声明
  2. 实现 Compute 函数（计算逻辑）
  3. 用 REGISTER_CPU_KERNEL 注册算子

13.3 算子开发编程语言 Ascend C

13.3.1 并行计算基本原理

并行计算涉及同时在多个硬件处理单元上执行任务，以提高整体计算性能和效率。

并行体系结构分类

从硬件/系统/应用层面：

类型 含义 关键技术
指令级并行 (ILP) 处理器内部多个指令同一时钟周期执行 超标量架构（Superscalar）、流水线（Pipelining）
线程级并行 (TLP) 多线程在多核/多处理器上并行运行 需程序员显式创建和管理线程
请求级并行 (RLP) 多个独立客户端请求被同时处理 Web 服务、数据库等场景

从软件设计/编程模型角度：

类型 含义 例子
数据级并行 相同操作作用于不同数据片段 发工资：各处理器计算不同员工的工资
任务级并行 不同任务在不同处理单元执行 发工资：一个处理器算税务扣除，另一个算福利待遇

Ascend C 就是基于昇腾 AI Core 形成的 SPMD（单程序多数据） 编程模型——每个 AI Core 执行相同的程序，但处理不同的数据分片。

阿姆达尔定律（Amdahl’s Law）

并行加速的上限受限于程序中的串行部分：

\[ S = \frac{1}{(1-P) + \frac{P}{N}} \]

其中 \(S\) 是加速比，\(P\) 是可并行化比例，\(N\) 是处理器数量。

如果一个程序有 10% 的代码是串行的（\(P\) = 0.9），即使在无限多处理器上，最大加速比也只能达到 10 倍。

13.3.2 Ascend C 编程体系

1. AI Core 硬件抽象

AI Core 包含三个基础计算单元：

单元 角色
矩阵计算单元（Cube） 矩阵乘累加运算
向量计算单元（Vector） 向量运算
标量计算单元（Scalar） 读取指令序列，发射指令到各单元
DMA 搬运单元 在 Global Memory ↔ Local Memory 之间搬运数据

AI Core 内部异步并行过程：

  1. Scalar 单元读取指令序列，把向量/矩阵/数据搬运指令发射给对应单元的指令队列
  2. 各单元异步并行执行接收到的指令
  3. Scalar 单元下发同步指令保证依赖关系
  4. 数据流：DMA 搬入 → Local Memory 计算 → DMA 搬出到 Global Memory

2. SPMD 编程模型

SPMD 的核心思路：

  • 将待处理数据切分成多片，分发给不同进程（AI Core）
  • 每个进程对自己的数据分片执行相同的处理任务
  • 多个 AI Core 共享相同指令代码，通过 block_idx 识别自己身份

为充分利用计算资源，Block 数量通常设为物理核数的整数倍。

3. 流水编程范式

Ascend C 的核函数内部可以拆分成多个流水任务，通过队列（TQue）完成通信和同步。

以向量算子为例，三阶段流水：

阶段 作用 负责单元
阶段1：CopyIn 数据从 Global Memory 搬入 Local Memory DMA
阶段2：Compute 向量/矩阵计算 Vector/Cube
阶段3：CopyOut 数据从 Local Memory 搬出到 Global Memory DMA

流水并行的关键：对于同一片数据，三个阶段是串行的；但对不同数据切片，三个阶段可以同时执行——CopyIn 搬第 3 块时，Compute 正在算第 2 块，CopyOut 正在搬出第 1 块。

TQue 分为 VECIN（搬入队列）和 VECOUT（搬出队列），阶段间通过它们传递数据——这就是 Ascend C 流水编程的核心机制。

13.4.2 Ascend C API 概述

API 的操作数都是 Tensor：GlobalTensor（外部存储）和 LocalTensor（核上内存）。

基础 API（四大类）

API 类型 功能 示例

内存管理 API 板上动态内存分配与释放 AllocTensor, FreeTensor

基础 API 的三种计算方式：

方式 特点 灵活性
整个 Tensor 参与计算 运算符重载，dst = src1 + src2 最简单，功能受限

13.4.4 任务间通信与同步

TQue 是 Ascend C 的核心通信队列，通过逻辑位置 TPosition 屏蔽物理存储细节。

TPosition 类型 用途场景 含义
GM 全局 全局内存（Global Memory）
VECIN 向量计算 搬入数据的存放位置
VECOUT 向量计算 搬出数据的存放位置
VECCALC 向量/矩阵计算 计算临时变量
A1 / B1 矩阵计算 整块矩阵 A/B（类比 L2 Cache）
A2 / B2 矩阵计算 切分小块矩阵 A/B（类比 L1 Cache）

13.4.5 资源管理

TPipe 是片上内存的统一管理者，通过 InitBuffer 接口为 TQue 分配内存。

1
2
3
4
5

TPipe pipe;
pipe.InitBuffer(que, num, len);
// que: 队列名
// num: 内存块数（num=2 时开启 Double Buffer 优化）
// len: 每块内存大小（Byte）

Double Buffer 机制：利用 MTE 指令队列与 Vector 指令队列的独立性和可并行性，将数据搬运与计算并行执行，隐藏数据搬运时间，提高 Vector 单元利用率。

13.4.6 临时变量

TBuf 管理 Compute 阶段的临时变量，存储位置通过 TPosition 模板参数设置。

1
2
3
4

TBuf<TPosition::VECCALC> calcBuf;               // 声明临时变量
pipe.InitBuffer(calcBuf, len);                   // 初始化
LocalTensor<half> temp = calcBuf.Get<half>();    // 获取指定长度 Tensor
LocalTensor<half> temp = calcBuf.Get<half>(128); // 指定长度

建议：将 TBuf 声明为算子类成员变量，避免重复申请与释放，提升性能。

TBuf vs TQue 关键差异：

  • TBuf 分配的内存只能参与计算，无法执行入队/出队操作
  • TBuf 一次只分配一块内存（TQue 可申请多块）
  • TBuf 不需要反复申请/释放

13.5 Ascend C 编程范式——以向量为例

本章最重要的部分：通过一个完整的 add_custom 向量算子案例，展示 Ascend C 编程范式的全流程。

开发流程概览

自定义向量算子的三个步骤：

  1. 算子分析：明确输入输出、数学表达式、所需 API
  2. 核函数定义与封装：声明 kernel 函数 + 调用算子类 Init/Process
  3. 算子类实现：Init（内存初始化）+ Process（CopyIn → Compute → CopyOut）

Step 1: 算子分析

以 Add 算子为例：

  • 数学表达式：\(z = x + y\)
  • 输入：x, y（half/float16）
  • 输出：z（与输入同类型）
  • Shape：(8, 2048)，ND 格式
  • 所需 API：DataCopy（搬运）、Add（计算）、EnQue/DeQue（队列管理）

class KernelAdd {
public:
    __aicore__ inline KernelAdd() {}
    __aicore__ inline void Init(GM_ADDR x, GM_ADDR y, GM_ADDR z) {
        // 计算本核负责的数据位置
        xGm.SetGlobalBuffer((__gm__ half*)x + BLOCK_LENGTH * GetBlockIdx(), BLOCK_LENGTH);
        yGm.SetGlobalBuffer((__gm__ half*)y + BLOCK_LENGTH * GetBlockIdx(), BLOCK_LENGTH);
        zGm.SetGlobalBuffer((__gm__ half*)z + BLOCK_LENGTH * GetBlockIdx(), BLOCK_LENGTH);
        // 初始化 TQue 内存
        pipe.InitBuffer(inQueueX, BUFFER_NUM, TILE_LENGTH * sizeof(half));
        pipe.InitBuffer(inQueueY, BUFFER_NUM, TILE_LENGTH * sizeof(half));
        pipe.InitBuffer(outQueueZ, BUFFER_NUM, TILE_LENGTH * sizeof(half));
    }

    __aicore__ inline void Process() {
        constexpr int32_t loopCount = TILE_NUM * BUFFER_NUM;
        for (int32_t i = 0; i < loopCount; i++) {
            CopyIn(i);
            Compute(i);
            CopyOut(i);
        }
    }

private:
    __aicore__ inline void CopyIn(int32_t progress) {
        LocalTensor<half> xLocal = inQueueX.AllocTensor<half>();
        LocalTensor<half> yLocal = inQueueY.AllocTensor<half>();
        DataCopy(xLocal, xGm[progress * TILE_LENGTH], TILE_LENGTH);
        DataCopy(yLocal, yGm[progress * TILE_LENGTH], TILE_LENGTH);
        inQueueX.EnQue(xLocal);
        inQueueY.EnQue(yLocal);
    }

    __aicore__ inline void Compute(int32_t progress) {
        LocalTensor<half> xLocal = inQueueX.DeQue<half>();
        LocalTensor<half> yLocal = inQueueY.DeQue<half>();
        LocalTensor<half> zLocal = outQueueZ.AllocTensor<half>();
        Add(zLocal, xLocal, yLocal, TILE_LENGTH);       // 核心计算！
        outQueueZ.EnQue<half>(zLocal);
        inQueueX.FreeTensor(xLocal);
        inQueueY.FreeTensor(yLocal);
    }

    __aicore__ inline void CopyOut(int32_t progress) {
        LocalTensor<half> zLocal = outQueueZ.DeQue<half>();
        DataCopy(zGm[progress * TILE_LENGTH], zLocal, TILE_LENGTH);
        outQueueZ.FreeTensor(zLocal);
    }

    TPipe pipe;
    TQue<TPosition::VECIN, BUFFER_NUM> inQueueX, inQueueY;
    TQue<TPosition::VECOUT, BUFFER_NUM> outQueueZ;
    GlobalTensor<half> xGm, yGm, zGm;
};

constexpr int32_t TOTAL_LENGTH = 8 * 2048;    // 数据总长度
constexpr int32_t USE_CORE_NUM = 8;           // 使用 8 个核
constexpr int32_t BLOCK_LENGTH = TOTAL_LENGTH / USE_CORE_NUM;  // 每核：2048
constexpr int32_t TILE_NUM = 8;               // 每核切分 8 块
constexpr int32_t BUFFER_NUM = 2;             // Double Buffer
constexpr int32_t TILE_LENGTH = BLOCK_LENGTH / TILE_NUM / BUFFER_NUM;  // 每块：128

切分层次：总数据 → 分配到 N 个核 → 每核数据切分为 M 块 → 每块再对半分（Double Buffer）

关键认知：Ascend C 中的"核数"是逻辑核的概念——即同时运行了多少个算子实例（进程数），而非物理处理器核数。建议逻辑核数为物理核数的整数倍。
