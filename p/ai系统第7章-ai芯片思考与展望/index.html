7.1 GPU 架构与 CUDA 关系

7.1.1 CUDA 基本概念

2006 年 11 月，英伟达推出 CUDA（Compute Unified Device Architecture，统一计算设备架构）——一个通用并行计算架构和编程模型，把 GPU 从"图形专用"变成了"通用并行计算设备"。

CUDA 基于 LLVM 构建编译器，开发人员用 CUDA C/C++ 编写并行程序，通过 CUDA API 将计算任务发到 GPU 执行。CUDA 还支持 Python、Fortran 等语言，兼容 OpenCL 和 DirectCompute。

CUDA 在软件方面由以下组件构成：

组件 功能
CUDA 库 基础运算库
CUDA API + Runtime 编程接口与运行时
CUFFT / CUBLAS 高级数学库（FFT / 线性代数）
CUDA TOOLKIT 编译器 + C++ 核
CUDA DRIVER 驱动 GPU，管理内存和图像
CUDA-X LIBRARIES ML/DL/HPC 加速库
APPS & FRAMEWORKS 对接 TensorFlow、PyTorch 等框架

7.1.2 CUDA 线程层次结构

CUDA 最基本的执行单位是线程（Thread）。线程通过三级层次组织并行计算：

  • Grid（网格）：一个 Kernel 启动的所有线程，共享全局内存
  • Block（线程块）：Grid 内的分组，块间并行、无法通信、无执行顺序；每个 Block 有共享内存
  • Thread（线程）：最小执行单位，同一 Block 内的线程可同步、可通过共享内存通信

简单记忆：把一个大问题 → 切分成独立 Block 并行解决的子问题 → 子问题再切分为 Block 内线程协作解决。

CUDA 编程核心：Kernel

Kernel 是在 GPU 上并行执行的函数，用 __global__ 声明，调用时用 <<<grid, block>>> 指定执行结构。

CUDA 编程流程总结：

  1. 编写 Kernel 函数描述并行计算任务
  2. 在主机上配置线程块和网格，将 Kernel 发送到 GPU 执行
  3. 在主机上处理数据传输和结果处理

CUDA 与硬件的对应关系：

  • 软件侧的 Thread → 硬件上的 CUDA Core（线程数量超配，CUDA Core 固定数量）
  • Block 只在一个 SM 上通过 Warp 调度，一旦调用就一直保留到执行完 Kernel
  • 多个 SM 组成的 TPC 和 GPC 实现硬件级并行计算

7.1.3 算力峰值计算

GPU 算力峰值（Peak FLOPS）是在理想情况下能达到的最大计算能力，单位为 FLOPS（GFLOPS / TFLOPS）。

决定因素：

  • CUDA 核心数量：算力与核心数成正比
  • 核心频率：频率越高，每秒执行的指令数越多
  • 每个核心的计算能力：每个核心每个时钟周期能执行的 FLOPS
  • 并行度：同时执行的线程数量

计算公式：

\[ \text{Peak FLOPS} = F_{clk} \times N_{SM} \times F_{req} \]

其中：

  • \(F_{clk}\)：GPU 时钟周期内指令执行数（FLOPS/Cycle）
  • \(N_{SM}\)：SM 数量
  • \(F_{req}\)：Tensor Core 核心运行频率（GHz）

7.2 从 GPU 对 AI 芯片思考

7.2.1 英伟达生态的思考点

从软件和硬件架构角度看，CUDA 和 SIMT 存在深度绑定关系。而当前 AI 芯片采用的 DSA 架构，其编程模型和硬件执行模型都还处于较为早期的状态。

流水掩盖：对 DSA 而言，实现架构层面的隐藏流水编排机制、提出易用性堪比 CUDA 的软件是可能的，但如果核心问题没解决，即使形式上与 CUDA 类似也会有易用性问题——开发者很难获得足够好的初始性能。

软硬件协同：DSA 架构一方面需要建立开放的软硬件架构联合对抗 CUDA 生态，另一方面需要明确面向不同层次开发者的易用性和软件开发形态。

1. SIMT 与 CUDA 的关系

CUDA 的成功之处在于通过 SIMT 架构掩盖了流水编排、实现了并行指令隐藏、提升了编程易用性。英伟达为维护 CUDA 生态，对 SIMT 硬件架构做了取舍（保留 SM、Warp、Thread 等概念），这使得 CUDA 近几年没有重大架构变化。

DSA 在硬件架构上比较激进，不是因为软件做得好，而是一开始没考虑编程体系——自然没有软件和硬件协同带来的架构约束。

2. DSA 硬件架构执行方式

DSA 硬件一般单核单线程，线程内指令通过多核共享 Cache 协作。裸接口通常是一个核一个线程，每个线程内串行调用 DSA 指令集，正确性部分靠软件同步、部分靠硬件保证。

7.2.2 借鉴与思考点

1. 流水编排

英伟达 GPU 依靠多线程交错执行来提升并行性能——大量线程通过不同 Block 和 Thread 读取数据、执行计算。而 SIMD 依赖 CPU 自身乱序、投机、缓存和预取能力。

关键洞察：即使为 DSA 的 SIMD 硬件封装了 SIMT 前端，如果遇到指令依赖，基础性能也会很差——流水编排仍然需要开发者手动实现。

2. SIMT 前端硬件

增加 SIMT 前端硬件后，通过 Warp（线程束） 隐藏线程指令流水：

  • 每个 Thread Block 需要很多并行线程，隐式分成若干个 Warp
  • 每个 Warp 包含串行交错的访存和计算
  • GPU 通过 Warp Scheduler 动态交错执行：Warp0 阻塞就切换到 Warp1

CUDA 没有解决 DSA 指令掩盖，只是给了开发者一个 Warp 概念来透传指令 API——所以 CUDA 的上手门槛并不低，需要充分了解 GPU 硬件细节。

3. 分支预测机制

SPMD 编程模型对分支预测和控制流的高容忍度是支撑易用性的重要方法。英伟达 GPU 使线程在 Warp-Base SIMD 上执行不同分支——每个线程都可以执行带条件的控制流指令，同时不同线程间可以走不同的控制流路径。

但当 Warp 内部线程分支到不同执行路径时，会发生分支执行冲突——不同时间执行不同路径会增加时耗。

动态 Warp Formation/Merging 是解决方案：

  • 分支后动态合并执行相同指令的线程
  • 从等待的 Warp 中形成新 Warp
  • 将 Warp X 和 Warp Y 合并为 Warp Z（更好地执行相同指令）

动态 Warp 分组更多在编译器层面解决分支预测问题，根据线程执行情况和数据依赖性动态组织 Warp 中的线程，提高并行计算性能和资源利用率。但不推荐在 CUDA 编程中出现大量 if/else 语句。

4. 交互方式

CUDA 运行时提供主机和设备的 C++ 交互方式（寒武纪 BANG C 语言就在这个层面参考了 CUDA）。在软件层面，CUDA 非常容易实现向量加法。

CUDA 的易用性对初阶用户来说是极致的——入门开发者任意写一个简单算子（Kernel），就能获得比 CPU 高 5~10 倍的峰值性能。

DSA 的软肋：流水和指令使用缺乏完备、隐式的支持，指令流水的支持需要开发者手动掩盖、切块，使用底层指令让写出正确 Kernel 耗时更长。

7.3 AI 芯片发展方向

2017 年，David Patterson 加入谷歌 TPU 团队，2020 年在伯克利发表演讲"A Decade of Machine Learning Accelerators: Lessons Learned and Carbon Footprint"，分享了 TPU 团队近几年的心得体会。本节摘录并深入探讨其中 8 点思考。

7.3.1 模型对内存和算力的需求

AI 模型近几年所需的内存和算力平均每年增长 50%，模型所需内存和算力增长 10~20 倍。但从芯片设计到实际应用需要约 5 年周期（设计 1 年 + 部署 1 年 + 优化 3 年）。

训练模型参数量的增长速度比推理更快：2016~2023 年 SOTA 训练模型的算力需求年均增长 10 倍；GPT-2 到 GPT-3 参数量提高了约 100 倍（15亿 → 1750亿）。

但 AI 芯片内存容量增长相对缓慢：

芯片 HBM 最大内存
A100 80 GB
H100 188 GB
TPU v4 32 GB
特斯拉 DOJO 16 GB
华为昇腾 64 GB
寒武纪 MLU 370 24/48 GB

7.3.2 模型结构快速演变

DNN 是一个发展迅速的领域：

  • 2016 年：MLP、LSTM 是主流
  • 2020 年：CNN、RNN、BERT 广泛应用
  • 如今：Transformer 家族（GPT-3.5、Phi-3、Gemma、Llama）参数从十亿级到万亿级

DSA 架构需要足够通用以支持未来可能出现的新模型——这是它面临的最大挑战之一。

7.3.3 生产部署提供多租户

大部分 AI 论文假设同一时间 NPU 只运行一个模型，但实际需要切换不同模型：

  • 机器翻译涉及语言对比，需要不同模型
  • 一个主模型配套多个模型实验
  • 对吞吐量和时延有不同要求，不同模型使用不同 Batch Size

因此需要多租户技术（Multi-Tenancy）：算力切分、显存虚拟化、内存寻址、虚拟内存页。

目前常见的 GPU 虚拟化技术包括：NVIDIA vGPU、AMD MxGPU、Intel GVT-g。

7.3.4 SRAM 与 DRAM 的权衡

根据 8 个模型的基准数据，其中 6 个涉及多租户技术。如果从 CPU 主机重新加载参数，上下文切换需要 10s——因此需要更快的 DRAM（片外存储）用于交换多种模型数据。

很多模型所需内存远大于单芯片最大 SRAM（片上存储）。部分芯片的设计思路是期望用 SRAM 解决所有任务、减少内存数据搬运时间，但这在多租户场景下很难实现。

AI 芯片不仅需要更大的 SRAM，更需要存储速度更快的片外 DRAM。

7.3.5 内存优先于计算速度

现代微处理器最大的瓶颈是能耗，而不是芯片集成度：

操作 能耗对比（相对值）
算术运算 1×
访问片上 SRAM 100×
访问片外 DRAM 5000~10000×

访问片外 DRAM 的能耗是算术运算的 5000~10000 倍！GPGPU 的功耗大多浪费在数据搬运上，而非核心计算。

图7.3.3 展示了谷歌 TPU v1 架构——65536（256×256）个矩阵乘法单元，700 MHz 时钟，峰值 92 TOPS：

TPU 使用脉动阵列以固定时间间隔使数据从不同方向流入处理单元。20 世纪 70 年代因只有单金属层才提出脉动阵列减少布线；如今多达 10 个金属层，最大难点是能耗，而脉动阵列能效高，使芯片容纳更多乘法单元，从而分摊内存访问开销。

7.3.6 DSA 既要专业也要灵活

DSA 的难点在于既要对模型进行针对性优化，又要保持一定灵活性：

  • 训练比推理更复杂：计算量更大，包含反向传播、转置和求导，需要更大内存存储中间结果
  • 需要支持更广泛的数据格式：BF16、FP16、HF32 等用于 AI 计算
  • 指令、流水、可编程性要求更高：需要灵活的编译器和上层软硬件配套
  • CUDA 在这个方面已积累多年，TPU 目前支持 INT8、BF16 等格式

7.3.7 半导体供应链的选型

计算逻辑进步很快，但芯片制程工艺发展相对较慢。SRAM 和 HBM 比 DDR4 和 GDDR6 速度更快、能效更高。大模型训练普遍使用 BF16，部分使用 FP8 推理。如果选型错误（如只用 FP32 模拟 BF16），训练迭代速度将减慢。

不同操作格式的能耗对比（7nm 工艺，单位 pJ）：

操作 INT8 BF16 FP16 FP32
加法 0.007 0.11 0.16 0.38
乘法 0.07 0.21 0.34 1.31

从 45nm 到 7nm，能耗平均降低 2.6 倍。BF16 乘法的能耗仅为 FP32 的 1/6——这就是为什么大模型训练偏爱 BF16。

7.3.8 编译器优化和 AI 应用兼容

DSA 编译器需要对 AI 模型进行分析和优化，将 AI 指令转换为高效底层代码：

  • 多核并行：实现 4096 个芯片的并行
  • 数据级并行：向量、矩阵、张量功能单元
  • 指令级并行：322~400 位超长指令字（VLIW）
  • 内存传输管理：编译器需要管理内存传输
  • 数据布局兼容：兼容不同功能单元和内存中的数据布局

与 CPU 的 GCC 和 GPU 的 CUDA 相比，DSA 的软件栈还不够成熟，但编译器优化能带来约 2 倍的性能提升（C++ 编译器一年能提升 5%~10% 就算很好了）。

算子融合（Operator Fusion） 能减少内存访问，省略中间结果写入 HBM 再读取的步骤。通过 MLPerf 基准测试，算子融合平均能带来超过 2 倍的性能提升。

FlashAttention 是 IO 感知优化的经典案例：

  • 运行速度比 PyTorch 标准注意力快 2~4 倍
  • 所需内存仅为标准注意力的 1/20~1/5
  • 核心思路：从 HBM 中读取块 → 在 SRAM 中计算 → 写到 HBM，避免从 HBM 读取/写入注意力矩阵
  • 算法没有减少计算量，而是从 IO 感知出发减少了 HBM 访问次数

7.4 超异构计算

7.4.1 异构与超异构场景

摩尔定律放缓，传统单一架构难以满足日益增长的计算需求。异构计算通过整合不同类型计算单元的优势来解决这一问题。

异构计算主要优势：

优势 说明
性能飞跃 CPU + GPU + FPGA 有机结合，实现 1+1>2
灵活定制 针对不同计算任务，灵活选择最合适的计算单元
降低成本 用更低成本实现更高性能，性价比更优
降低功耗 动态调整资源分配，提升能源利用效率

特斯拉 HW3 FSD 芯片是异构计算的典型案例——单一芯片包含 CPU、GPU、NPU 多种架构：

作为汽车芯片，它要负责雷达、GPS、地图等多种功能，传统单一架构难以高效完成。将各组件有机结合的异构芯片可以更好地处理复杂情况：

7.4.2 计算体系迎来异构

异构计算的出现和发展源于：

  • 传统冯·诺依曼结构受制于存储和计算单元之间的数据交换瓶颈
  • 半导体工艺发展使 CPU 主频提升受限
  • 深度学习等 AI 技术对计算能力提出了更高要求

CPU-GPU 异构工作流：

  1. CPU 把数据准备好（CPU 内存）
  2. 数据从 CPU 内存复制到 GPU 内存（Step1）
  3. CPU 指示 GPU 工作，配置并启动 GPU 内核（Step2）
  4. 多个 GPU 内核并行执行（Step3）
  5. 处理结果复制回 CPU 内存（Step4）
  6. CPU 对结果进行后续处理

异构架构三大优势：

  • 适用于高性能计算：单一计算类型和架构已无法处理更复杂多样的数据
  • 适用于海量数据：个人每天 ~1 GB，智能汽车 ~50 GB，智能医院 ~3 TB，智慧城市 ~50 PB
  • 共享内存空间：消除冗余内存副本，CPU 和 GPU 统一内存池

ASIC 在异构中的角色：通过驱动程序与 CSR 和可配置表项交互，CPU 控制数据输入/输出和运行控制。

从 CPU 到 ASIC，架构越来越碎片化——指令复杂度决定了系统的软硬件解耦程度。这形成了易用性和性能之间的权衡：

处理器 灵活性 性能效率
GPU 较好 不够极致
DSA 较差 优异
FPGA 中等 功耗成本较高
ASIC 完全固定 极致

异构计算的问题：复杂系统需要灵活处理器，性能挑战需要定制加速器——单一处理器无法兼顾性能和灵活性。

7.4.3 从异构到超异构

超异构出现的三大驱动力：

  1. 需求驱动：元宇宙等新兴技术需要算力提升 1000 倍
  2. 工艺和封装支撑：Chiplet 封装使单芯片层次构建规模数量级提升的超大系统
  3. 系统架构持续创新：通过架构创新实现多个数量级的性能提升

超异构并非简单集成——而是把更多异构计算整合重构，各类型处理器间充分、灵活的数据交互，形成统一的超异构计算体系。

超异构基本特征：

  • 超大规模计算集群：CPU、GPU、FPGA、DSA 等多种计算单元通过高速互联网络连接
  • 复杂计算系统：由分层分块组件组成，需统一管理和调度

7.4.4 超异构的挑战与思考

1. 超异构的软件层

软件需要跨平台复用：跨架构、跨处理器类型、跨厂家平台、跨位置、跨设备类型——软件架构的复杂性增长是最大挑战。

解决办法：

  • 开放接口/架构及生态：形成标准开放接口
  • 软件兼容：减少定制化开发，兼容已有软件生态
  • 编程体系：提供门槛更低的编译体系（如 CUDA、Ascend C）
  • 开放架构：防止架构过多导致市场碎片化

2. 硬件定义软件 vs 软件定义硬件

模式 核心理念
硬件定义软件（HDW） 硬件加速器 + 专用指令集，充分发挥硬件性能
软件定义硬件（SDH） 软件抽象层（SAL）解耦软硬件，跨平台运行
软硬件协同定义（Co-Design） 软硬件团队紧密合作，1+1>2

系统复杂度决定了哪种关系占主导：复杂度低的系统（如 CPU）→ 硬件定义软件；复杂度高的系统（如 GPU）→ 软件定义硬件，系统实现变成持续优化和迭代的过程。

3. 计算体系与编译体系

超异构架构需要构建高效、标准、开放的接口和架构体系，避免场景覆盖的碎片化。当前正处于计算体系变革和编译体系变革的 10 年。

4. 超异构的未来

跨平台统一计算架构是未来计算资源中心化的关键：

  • 将孤岛计算资源连接起来，实现计算资源池化
  • 支持应用软件跨同类处理器、跨不同类处理器、跨芯片平台、跨云边端运行
  • Chiplet + 超异构组合 → 系统规模数量级提升 → 性能数量级提升

未来计算架构的三大特征：

  • 体系异构：CPU、GPU、FPGA、DSA 多架构组成，接近 CPU 的灵活性 + ASIC 的性能效率
  • 平台化：软件定义一切、硬件加速一切，完全可软件编程的硬件加速平台
  • 开放生态：拥抱开源，支持云原生、云网边端融合，用户无平台依赖

小结与思考

第 7 章是整个 AI 硬件篇的收官之战。它不教你怎么用 CUDA 写代码，而是帮你建立一种系统性的思考框架。几个关键问题值得反复咀嚼：

  • CUDA 到底牛在哪？ 牛在它不仅是一个编程模型，更是一个"生态操作系统"——从硬件（CUDA Core、Tensor Core）到编译器（LLVM-based），到数学库（cuBLAS、cuDNN），再到框架插件（PyTorch/TF 无缝对接），每一层都严丝合缝。这是所有 DSA 追赶者面临的最大鸿沟。

  • DSA 能从 CUDA 学到什么？ ① SIMT 前端的 Warp 调度——用硬件掩盖流水阻塞，而非让开发者手动编排；② 分支预测的编译器层面优化——动态 Warp 分组让开发者对硬件执行无感知；③ 易用性优先——让入门开发者随便写个 Kernel 就能获得不错的性能。

  • David Patterson 的 8 点思考中最核心的是什么？ 内存优先于计算速度——访问片外 DRAM 的能耗是算术运算的 500010000 倍！AI 芯片的价值不在于堆更多的乘法器，而在于设计更聪明的数据流、减少 HBM 访问。FlashAttention 就是最好的例证——不减少计算量，只减少 IO，性能提升 24 倍。

  • 超异构是终点吗？ 超异构 = CPU + GPU + FPGA + DSA 的组合，目标是用接近 CPU 的灵活性实现接近 ASIC 的性能。但真正的挑战不在硬件层面（Chiplet 封装已经很成熟了），而在软件层面——如何让开发者用一套代码跑通所有异构单元？如何让编译器自动选择最优的计算单元？这些问题不解决，超异构就只是一个概念。
