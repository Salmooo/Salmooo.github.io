AI系统：第5章 TPU——以谷歌为例

《AI系统：原理与架构》学习笔记 — 深入谷歌TPU系列，从脉动阵列到3D Torus，从推理专用到训练超级计算机

6 minute read

本章概览

前面几章我们讨论了 CPU 和 GPU，这两种通用计算芯片在 AI 时代扮演了重要角色。但谷歌的思路更"激进"——既然现有的通用芯片跑 AI 不够高效，那为什么不从零开始，为 AI 量身打造一款芯片呢？

于是，TPU（Tensor Processing Unit） 诞生了。从 2016 年第一代 TPU v1 专为推理而生，到 2021 年 TPU v4 能够驱动 4096 颗芯片组成的 exaFLOPS 级别超级计算机，谷歌只用了短短几年时间就把"专用芯片"这条路走到了极致。

本章将带你深入谷歌 TPU 的"心脏"——理解脉动阵列的优雅设计，对比各代 TPU 的架构演进，并最终站在 Pod 超级计算机的视角，俯瞰谷歌如何用"光"连接 4096 颗芯片。

5.1 谷歌 TPU 概述

5.1.1 什么是 TPU？

TPU 是谷歌专为深度学习和 AI 任务设计的专用集成电路（ASIC）。和 CPU、GPU 这些"多面手"不同，TPU 从第一天起就只有一个使命：高效执行神经网络计算。

下表展示了历代 TPU 芯片的参数演进：

从 28nm 到 7nm，从 75W 到 300W，从 92 TOPS 到 275 TOPS——每一代都在性能和能效上实现了质的飞跃。

5.1.2 历代 TPU 产品

除了单芯片的迭代，从 TPU v2 开始，谷歌还推出了TPU Pod——由众多 TPU 单元构成的超大规模计算系统，专为处理大量并行计算任务而设计。

有意思的是，谷歌还把这些技术"下放"到了消费电子领域：2017 年的 Pixel 2/3 搭载了 Pixel Visual Core，之后的 Pixel 4 搭载了基于 Edge TPU 架构的 Pixel Neural Core。到了 2023 年的 Tensor G3，谷歌更是把生成式 AI 塞进了手机芯片。

5.1.3 TPU 架构演进的核心思路

每一代 TPU 都有鲜明的设计主题：

  • TPU v1：专为推理而生，INT8 量化 + 脉动阵列，用最少的功耗做最多的矩阵乘法
  • TPU v2：进军训练，引入 BF16 精度、HBM 高带宽内存、芯片间互联
  • TPU v3：液冷加持，MXU 翻倍，Pod 规模从 256 芯片扩到 1024 芯片
  • TPU v4：7nm 工艺，Sparse Core 加速 Embedding，3D Torus + 光路交换机打造 exaFLOPS 集群

贯穿始终的设计哲学：简单、专用、高效。TPU 的控制逻辑仅占芯片面积的 2%（远低于 CPU 和 GPU），把宝贵的面积全留给了计算。

5.2 谷歌 TPU v1 脉动阵列

如果说前几代 TPU 中有什么设计最让人拍案叫绝，那一定是脉动阵列（Systolic Array）。它是 TPU 的"心脏"，也是理解专用 AI 芯片为何高效的关键。

5.2.1 TPU v1 芯片架构

TPU v1 的几个关键元素：

  • DDR3 DRAM / Weight FIFO：模型权重通过 PCIe 从主机"预加载"到芯片的 DDR3 内存中，再传输到 Weight FIFO 供 MXU 使用。
  • MXU（矩阵乘法单元）：256×256×8 bit 的脉动阵列，每个时钟周期输出 256 个 16 bit 计算结果。这是"TPU 的心脏"。
  • 累加器（Accumulator）：4 MB 的 32 bit 累加单元，存储 MXU 的计算结果。
  • 控制指令（Control）：四级流水线，12 条 CISC 指令，包括 Read_Host_Memory、Read_Weight、Matrix_Multiply/Convolve、Activate、Write_Host_Memory 等。

注意芯片布局图中那两大块：本地统一缓冲器（Local Unified Buffer） 和 矩阵乘法单元（MXU），它们占据了芯片绝大部分面积。而控制器只占 2%——这就是"专用"的力量。

5.2.2 TPU v1 的三大优化特性

特性一：低精度（INT8 量化）

神经网络在推理时并不需要 FP32 那样的高精度。TPU 通过**量化（Quantization）**技术，将权重和激活值从 FP32/FP16 转换为 INT8，在保持模型精度的同时，大幅优化了存储和计算效率。

量化后的 INT8 虽然无法维持单个数据点的超高精度，但得益于神经网络的泛化能力，整体数据分布保持大致准确，推理速度却大幅提升。

特性二：脉动阵列 & MXU

这是 TPU 最核心的设计。传统的 CPU 计算流程是"读寄存器 → ALU 计算 → 写回寄存器"，数据每次计算都要经过寄存器。而在脉动阵列中，数据在 ALU 之间"流动"，一次读取就可以被多次复用。

左图是 CPU：数据 ALU 计算前后都要经过寄存器。右图是 TPU：数据在 ALU 之间更快地流动且复用，就像心脏泵血一样有节奏——这就是"Systolic"（心脏收缩）名字的由来。

TPU v1 的 MXU 本质上是一个 256×256 = 65536 个 ALU 的超大脉动阵列，一个时钟周期就能处理 65536 个 INT8 乘加运算。乘以 700 MHz 的频率，每秒可处理约 92 TOPS 的计算量。

特性三：专用硬件

CPU 和 GPU 为了当好"多面手"，设计极其复杂，导致行为难以预测、时延不稳定。TPU 则不同——所有设计只为一个任务：神经网络推理。简单、可控、高效。

5.2.3 脉动阵列原理详解

与其说"脉动阵列的原理是什么"，不如问：为什么这种架构比传统方式有这么大的优势？

在传统计算方式（左图）中，数据每计算一次就要存储一次，下一次调取时又要从存储器中重新获得——反复的"存取存"带来了巨大开销。而在脉动阵列（右图）中，数据是在通过所有 PE（处理单元）计算之后才被存储，大大减少了数据被访问的次数。

下面以两个 3×3 矩阵为例，图解脉动阵列的计算过程：

注意数据以阶梯状输入阵列——矩阵 A 和矩阵 B 在不同的行列维度上被分开，因为矩阵乘法需要 A 的每一行去点乘 B 的每一列。

数据一波一波流入 MXU，计算结果被存放在寄存器中累加和输出。整个过程就像心脏泵血，每个时钟周期进行一次庞大的计算，并流入下一个需要这个结果的地方。

核心洞察：因为每个时钟周期可以进行 256×256 次计算，且计算结果被串联复用，TPU 能够在更少的内存访问次数下完成更高的计算量——这就是 TPU 能效比领先的根本原因。

谷歌研究员用 Roofline 性能模型在 CPU（Haswell E5-2699 v3）、GPU（NVIDIA K80）和 TPU v1 上做了对比实验，基于 MLP、LSTM、CNN 三类共 6 个模型：

指标 Haswell CPU NVIDIA K80 GPU TPU v1
Die 面积 (mm²) 662 561 更小
制程 (nm) 22 28 28
TDP (W) 145 150 75
TOPS (INT8) 2.6 - 92
片上缓存 51 MB 8 MB 28 MB
整机功耗 (W) 504 1838 861

TPU v1 的压倒性优势：

  • 计算性能：92 TOPS，远超同时期的 CPU 和 GPU
  • 片上缓存：大缓存减少片外访问，对抗当时内存访问速度慢的瓶颈
  • 量化：首次在推理场景引入 INT8，利用神经网络鲁棒性，支持更大 batch

GPU 的目标是降低时延，TPU 的目标是提高吞吐——目标不同，设计思路自然不同。

5.3 谷歌 TPU v2 训练芯片

2017 年，谷歌发布了 TPU v2，官方的描述是"用于训练神经网络的特定领域超级计算机"。同一年，谷歌还发表了那篇改变 NLP 格局的论文 Attention is All You Need——而 TPU v2，就是站在 Attention 背后的超级计算机。

5.3.1 训练 vs 推理：为什么需要 TPU v2？

推理只需要一次前向传播，对精度要求相对较低（INT8 即可）。但训练完全不同：

  • 更复杂的数据并行：一个模型需要迭代百万次，所有参数在每个迭代中都要调整，需要协调跨集群并行计算
  • 更高的计算复杂度：反向传播需要对每个权重参数和输入计算偏导数
  • 更大的内存需求：临时变量（激活值、优化器状态）会将模型大小膨胀至原始大小的 8~9 倍
  • 更高的精度要求：INT8 在训练中会导致不稳定和极大随机性，必须使用 FP16/BF16/FP32

5.3.2 TPU v1 → TPU v2 的四大改动

改动一：存储位置合并

TPU v1 有 Accumulator（存矩阵乘积）和 Activation Storage（存激活输出）两个独立缓冲区。TPU v2 将它们合并为 Vector Memory，更像传统架构中的 L1 Cache，提升了可编程性。

改动二：向量单元替代激活管线

TPU v1 的 Activation Pipeline 是针对特定激活函数场景的专用 ALU，无法满足训练需求。TPU v2 将其替换为通用的 Vector Unit，能处理各种向量激活函数。

改动三：MXU 连接方式调整

TPU v1 中 MXU 与向量内存直接相连，TPU v2 中 MXU 改为与向量单元连接——所有数据传输和计算都由向量单元分发。这样 MXU 成为向量单元的协处理器，对编译器和编程人员更友好。

改动四：DDR3 → HBM

训练过程中产生的中间变量和权重远多于推理，DDR3 的回写速度根本无法满足。TPU v2 将 DDR3 替换为 HBM（高带宽内存），读写速度提升了 20 倍。

5.3.3 TPU v2 计算核心

TPU v2 的核心由三部分组成：

  • 标量单元（Scalar Unit）：处理计算的起点，从指令存储器取出 322 bit 宽的 VLIW（超长指令字），执行标量操作，并将向量指令和矩阵指令分发给对应单元。
  • 向量单元（Vector Unit）：128 个向量通道，每个通道配备双发射 32 bit ALU。每个通道还有 8 路子通道，特别适用于批量归一化操作。

  • MXU：TPU v2 选择 4 个 128×128 的 MXU 而非 1 个 256×256 的 MXU。谷歌的模拟器显示，4 个 128×128 的 MXU 卷积模型利用率达 37%~48%，明显高于单个 256×256 的 22%~30%。这是基于利用率和面积效率的综合考量。

TPU v2 的 MXU 使用 BF16 格式进行乘法运算——BF16 保留与 FP32 相同的指数范围（8 位指数），但尾数仅 7 位，在 FP32 的臃肿和 FP16 的低范围之间找到了完美平衡。

5.3.4 芯片互联：Pod 的基石

TPU v1 是单芯片推理协处理器，没有互联能力。TPU v2 则不同——它内置了互联路由模块（Interconnect Router），实现了 2D Torus（二维环面） 连接。

每个芯片有 4 个自定义的**核间互联（ICI）**链接，每个方向带宽 496 Gb/s。这使得芯片之间可以直接连接，256 个 TPU v2 芯片组成一个 16×16 的二维环网，双向带宽高达 15.9 Tb/s——是传统 InfiniBand 集群交换机的 2.5 倍，同时还省去了网卡、交换机和通过 CPU 主机通信的时延成本。

看这张平面布局图：2 个 TPU Core 一上一下，互联路由模块位于正中央，2 个 MXU 分别在顶部和底部中心——大部分面积都给了计算核心和互联。

5.4 谷歌 TPU v3 Pod 服务器

5.4.1 TPU v3：TPU v2 的"增强版"

TPU v3 本质上就是 TPU v2 的增强版。相比 v2：

  • 时钟频率提升约 1.35 倍
  • ICI 带宽和内存带宽提升 30%
  • MXU 数量从 2 个翻倍到 4 个
  • 峰值性能提高 2.7 倍

但最关键的改进其实是散热——TPU v3 用液冷代替了风冷：

TPU v2 板卡上 4 个芯片全靠风冷，而 TPU v3 使用液冷系统管理 4 个芯片的温度。正是这个液冷系统为 TPU v3 提供了 1.6 倍的功率余量，让它能在仅增大 10% 体积的情况下塞进 4 个 MXU 和更大的 HBM。

5.4.2 Pod：从"芯片"到"超级计算机"

谷歌官方对 Pod 的定义很简单："TPU Pod 是一组通过专用网络连接在一起的连续 TPU 单元。“但它的意义远不止于此。

TPU v2 Pod：每个机柜含 64 个 CPU 板和 64 个 TPU 板，共 128 个 CPU 芯片和 256 个 TPU v2 芯片。CPU 集群在两侧，TPU 集群在中间——这种紧凑排列是为了避免信号衰减（BlueLink 或 OPA 的铜缆和光纤长度不能太长）。

TPU v3 Pod：规模比 v2 Pod 大了整整 4 倍——1024 块 TPU v3 芯片，算力超过 100 PFLOPS。更多的铜管、更多的电缆、更大的规模。

5.4.3 Pod 通信方式

在分布式训练中，同步训练是主流——所有节点完成本轮通信后再进行下一轮计算。系统的性能受限于最慢的计算节点和最慢的网络消息。

谷歌在 TPU v2/v3 Pod 中采用 2D Torus 网络结构：

每个 TPU 芯片与相邻的 TPU 芯片直接连接，形成一个二维平面网络。这种设计减少数据在芯片间传输时的通信时延和带宽瓶颈。基于此，谷歌优化了同步训练中的 All-Reduce 操作，在同等资源条件下性能超越了异步 SGD。

TPU v2 Pod（256 芯片）算力 11.5 PFLOPS，TPU v3 Pod（1024 芯片）算力 100 PFLOPS。两者的核心架构没有本质区别，主要提升的是规模化能力。

5.5 谷歌 TPU v4 与光路交换

TPU v4 和 TPU v3 之间隔了整整 4 年。这 4 年间，英伟达迭代了三代架构（Volta → Ampere → Hopper），PyTorch 取代 TensorFlow 成为首选框架，业界对大算力的饥渴只增不减。TPU v4 正是在这样的背景下诞生的。

5.5.1 TPU v4 的重大升级

每个 TPU v4 包含 2 个 Tensor Core，每个 Tensor Core 由 6 个单元组成：4 个 MXU + 1 个标量单元 + 1 个向量单元。2 个 HBM 内存模块分别置于 Tensor Core 左右两侧以降低电缆时延。

关键升级点：

  • 制程工艺：16nm → 7nm，晶体管数量是 TPU v3 的 4 倍
  • MXU 数量：再次翻倍
  • 内存：片上内存增加 9 倍达 244 MB，HBM 带宽 1.2 TB/s（提升 33%）
  • Sparse Core：全新引入，专为稀疏计算优化
  • 3D Torus：首次引入，支持多达 4096 个 TPU v4 核心互联

5.5.2 Sparse Core：Embedding 加速利器

在 NLP 和推荐系统中，Embedding 层处理的是极度稀疏的离散特征——输入矩阵中存在大量 0，传统矩阵乘法会白白浪费计算资源。

Sparse Core 包含 16 个计算瓦片（Tiles），每个瓦片有关联的 HBM 通道。每个瓦片包含：

  • Fetch Unit：从 HBM 读取激活函数和参数
  • scVPU：可编程的 8 路 SIMD 向量处理单元
  • Flush Unit：反向传播时将更新的参数写回 HBM
  • 2.5 MB Spmem：稀疏向量内存

此外还有 5 个跨通道单元执行嵌入操作。Sparse Core 可将依赖 Embedding 的模型加速 5~7 倍，而面积和功耗仅占 5%。

5.5.3 TPU v4 Pod：3D Torus + 光互联

TPU v4 Pod 将 64（4×4×4）个 TPU v4 芯片组成一个立方体（Cube），再将 64 个 Cube 通过**光互联（OCS）**连接在一起，形成共有 4096 个 TPU v4 芯片的超级计算机，总计提供 1.126 exaFLOPS（BF16）的峰值算力。

在 3D Torus 网络中，每个节点通过 3 个维度（X-Y-Z）与相邻节点相连，形成三维网格结构。每个节点都与上下、左右、前后的 6 个节点互联。

对比一下：英伟达 DGX SuperPod 最多连接 256 个 H100 芯片，且需要约 568 个 InfiniBand 交换机。而 TPU v4 Pod 用 4096 个芯片 + 48 个 OCS 光交换器就搞定了——OCS 的成本不到系统成本的 5%，功耗不到系统功率的 3%。

5.5.4 光路交换机（OCS）

OCS 是 TPU v4 Pod 的"秘密武器”。传统交换机的流程是"光→电→光"，两次转换造成大量性能损耗。而 OCS 提供光到光的直接转换，节省了大量电能，更重要的是解决了时延问题。

Palomar OCS 的核心是 2D MEMS 反射镜阵列——通过控制微反射镜的位置来调整光路，实现光路切换。它具有低损耗、低切换时延（毫秒级别）、低功耗、低成本等优点。

5.5.5 TPU v4 的优缺点

优点：

  • 低时延：3D Torus 相邻节点间距离短且可直接连线
  • 低网络成本：3D Torus 拓扑的交换机、线材、连接器保有量低于 Clos 拓扑
  • 路由可重配：OCS 支持动态可重配路由，更易隔离/下线故障节点
  • 更好的集群布局：物理相邻=逻辑相邻，密集 I/O 落在局部域

缺点：

  • 系统成熟度低：Clos 拓扑具备非阻塞特性，而 3D Torus 无法保证
  • 拓扑僵硬：扩容 Clos 只需增加叶交换机，扩缩 3D Torus 则需重新配置整个拓扑
  • 负载均衡挑战：Clos 的替代路径数量更多

小结与思考

这一章从 TPU v1 的脉动阵列一路讲到 TPU v4 的光路交换，信息量确实不小。让我们用几个问题来收尾：

  • TPU 和 GPU 的本质区别是什么？ GPU 追求通用性，TPU 追求专用性。GPU 用多级缓存和计算核心降低时延，TPU 用脉动阵列和流水线提高吞吐——目标不同，设计哲学自然不同。

  • 脉动阵列为什么高效？ 数据在 PE 之间"流动"复用，而不是每次计算都要"存-取-算"。对于矩阵乘法这种需要大量数据复用的场景，这大幅减少了内存访问次数。

  • 从 v1 到 v4，不变的"道"是什么？ 控制逻辑只占 2% 的芯片面积，其余全部给计算和内存——简单就是高效。同时，每一代都在强化芯片间的互联能力，因为单芯片的算力终究有限。

  • 为什么光互联是未来？ 当芯片数量达到 4096 级别，电互联的时延和功耗已经成为瓶颈。光互联的"光到光"转换，省去了光电转换的损耗，是通往 exaFLOPS 时代的必经之路。

正如书中所言：TPU 的设计强调算法与芯片之间的紧密协同，以及对新数据格式和稀疏计算的支持——这预示着 AI 计算集群的重要性日益增加。
