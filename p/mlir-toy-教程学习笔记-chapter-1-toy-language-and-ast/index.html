🔍 核心概念解析

概念 1：Toy 语言

定义： Toy 是一个基于张量的简单语言，用于演示编译器前端到 MLIR 代码生成的完整流程。

语言特性：

  • 数据类型：仅支持 64 位浮点数（double）
  • 张量维度：支持 rank <= 2 的张量
  • 值语义：所有值都是不可变的（immutable），每个操作返回新分配的值
  • 内存管理：自动管理内存释放
  • 类型推断：通过类型推断进行静态类型检查
  • 泛型函数：函数参数是未定秩的（unranked），在调用点根据实际类型进行特化

def main() {
  # 定义变量 a，形状从字面量推断为 <2, 3>

  print(transpose(a) * transpose(b));
}

个人理解：

Toy 语言设计得非常简单，专注于展示编译器的核心概念。它不支持循环、条件语句等控制流，但足以演示张量操作和函数定义。这种简化使得我们可以专注于编译器前端的实现，而不是被复杂的语言特性分散注意力。

概念 2：抽象语法树（AST）

定义： AST 是源代码的树形表示，每个节点代表源代码中的一个语法结构。

作用：

  • 将源代码的结构化表示与具体语法分离
  • 便于后续的语义分析、类型检查和代码生成
  • 提供统一的接口遍历和操作语法结构

关键点：

  • 节点类型：使用枚举 ExprASTKind 标识不同的 AST 节点类型
  • 继承体系：所有表达式节点继承自 ExprAST 基类
  • 内存管理：使用 std::unique_ptr 管理节点生命周期
  • 位置信息：每个节点保存源代码位置（Location），用于错误报告
  • LLVM RTTI：使用 classof 方法实现类型安全的类型转换

// AST 节点基类
class ExprAST {
public:
  enum ExprASTKind {
    Expr_VarDecl, Expr_Return, Expr_Num,
    Expr_Literal, Expr_Var, Expr_BinOp,
    Expr_Call, Expr_Print,
  };
  
  ExprAST(ExprASTKind kind, Location location)
      : kind(kind), location(std::move(location)) {}
  
  ExprASTKind getKind() const { return kind; }
  const Location &loc() { return location; }
  
private:
  const ExprASTKind kind;
  Location location;
};

个人理解：

AST 的设计体现了面向对象和现代 C++ 的最佳实践。使用智能指针避免了内存泄漏，使用 RTTI 实现了类型安全的访问模式。这种设计既保证了代码的安全性，又保持了良好的可扩展性。

概念 3：词法分析器（Lexer）

定义： 词法分析器将源代码字符流转换为 Token 流，识别关键字、标识符、数字、操作符等。

作用：

  • 识别语言的基本词汇单元（Token）
  • 过滤空白字符和注释
  • 跟踪源代码位置信息

关键点：

  • Token 类型：使用枚举定义所有 Token 类型
  • 位置跟踪：记录每个 Token 的文件名、行号、列号
  • 抽象基类：Lexer 是抽象基类，LexerBuffer 是具体实现
  • 前瞻字符：使用 lastChar 实现单字符前瞻，用于识别多字符 Token

概念 4：递归下降语法分析器（Parser）

定义： 递归下降语法分析器通过递归函数调用实现语法规则，将 Token 流转换为 AST。

作用：

  • 根据语法规则解析 Token 流
  • 构建 AST 表示
  • 报告语法错误

关键点：

  • 递归函数：每个语法规则对应一个解析函数
  • 运算符优先级：通过 getTokPrecedence() 处理运算符优先级
  • 错误处理：使用 parseError() 统一处理解析错误
  • 左结合性：通过递归调用实现左结合的二元运算符

// 1. Token 枚举定义
enum Token : int {
  tok_semicolon = ';',        // 单字符 Token 直接使用 ASCII 值
  tok_eof = -1,                // 特殊 Token 使用负值
  tok_identifier = -5,
  tok_number = -6,
};

// 2. 抽象基类 Lexer
class Lexer {
public:
  Token getCurToken() { return curTok; }
  Token getNextToken() { return curTok = getTok(); }
  void consume(Token tok) {
    assert(tok == curTok && "consume Token mismatch expectation");
    getNextToken();
  }
  
private:
  virtual llvm::StringRef readNextLine() = 0;  // 纯虚函数
  Token getTok();  // 核心词法分析逻辑
};

// 3. 具体实现 LexerBuffer
class LexerBuffer final : public Lexer {
  llvm::StringRef readNextLine() override {
    // 从内存缓冲区读取一行
  }
};

分析：

  • Token 设计：单字符 Token 直接使用 ASCII 值，特殊 Token 使用负值，节省空间
  • 抽象设计：Lexer 是抽象基类，支持不同的输入源（文件、内存等）
  • 位置跟踪：通过 curLineNum 和 curCol 跟踪当前位置
  • 前瞻机制：使用 lastChar 实现单字符前瞻

学习笔记：

Lexer 的设计采用了模板方法模式，将通用的 Token 识别逻辑放在基类，将输入源相关的逻辑放在子类。这种设计使得 Lexer 可以轻松支持不同的输入源，同时保持代码的简洁性。

// 1. 解析主入口
std::unique_ptr<ModuleAST> parseModule() {
  lexer.getNextToken();  // 初始化词法分析器
  std::vector<FunctionAST> functions;
  while (auto f = parseDefinition()) {
    functions.push_back(std::move(*f));
    if (lexer.getCurToken() == tok_eof)
      break;
  }
  return std::make_unique<ModuleAST>(std::move(functions));
}

// 2. 运算符优先级处理
std::unique_ptr<ExprAST> parseBinOpRHS(int exprPrec,
                                       std::unique_ptr<ExprAST> lhs) {
  while (true) {
    int tokPrec = getTokPrecedence();
    if (tokPrec < exprPrec)
      return lhs;
    
    int binOp = lexer.getCurToken();
    lexer.consume(Token(binOp));
    auto rhs = parsePrimary();
    
    // 处理右结合性：如果下一个运算符优先级更高，先解析它
    int nextPrec = getTokPrecedence();
    if (tokPrec < nextPrec) {
      rhs = parseBinOpRHS(tokPrec + 1, std::move(rhs));
    }
    
    lhs = std::make_unique<BinaryExprAST>(..., binOp, 
                                          std::move(lhs), std::move(rhs));
  }
}

// 3. 错误处理
template <typename R, typename T, typename U = const char *>
std::unique_ptr<R> parseError(T &&expected, U &&context = "") {
  auto curToken = lexer.getCurToken();
  llvm::errs() << "Parse error (" << lexer.getLastLocation().line << ", "
               << lexer.getLastLocation().col << "): expected '" << expected
               << "' " << context << " but has Token " << curToken << "\n";
  return nullptr;
}

分析：

  • 递归下降：每个语法规则对应一个递归函数
  • 优先级爬升：通过比较运算符优先级实现正确的结合性
  • 错误处理：统一的错误报告机制，包含位置信息
  • 模板错误处理：使用模板函数减少代码重复

学习笔记：

Parser 的实现展示了递归下降解析器的经典模式。运算符优先级的处理是其中的难点，通过"优先级爬升"算法可以正确处理左结合和右结合的运算符。错误处理机制的统一设计使得错误信息更加友好和一致。

🤔 问题与思考

遇到的问题

  1. 问题描述：理解运算符优先级的处理机制
    • 现象：parseBinOpRHS 函数的逻辑比较复杂
    • 原因：需要同时处理运算符优先级和结合性
    • 解决方案：通过"优先级爬升"算法，比较当前运算符和下一个运算符的优先级来决定解析顺序

深入思考

  • 问题 1：为什么使用 std::unique_ptr 而不是原始指针？

    • 思考：现代 C++ 强调资源管理自动化
    • 答案：std::unique_ptr 提供了自动内存管理，避免了内存泄漏，同时明确了所有权语义

  • 问题 2：为什么 AST 节点要保存位置信息？

    • 思考：位置信息对于错误报告和调试非常重要
    • 答案：当解析或后续处理出错时，可以准确报告错误位置，提高开发体验

与其他章节的联系

  • 与 Chapter 2 的关系：Chapter 2 将展示如何将 AST 转换为 MLIR IR
  • 为后续章节做的铺垫：
    • AST 结构为 MLIR 代码生成提供了输入
    • 位置信息在 MLIR 中也会被保留，用于错误报告
    • 解析器的错误处理机制在后续章节中会扩展为语义错误检查
