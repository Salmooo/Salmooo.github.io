第5章: Which Platform? — Build Your Own

Jun 05, 2026
4 minute read

章节概览

第4章告诉我们"需要什么资源"；第5章回答"如何自己造出承载这些资源的硬件平台"。

本章是全书硬件工程味道最浓的一章——从 PCB 拓扑、连接器选型、时钟延迟匹配（精确到 0.44ns）、电源树设计、地弹抑制、散热方案到 FPGA 配置机制和信号完整性，全面覆盖自建原型平台的每一个工程决策。

核心张力贯穿全文：为目标项目定制的专用性 vs 为未来项目预留的通用灵活性。作者坦率指出：“在作者的经验中，大多数自建平台从未被复用于后续项目”——不是因为不想，而是因为时钟资源、互联和电压域的灵活性不足以适配新设计。本章最后探讨了如何在组织层面建立标准化平台生态体系来打破这个困境。

核心知识点

  • “Pins-in / Pins-out” 拓扑：在 SoC 引脚处做物理分界，初版芯片到达后可直插替换 FPGA 板——软件在数小时内即可跑起来
  • 母板固定资源 + 子板项目专用外设 = 复用性与性能之间的最佳折中。固定外设会永久占用 FPGA IO 引脚（最宝贵的资源），即使不使用也无法释放
  • 互联是比逻辑更稀缺的资源：64 位 AHB 总线实际需要 200+ 引脚（地址+写数据+读数据+控制信号），即使复用也需 130+ 引脚
  • 关键教训：如果一颗 FPGA 不需要连接该总线，其 130+ IO 引脚只能设为高阻态——永久浪费
  • HAPS 时钟延迟匹配的量化模型：定义两个延迟量子 X（stub 延迟，0.44ns）和 Y（标准同轴电缆延迟，1.45ns），任意 FPGA 时钟输入延迟 = 2X+Y = 2.33ns
  • 电源分配的核心原则：每个子板独立供电，不要从主板"借电"——Piggy-Back 供电是灾难的来源
  • 地弹（Ground Bounce）机制：大量 IO 同步翻转 → 大地电流经地引脚回流 → 地参考瞬时抬升 → 输入引脚误判逻辑电平。应对：低阻抗地平面 + 默认最小驱动电流而非 24mA
  • 散热不可忽视：高利用率 + 高时钟 + 封闭机箱 = 芯片损坏。FPGA 内置系统监控器（温度 ±4°C，电压 ±1%）是免费的保护机制
  • System ACE + CompactFLASH：最流行的非专家友好配置方案——CF 卡可存多个配置+文档，终端用户通过开关或远程选择加载哪个
  • 自建标准化平台需要组织级投入：开发协调、库存管理、文档、培训、技术支持——本质上是一项"公司内部的小生意"

详细笔记

5.1 平台的最佳形态

四个维度评估平台拓扑：尺寸/外形、模块化、互联、灵活性。

5.1.1 尺寸和外形

脐带线缆（Umbilical Cable）拓扑：FPGA 平台通过线缆连接到目标系统的 SoC 插座。优势：易于向外部合作伙伴演示。但需要考虑 FPGA IO 引脚是否有足够的电流驱动能力来驱动线缆负载。

更常见的反向做法：初版芯片插接到 FPGA 原型平台预先准备好的位置——这需要一种关键的物理拓扑：

这种预规划使芯片回片后集成时间减少了几个数量级。

5.1.2 模块化

核心权衡：全部元件放一块板（性能更好）vs 不同元件分板放置（灵活性更好）。

母板+子板 折中方案：

  • 母板：FPGA + 全局资源（每个项目必用）
  • 子板：项目专用外设（随项目更换）
  • 后续项目只需重新设计 1-2 块子板，而非整个平台

5.1.2.1 母板+子卡 vs “全部上板”

一个关键的陷阱示例（图48）：

外设在母板上直接连接到 FPGA 引脚。如果某个设计不用该外设，这些 IO 引脚仍然被占用——无法作任何其他用途。“如果不确定是否需要的，就放上去”——这种态度会严重限制母板上最宝贵的资源：FPGA 的 IO 引脚。

解决方案：外设放在独立子卡上，需要时接入，不需要时移除。

5.1.3 互联

总线宽度陷阱：64 位 AHB 总线实际包含：

  • 64 位地址
  • 64 位写数据
  • 64 位读数据
  • 10-20+ 控制和状态信号
  • 合计：200+ 连接

即使地址和数据复用，仍需要 130+ 全局引脚和走线。

固定全局总线的浪费（图49、图50）：一个 4 FPGA 系统中，只有 3 颗需要连接 AHB 总线，但第 4 颗 FPGA 的 130+ 引脚被硬连线到总线走线上——只能设高阻态，永久无法作其他用途。

两条路：

  • 专用互联（匹配 SoC 顶层互联 → 分区容易 → 后续项目复用困难）
  • 通用互联（设计无关 → 复用性好 → 分区时需要更多规划）

建议：用线缆或开关矩阵而非固定 PCB 走线来增加互联灵活性。

5.1.4 灵活性

一个刺眼的行业事实：

“在作者的经验中，大多数自建平台从未被复用于后续项目。不仅因为复用本就不是初衷，而且即使有复用意图，板子本身的资源或互联也无法支持。”

五大灵活性维度：

  1. FPGA 资源总量 — 能否扩展
  2. 外设选择 — 是否模块化
  3. 互联 — 是否固定
  4. 板级时钟资源 — 最常导致不可复用的元凶
  5. 电压域 — 新项目的 IO 电压需求可能完全不同

5.2 可测试性

板级可测试性 ≠ FPGA 内部设计插桩（第11章）。

自建平台必须能测试：

  • 所有互联的连续性（尤其是多连接器的模块化系统）
  • FPGA 引脚烧毁的诊断
  • 可测试性投入在平台故障时多次回报

5.3 板级时钟资源

核心概念：“Super-FPGA”

把整个板级系统看作一个"超级 FPGA"——各颗 FPGA 器件 + 连接它们的顶层资源。SoC 的时钟网络层级需要用板级资源来承载顶层。

“Super-FPGA” 需要复制的板级资源：

  • 板级 PLL / 时钟复用器 / 时钟分频器 / 倍频器 / 时钟同步电路

简化策略：用工具自动将 SoC 的门控时钟转为等效的全局 FPGA 时钟 + 时钟使能（第7章详述），减少时钟域总数。

时钟灵活性不足是第一大复用杀手：

“在作者的经验中，顶层时钟资源的限制是自建板在后续项目中不能被复用的最常见原因。”

5.3.1 板内外时钟延迟匹配——HAPS 的量化模型

这是本章最精彩的技术细节。

定义两个基本延迟量子：

$$ X = \text{stub delay（FPGA 引脚到附近连接器的延迟）} = 0.44\text{ns} $$$$ Y = \text{标准同轴电缆 delay（特定长度 + 高质量屏蔽 + 同轴连接器）} = 1.45\text{ns} $$

关键设计原则：

  • 所有同轴电缆统一类型和长度 → Y 值恒定
  • PCB 走线按规定长度做延迟匹配（可能需要蛇形走线，增加 PCB 层数）
  • 同一板族中 X 和 Y 值保持不变 → 允许新旧板混合使用

时钟源到任意 FPGA 时钟引脚的总延迟：= 2X + Y = 2.33ns

注意：收到的各 FPGA 时钟虽然互相同步，但与主时钟输入 F 不同步——需要 PLL 来补偿。

5.3.2 PLL（锁相环）

板级 PLL 的作用：

  • 消除跨板时钟网络中的插入延迟
  • 减少系统不同部分之间的时钟偏斜

关键要求：PLL 输出到各 FPGA 的 stub 长度和电缆长度严格匹配。

5.3.3 系统时钟生成清单

五大考量：

  1. 时钟来源：本地板上生成 / 外部输入 / FPGA 内部生成
  2. 时钟频率：估计 FPGA 时钟频率范围，支持精细粒度任意频率生成
  3. 容忍偏斜：FPGA 间同步——所有 FPGA 在可接受偏斜范围内收到时钟
  4. 板间同步：大型多板系统——所有时钟到达所有 FPGA 的偏斜可接受
  5. 高速时钟：引脚复用需要的高速时钟两端必须同步（CPM 工具支持）

自建板应包含的时钟要素：

  • 板级时钟合成器：PLL + 晶体振荡器（可插拔）→ 多时钟系统需多路独立的合成器
  • 输入时钟源选择器：本地/外部/FPGA 产生的多路选择 MUX → 手动开关或软件控制
  • 时钟分配网络：板上和板间延迟均衡化
  • 外部时钟源：板间传播延迟需通过相位偏移补偿 → 多级 PLL 级联可能导致不稳定和失锁
  • 高速多路复用时钟：信号复用到单引脚需要的高速同步时钟
  • 时钟缩放：分频器（FPGA 内部）+ PLL（板级）

5.4 时钟控制与配置

复杂的多 FPGA 板可能需要大量开关来覆盖所有时钟选项。替代方案：可编程开关由板载微控制器或外部主机控制（第6章详述商业平台的软件控制方案）。

5.5 板级电压域

SoC 的内部电压岛（通过 UPF 文件描述，由 EDA 工具推断）一般不出现在 RTL 中，不会被映射到 FPGA 原型中。如果 RTL 中包含了电压控制逻辑和 retention 寄存器，需要将其固定到非活跃状态（第7章）。

外部电压需求必须正确处理：FPGA IO 引脚按 Bank 分组，共享不同电压阈值，由不同电压环供电。

电压不灵活的代价：电压而非性能或互联成为分区决策的支配因素 → 其他优先级被妥协。新的电压需求是已有板子不能复用的另一大原因。

HAPS-64 的电压配置工具（图53）：将 FPGA 引脚和子卡连接器分组到不同区域，每个区域通过下拉菜单设置电压（包括"OFF"选项），配置工具控制板上的可编程开关设置参考电压和轨电压。

5.6 电源供给与分配

核心原则：“每个板独立供电”

如图54（反例）：子板从主板取电→子板合并电流超过主板供电能力→烧毁主板供电电路和走线。

如图55（正例）：一个公共外部电源为所有板供电→各板独立取电。

关键警告：

“即使是安装了自己 FPGA 的主要子系统板，从主板供电也是坏实践。FPGA 上电和配置时期（甚至 live reset 期间）的瞬时大电流，可能超出板间连接器的承载能力。‘瞬间低压掉电’（Momentary Brown-Out）是原型项目后期最难诊断的问题之一。”

保护措施：每个电源出口处加可复位保险丝（Resettable Fuse），连接器引脚根据预期电流适当选择规格。

5.6.1 板级电源分配

三种电压需求：

电压类型 用途 特点
Core Voltage FPGA 内部逻辑 最低，制造商决定
IO Voltage FPGA 引脚 按 Bank 分组，可变；多 IO 电压 = 多"功率岛"（Power Island）→ 复杂化 PCB 布局
IO Current 外部设备驱动 超出 24mA 需加外置缓冲器（放在子卡上）

功耗估算：取决于切换速率和 FF 使用量 → 统计性估算 → 保守为好。供应商提供功耗估算工具。

电源监控：由于核心和部分 IO 电压显著低于过去器件，变化容限比例缩小。强烈推荐电源监控电路。

附加参考电压：为外部 ADC/DAC 等模拟电路预留可编程参考电压供电。

5.6.2 电源分配的物理设计

FPGA 的电流尖峰问题：

高速时钟 + 大量 FF 同步翻转 → 时钟边沿产生巨大电流尖峰 → 如果供电不足 → 电源电平下降或电源线上噪声 → 不可靠运行。

设计规范：

  • 低阻抗电源平面 + 连续充足铜箔
  • 低阻抗地回路同样重要
  • 大容量电容（100μF）：平滑慢速电流浪涌
  • 高频小电容（0.1μF）：平滑快速电流尖峰
  • 两种电容都尽可能靠近 FPGA 电源引脚

地弹（Ground Bounce）机制与对策：

  • 每 N 个 IO 引脚共享一个地引脚（N 取决于 FPGA 型号）
  • 大量 IO 同时同向翻转 → 大回流电流 → 地参考瞬时抬升 → 输入引脚误判逻辑
  • 对策：给 IO 间的地引脚提供低阻抗接地 + 默认使用最小驱动电流而非全 24mA，在保证信号质量的同时减少回流

5.7 系统可靠性管理

5.7.1 电源监控

每路供电电路持续监测电压 → 超出范围时以下动作可选其一或多选：

  1. 可见指示（LED），大系统需链接到主机监控程序（尤其远程或封闭安装时）
  2. 触发全局复位减少工作电流
  3. 取消所有 FPGA 配置→ 最小电流消耗 + 远程用户可见的故障信号

5.7.2 温度监控与管理

热问题三要素：大 FPGA + 高利用率 + 高时钟 + 封闭机箱 = 过热损坏风险。

FPGA 系统监控器（免费且强大的内置保护机制）：

  • 监测芯片平均结温（±4°C 精度）
  • JTAG 端口可读取（无需实例化）或通过设计内部实例化模块访问
  • 可设温度报警 → 超出阈值时的自动操作

过热响应链：

  1. 可见报警（LED）
  2. 开启风扇
  3. 系统进入复位状态
  4. 取消所有 FPGA 配置（最小化功耗）

CONFPRO 工具（图57）：运行在 HAPS 板 MicroBlaze CPU 上的监控程序 → 读取 6 颗 Virtex-6 的系统监控器 → 控制各 FPGA 风扇。同一微控制器还控制 FPGA 时钟、复位和配置。

板级温度传感器（在靠近 FPGA 等热区放置）→ 可编程报警输出 → 与片上传感器组合 → 建立完整监控系统。

5.7.3 FPGA 散热

  • 散热片：每颗 FPGA 标配，自粘式 → 简单易装
  • 风扇：散热片不够时加装（+ 散热片组合更好）→ 板上预留风扇电源和连接器
  • 风扇不需常开 → 用 FPGA 温度监控器的闭环控制

“长期无散热片/无风扇运行的 FPGA 板会受到温度应力，可靠性降低。”

5.8 FPGA 配置

FPGA 是 SRAM 易失性器件 → 每次上电必须重新配置。

配置模式：并行/串行、主/从、JTAG。

模式 特点
Master FPGA 控制配置过程（上电或触发引脚触发）
Slave 外部设备控制配置接口
Serial 比并行慢但占用信号少
JTAG 覆盖所有其他模式——最常见的调试下载方式

5.9 信号完整性

时钟偏斜的两大来源：源信号抖动 + 噪声电源和信号环境。

最小化负面影响的要点：

  • 所有时钟信号阻抗一致性 — 关键
  • 合理布线减少串扰
  • 良好稳压电源 + 低阻抗路径
  • 并行数据和时钟路径的走线长度和延迟匹配

5.10 全局启动与复位

多 FPGA 原型的核心要求：所有 FPGA 必须在同一时钟边沿退出复位。

PLL 锁定等待：系统启动必须延迟到所有时序电路（板级 PLL + FPGA 内部 PLL）稳定锁定后。

全局复位电路：在全部 PLL 锁定之前保持整个系统在复位状态。板级设计时应确保外部 PLL 的锁定和复位信号能被至少一个 FPGA 的引脚访问。

5.11 鲁棒性（Robustness）

为原型走出实验室的物理保障（第12章深入讨论）：

  • 加强支撑结构（支柱/间隔柱/螺丝固定子卡）
  • 坚固外壳或机架安装
  • 外壳需保留测试点、扩展卡插槽、外设端口
  • 外壳需满足现场使用的电源和散热需求

5.12 建立内部标准化平台

Tile-Based 方法：大公司多站点间的标准化实践——统一外形、统一连接器、统一引脚定义。

标准化的代价：

  • 需要大量协调和管理
  • 新板开发面临"留在标准内 vs 为满足当前需要而打破标准"的抉择
  • 技术换代时（如总线加宽超出标准连接器容量），遗留体系成为障碍
  • 连接器是关键：不仅信号数要够，电气特性要好，质量要高，物理上要坚固——“选连接器看价格是长期的假节约”

标准化平台四大基础设施：

维度 内容
开发 协调多站点原型专家、维护可用板卡目录、规范/文档/路线图
分配 快速从现有板卡组装平台、制造团队需要看到未来项目预测
文档 标准本身的规范文档、合规认证、每块板的完整使用文档、培训材料/参考设计
支持 多站点多项目并行支持、现场人员、集中化实验室外包原型服务

一个有趣的现象：有些集中化的原型设计服务团队不自己造板，而是从商业供应商处组装平台。——引出第6章：Ready-Made 平台。

个人总结

这一章读下来最大的感受是：FPGA 原型验证不仅需要 SoC/RTL 知识，还需要一个真正硬件工程师的全部技能。

本章暴露了一个在许多 EDA/软件背景读者可能没有预料到的现实：原型平台的硬件设计复杂性不亚于原型验证本身的工程工作量。从时钟延迟匹配到纳秒级，到地弹机制的理解和抑制，到每颗 FPGA 旁边放两个特定值的电容器——这些都是通过痛苦经验积累的工程准则。

关于平台复用性的行业坦白也值得深思。作者直言"大多数自建平台从未被复用"——这对管理者而言是一个关键的采购决策信号。如果你只做一次原型，自建可能是对的。但如果有第二个项目甚至更多，第6章的商业平台就值得认真考虑。

最后，“Super-FPGA” 的思维模型很有启发。将整个板级系统视为一颗巨大的 FPGA，迫使你重新审视什么资源必须在芯片外（时钟树顶层、电源、互联），什么可以在芯片内（逻辑、本地存储器、DSP）。

实践思考

  • Pins-In/Pins-Out 分区可能是全书 ROI 最高的单一设计决策。芯片回片后数小时跑起已验证的软件——这个速度取决于一个在平台设计阶段做出的架构选择
  • 时钟灵活性是自建平台复用的第一杀手——如果你的板子只打算用一次，没问题；如果管理层指望多次使用，请投入不成比例的精力在可配置时钟资源上
  • 地弹和电源噪声是最难诊断的原型问题——在原理图阶段花额外一周做电源完整性规划，比在 bring-up 阶段花一个月追踪"偶发性不可靠"值得
  • 散热不可事后补救——散热片+风扇+温度监控应该在管脚分配和电压域规划同时进行，不是"等发现烫了再加"
  • System ACE + CF 卡的配置方案对团队协作特别有价值：硬件工程师做好 bitstream → 拷贝到 CF 卡 → 软件工程师插卡即用——不需要他们了解 JTAG 或下载线缆
