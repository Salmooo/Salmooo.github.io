AI系统：第8章 传统编译器

《AI系统：原理与架构》学习笔记 — 从编译器基础概念出发，深入GCC与LLVM两大编译器巨头的架构设计，掌握IR、Pass、三地址码、SSA等编译核心概念

5 minute read

本章概览

如果说 AI 芯片是"发动机"，那编译器就是"翻译官"——它把程序员写的高级语言转化为芯片能直接执行的机器码。在 AI 系统栈中，AI 编译器正变得越来越重要，而要理解 AI 编译器，必须先理解传统编译器。

本章从零开始讲编译器：什么是编译器？它和解释器有什么区别？JIT 和 AOT 分别适合什么场景？Pass 和 IR 扮演什么角色？——打好这些基础后，我们深入两大开源编译器 GCC 和 LLVM/Clang，理解它们的架构设计、编译流程、优化策略，以及 LLVM IR 的精妙之处。

编译技术是计算机科学皇冠上的一颗明珠——它是连接"人写的代码"与"机器运行的程序"的桥梁。

8.1 编译器基础概念

8.1.1 编译器 vs 解释器

编译器（Compiler）和解释器（Interpreter）都能将高级语言转换为机器语言，本质区别在于时机：

  • 编译器：在程序运行之前把整个源代码一次性翻译成机器码（如 C、C++ → 可执行文件）
  • 解释器：在程序运行之时逐行翻译并执行（如 Python、JavaScript）

编译器 = 整体翻译（先翻译完再执行）→ 快但不够灵活；解释器 = 同声传译（边翻译边执行）→ 灵活但慢。

当前主流开源编译器（LLVM、GCC）通常采用三段式架构：

  • 前端（Frontend）：词法分析 → 语法分析 → 语义分析 → 生成中间代码（IR）
  • 中间优化器（Optimizer）：对 IR 做各种优化（去冗余、子表达式消除等）
  • 后端（Backend）：将优化后的 IR 针对具体硬件生成目标机器码

8.1.2 JIT 与 AOT 编译方式

程序有两种运行方式：

  • AOT（Ahead of Time，提前编译）：代码执行前全部翻译为机器码。代表：C/C++ 应用。
  • JIT（Just in Time，即时编译）：边翻译边运行。代表：JavaScript、Python、PyTorch JIT。

维度 JIT AOT
优点 可据当前硬件实时优化；可据运行情况动态调优；支持动态链接 无运行时编译开销；启动即达最高性能；执行快
缺点 占用运行时资源、可能卡顿；初始编译耗时 安装时间增加；占用更多内存；牺牲部分一致性

在 AI 框架中，TensorFlow（静态图）偏向 AOT，PyTorch 2.X（Inductor）和计图（Jittor）偏向 JIT。两者并不互斥——MLIR + TensorFlow 同时支持 AOT 和 JIT。

8.1.3 Pass 与中间表示 IR

Pass 是编译器对程序的一次完整扫描或处理——可以理解为编译器流水线上的一道"工序"。LLVM 中的 Pass 分三类：

  • Analysis Pass：只分析、不修改 IR（如收集程序信息）
  • Transform Pass：修改和优化 IR（如函数内联 Inline）
  • Utility Pass：辅助工具类 Pass

Pass 由 PassManager 统一调度，多个 Pass 组成优化流水线（Pipeline）。

IR（中间表示） 是编译器最核心的数据结构。编译器的前端、优化器、后端之间唯一交换的数据就是 IR——这实现了模块解耦。

IR 按抽象层级分为三层：

  • HIR（High IR）：靠近源码，适合源码级分析（如 AST）
  • MIR（Middle IR）：与源码和目标平台都无关，适合通用优化（如三地址码 TAC）
  • LIR（Low IR）：靠近机器指令，适合硬件相关优化

多层 IR 的核心优势：保留更多源码信息、优化更灵活、Pass 执行更高效。

8.2 传统编译器介绍

8.2.1 什么是编译器？

“编译器只是一段程序，用来将编程语言 A 翻译成编程语言 B。“简而言之，它将人可读的高级语言（C、Python）转化为机器可执行的低级指令（二进制）。

编译器出现的根本原因：直接用二进制编写程序太痛苦了。1957 年第一个编译器 IBM Fortran 诞生，此后编译器经历了六个发展阶段：

阶段 年代 关键事件
第一阶段 1950s 第一个编译器，将算术公式翻为机器码
第二阶段 1960s Fortran、COBOL、LISP 等多种语言和编译器涌现
第三阶段 1970s 结构化编程（Pascal、C），注重可读性与可维护性
第四阶段 1980s 并行/分布式语言（Ada、Prolog），关注并行能力
第五阶段 1990s 跨平台/动态特性（Java、C#、Python），关注安全与效率
第六阶段 2000s~ AI 框架爆发（Torch→TF→PyTorch），AKG、MLIR 等 AI 编译器涌现

编译器基本构成：前端 → 优化器 → 后端

8.2.2 GCC vs LLVM/Clang

两大编译器巨头的对比：

维度 GCC Clang/LLVM
开源协议 GNU GPL Apache 2.0
代码架构 一体化（紧耦合） 模块化（库化）
支持平台 UNIX、Windows、Mac UNIX、Mac
前端语言 C/C++/Fortran/Java/Go 等 主要 C/C++/Obj-C
构建工具 Make CMake
调试器 GDB LLDB
错误提示 较晦涩 清晰，含修复建议

GCC 兼容性最强（跨平台事实标准），但代码耦合度高、难以独立复用。苹果因为 GCC 对 Objective-C 支持不力且代码难拆分，2005 年启动 Clang 项目，基于 LLVM 构建了全新的编译器体系。

8.3 GCC 编译流程详解

GCC 的编译过程分为四个阶段：预处理 → 编译 → 汇编 → 链接

① 预处理（.c → .i）

  • 头文件展开、宏替换（#define HELLOWORD → "hello world\n"）
  • 条件编译判断（#if、#ifdef）
  • 删除注释和多余空白
  • 保留 #pragma 指令
1

gcc -E hello.c -o hello.i

② 编译（.i → .s）

  • 词法分析、语法分析、语义分析
  • 将预处理后的代码转成汇编代码（.s 文件）
  • 这是 GCC 前端的主要工作
1

gcc -S hello.i -o hello.s

③ 汇编（.s → .o）

  • 汇编器（as）将汇编代码转成机器指令
  • 生成可重定位目标文件（.o）
  • 这是 GCC 后端的核心工作
1

gcc -c hello.s -o hello.o

④ 链接（.o → 可执行文件）

  • 链接器将目标文件 + 库文件 → 可执行文件
  • 静态链接：库函数直接复制进可执行文件（体积大、独立运行）
  • 动态链接：只记录文件名，运行时加载动态库（体积小、依赖环境）
1

gcc -o hello.o -o hello

使用 gcc -v hello.c -o hello 可查看完整编译过程。

8.4 LLVM 架构设计

8.4.1 LLVM 发展历程

LLVM 起源自 2000 年伊利诺伊大学的研究项目，最初全称是"低级虚拟机（Low Level Virtual Machine）"。2005 年苹果雇佣了 LLVM 之父 Chris Lattner 为 MacOS/iOS 构建开发工具，LLVM 从此成为苹果生态的核心组件。

如今 LLVM 已不再只是"虚拟机”——它已发展为包含 LLVM IR、LLVM 调试工具、C++ 标准库等在内的庞大编译器工具集合品牌。

8.4.2 LLVM 架构特点：模块化 + 统一 IR

LLVM 最大的创新在于：组件独立、库化架构、统一 IR。

  • 前端工程师只需实现前端 → 产生 IR，无需关心后端
  • 后端工程师只需对接目标硬件特性与 IR，无需关心前端
  • 核心纽带就是统一的 LLVM IR

传统 GCC 中每种语言和每种硬件之间是多对多映射（m × n 种编译器），LLVM 通过统一 IR 简化为 m + n 种编译器——这是质变。

LLVM 的 Clang 前端将 C/C++/Obj-C 代码经过：

  1. 词法分析 → 生成 token 流
  2. 语法分析 → 构建 AST（抽象语法树）
  3. 语义分析 → 类型检查，生成 LLVM IR

优化层通过多个 Pass 提升 IR 性能：常量折叠、循环优化、死代码消除等。

后端负责将优化后的 IR 转为目标机器码：指令选择 → 寄存器分配 → 指令调度 → 代码生成。

LLVM IR 流程：C 代码 → 预处理 → .ll/.bc 文件 → llc 工具 → 汇编代码 → 链接 → 可执行文件。

8.5 LLVM IR 详解

8.5.1 什么是 LLVM IR？

LLVM IR 是 LLVM 的核心——它是一种类似汇编的底层语言，但强类型、精简指令集（RISC）风格、平台无关。

IR 不只有一种形态：前端出 AST 形的 IR，中端优化后出 DAG 图给后端——IR 是"抽象表达”，没有固定形态。

LLVM IR 的优势：

  • 更独立：可在编译器之外的工具中重用（如静态分析器）
  • 更正式：有明确定义的 C++ API
  • 更接近硬件：提供类似 RISC-V 的模拟指令集 + 强类型，底层又足够灵活

以下 C 代码：

void test(int a, int b) {
    int c = a + b;

编译为 LLVM IR（clang -S -emit-llvm test.c）：

define dso_local void @test(i32 noundef %0, i32 noundef %1) {
  %3 = alloca i32, align 4     ; 局部变量
  %4 = alloca i32, align 4
  %5 = alloca i32, align 4
  store i32 %0, ptr %3, align 4 ; a 存入
  store i32 %1, ptr %4, align 4 ; b 存入
  %6 = load i32, ptr %3, align 4
  %7 = load i32, ptr %4, align 4
  %8 = add nsw i32 %6, %7      ; c = a + b
  store i32 %8, ptr %5, align 4
  ret void
}

基本语法：@ 表示全局，% 表示局部变量，alloca 分配栈内存，store/load 读写，add 运算，i32 是 32 位整数，align 是字节对齐。

8.5.3 三地址码（TAC）

LLVM IR 采用三地址码格式——每条指令最多三个操作数：两个源 + 一个目标。

1
2

%result = add i32 %a, %b
→ 四元组表示: (add, %a, %b, %result)

三地址码的优点：简单、清晰、便于优化（常量折叠、死代码消除、寄存器分配都依赖这种格式）。

8.5.4 LLVM IR 设计原则

  • SSA（静态单赋值）：每个变量只被赋值一次，极大简化数据流分析和优化
1
2
3
4
5
6
7

; 错误（非 SSA）：%0 被赋值两次
%0 = mul i32 1, 2
%0 = add i32 %0, 3

; 正确（SSA）：
%0 = mul i32 1, 2
%1 = add i32 %0, 3
  • 强类型系统：每个值都有明确类型（i32、float、struct 等）
  • 模块化设计：Module → Function → BasicBlock → Instruction 四级结构
  • 目标无关：IR 不依赖具体 CPU 架构，后端负责翻译

8.6 LLVM 前端——Clang 深度解析

Clang 前端三步：

① 词法分析（Lexical Analysis）

将源代码分解为 token 流（关键字、标识符、运算符、常量等），并精确记录每个 token 在源文件的位置。

1
2

clang -cc1 -dump-tokens hello.c
# 输出：int 'int' Loc=<hello.c:5:1>  identifier 'main' Loc=<hello.c:5:5> ...

② 语法分析（Syntax Analysis）

根据语法规则将 token 流组织成 AST（抽象语法树）。只验证结构，不做语义判断。

1
2

clang -fsyntax-only -Xclang -ast-dump hello.c
# 输出：FunctionDecl、ParmVarDecl、CompoundStmt、ReturnStmt 等 AST 节点

③ 语义分析（Semantic Analysis）

借助符号表检查类型匹配、变量声明范围、函数调用正确性等——确保代码"意思是对的"。

1
2

int a[4];
int a[5];  // ❌ error: redefinition of 'a' with a different type

语义分析输出的是类型无误的 AST，作为编译器后端的输入。

8.7 LLVM 优化层——Pass 机制

LLVM 优化层输入 AST，输出 DAG 图。每一种优化方式称为一个 Pass。

  • Analysis Pass：只读分析（如计算支配树 DominatorTree）
  • Transform Pass：修改 IR（如死代码消除 ADCE、常量合并 MDGC）
1
2

clang -emit-llvm -c hello.c -o hello.bc
opt -passes='instcount,adce,mdgc' -o hello-tmp.bc hello.bc -stats

Pass 依赖关系：

  • 显式依赖：Pass 管理器自动安排依赖的 Pass 先运行
  • 隐式依赖：需手动按正确顺序将 Pass 加入队列

8.8 LLVM 后端——代码生成

后端将优化后的 IR 转换为目标平台的机器码，涉及四层指令表示：

层级 表示 描述
1 LLVM IR 高级抽象，平台无关
2 SelectionDAG 节点 基本块级的数据流依赖图
3 MachineInstr 机器相关的三地址指令
4 MCInst 最终二进制编码指令

白色 Pass 非必要，灰色 Pass 必要。越往下越接近真实硬件指令。

关键步骤

① 指令选择（Instruction Selection）

将 LLVM IR 转为目标特定的 SelectionDAG 节点 → 用树模式匹配算法选出最优指令序列。

  • 灰色粗实线 = 强顺序约束（相邻指令必须紧挨）
  • 虚线 = 非数据流链（强制顺序）

② 寄存器分配

虚拟寄存器 → 物理寄存器映射。物理寄存器有限，溢出时需"spill"到内存。关键算法：线性扫描分配、图着色分配。

③ 指令调度

  • Pre-RA 调度：寄存器分配前，最大化并行度
  • Post-RA 调度：寄存器分配后，处理资源冲突、插入填充指令

④ 代码输出

指令融合 → 启发式优化 → 循环优化 → 生成汇编/机器码。

小结与思考

这一章从编译器最基础的概念讲起，一路深入到 GCC 编译四阶段、LLVM 模块化架构和 LLVM IR 的精妙设计。最后回顾几个关键问题：

  • 编译器和解释器的本质区别是什么？ 编译器在运行前一次性翻译整个程序（快但不够灵活），解释器边运行边翻译（灵活但慢）。AI 框架中 AOT 和 JIT 各有适用场景——移动端推理偏 AOT，训练/调试偏 JIT。

  • 为什么 LLVM 能取代 GCC 成为苹果首选？ 模块化 + 统一 IR。GCC 紧耦合、难拆分，LLVM 前后端独立、复用性强——新增语言只需写前端，新增硬件只需写后端。

  • LLVM IR 最核心的设计理念是什么？ ① SSA（静态单赋值）简化数据流分析；② 三地址码提供简洁统一的指令表示；③ 平台无关——所有优化在 IR 层完成，后端只做翻译。

  • Pass 机制为什么重要？ Pass 是编译优化的最小单元，通过 PassManager 灵活组合分析 Pass 和转换 Pass，形成完整优化流水线——这种机制也被 AI 编译器（如 MLIR）广泛使用。

理解传统编译器是理解 AI 编译器的基础。下一章将直接进入 AI 编译器的世界——看看 AI 框架如何将"神经网络"编译成"芯片上的计算图"。
