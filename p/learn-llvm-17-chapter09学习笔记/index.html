2. JIT 编译概述

2.1 AOT 与 JIT

类型 说明
AOT (Ahead-of-Time) 提前编译整个应用，编译完成后才能运行
JIT (Just-in-Time) 在应用运行时编译，边编译边执行

2.2 JIT 典型用例

场景 说明
虚拟机 字节码 → JIT 编译为机器码（Java、C#、USCD Pascal 1977）
表达式求值 电子表格、金融模拟、lldb 调试时计算表达式
数据库查询 执行计划 → JIT 转为机器码加速查询

2.3 LLVM JIT 核心思想

静态编译器将 IR 编译为机器码并写入磁盘；若将目标文件保存在内存中，再在内存中执行链接，解析重定位，即可得到 JIT 编译器。

1

IR → 内存中的目标文件 → 内存中链接 → 可执行代码

2.4 ORC 引擎

ORC (On-Request Compilation) 是 LLVM 最新的 JIT API，基于分层设计：

层次 说明
编译层 将 IR 编译为对象文件
链接层 在内存中链接对象文件
惰性编译层 按需编译（可选）
转换层 可堆叠，用于优化或事件通知
  • MCJIT：旧 API，仍可用，但建议新项目使用 ORC。
  • LLJIT：预配置的 ORC 封装，迁移自 MCJIT，开箱即用。

  • lookup("calc_expr_func") 触发编译与链接。
  • toPtr<int (*)()>() 将符号地址转为函数指针并调用。

  virtual void visit(DefDecl &Node) override {
    llvm::StringRef FnName = Node.getFnName();
    llvm::SmallVector<llvm::StringRef, 8> FunctionVars = Node.getVars();
    (JITtedFunctionsMap)[FnName] = FunctionVars.size();

    Function *DefFunc = genUserDefinedFunction(FnName);
    // ...
    BasicBlock *BB = BasicBlock::Create(M->getContext(), "entry", DefFunc);
    Builder.SetInsertPoint(BB);

    unsigned FIdx = 0;
    for (auto &FArg : DefFunc->args()) {
      nameMap[FunctionVars[FIdx]] = &FArg;
      FArg.setName(FunctionVars[FIdx++]);
    }

    Node.getExpr()->accept(*this);  // 生成表达式，结果在 V 中
  };

  virtual void visit(FuncCallFromDef &Node) override {
    llvm::StringRef CalcExprFunName = "calc_expr_func";
    FunctionType *CalcExprFunTy = FunctionType::get(Int32Ty, {}, false);
    Function *CalcExprFun = Function::Create(
        CalcExprFunTy, GlobalValue::ExternalLinkage, CalcExprFunName, M);

    BasicBlock *BB = BasicBlock::Create(M->getContext(), "entry", CalcExprFun);
    Builder.SetInsertPoint(BB);

    Function *CalleeFn = genUserDefinedFunction(CalleeFnName);
    auto CalleeFnVars = Node.getArgs();
    llvm::SmallVector<Value *> IntParams;
    for (unsigned i = 0, end = CalleeFnVars.size(); i != end; ++i) {
      int ArgsToIntType;
      CalleeFnVars[i].getAsInteger(10, ArgsToIntType);
      IntParams.push_back(ConstantInt::get(Int32Ty, ArgsToIntType, true));
    }

    Value *Res = Builder.CreateCall(CalleeFn, IntParams, "calc_expr_res");
    Builder.CreateRet(Res);
  };

calc_expr_func 为 i32 ()，内部调用用户函数并返回结果。

  virtual void visit(FuncCallFromDef &Node) override {
    llvm::StringRef FuncCallName = Node.getFnName();
    unsigned NumArgs = Node.getArgs().size();
    unsigned NumOriginallyDefinedArgs = 0;
    auto LookUpFunctionCall = JITtedFunctionsMap.find(FuncCallName);
    if (LookUpFunctionCall != JITtedFunctionsMap.end())
      NumOriginallyDefinedArgs = LookUpFunctionCall->second;
    else {
      llvm::errs() << "Specified function definition does not exist!\n";
      HasError = true;
    }
    if (NumArgs != NumOriginallyDefinedArgs) {
      llvm::errs() << "Number of parameters specified to the function does not "
                   << "match it's definition!\n";
      HasError = true;
    }
  };

4.7 限制与设计说明

  • 不支持负数、小数。
  • 不允许重定义同名函数：ORC 符号表不允许重复定义，会报 Duplicate definition of symbol '_f'。
  • 其他符号来源：StaticLibraryDefinitionGenerator、DynamicLibrarySearchGenerator、addObjectFile() 等。

  static llvm::Expected<std::unique_ptr<JIT>>
  create() {
    auto SSP = std::make_shared<
        llvm::orc::SymbolStringPool>();
    auto EPC =
        llvm::orc::SelfExecutorProcessControl::Create(
            SSP);
    if (!EPC)
      return EPC.takeError();
    llvm::orc::JITTargetMachineBuilder JTMB(
        (*EPC)->getTargetTriple());

    auto DL = JTMB.getDefaultDataLayoutForTarget();
    if (!DL)
      return DL.takeError();

    auto ES =
        std::make_unique<llvm::orc::ExecutionSession>(
            std::move(*EPC));

    return std::make_unique<JIT>(
        std::move(*EPC), std::move(ES), std::move(*DL),
        std::move(JTMB));
  }

  static llvm::Expected<llvm::orc::ThreadSafeModule>
  optimizeModule(
      llvm::orc::ThreadSafeModule TSM,
      const llvm::orc::MaterializationResponsibility
          &R) {
    TSM.withModuleDo([](llvm::Module &M) {
      llvm::PassBuilder PB;
      llvm::LoopAnalysisManager LAM;
      llvm::FunctionAnalysisManager FAM;
      llvm::CGSCCAnalysisManager CGAM;
      llvm::ModuleAnalysisManager MAM;
      FAM.registerPass(
          [&] { return PB.buildDefaultAAPipeline(); });
      PB.registerModuleAnalyses(MAM);
      PB.registerCGSCCAnalyses(CGAM);
      PB.registerFunctionAnalyses(FAM);
      PB.registerLoopAnalyses(LAM);
      PB.crossRegisterProxies(LAM, FAM, CGAM, MAM);
      llvm::ModulePassManager MPM =
          PB.buildPerModuleDefaultPipeline(
              llvm::OptimizationLevel::O2);
      MPM.run(M, MAM);
    });

    return TSM;
  }

  if (auto Err = (*JIT)->addIRModule(
          orc::ThreadSafeModule(std::move(M),
                                std::move(Ctx))))
    return Err;

  auto MainSym = (*JIT)->lookup("main");
  if (!MainSym)
    return MainSym.takeError();

  llvm::orc::ExecutorAddr MainExecutorAddr = MainSym->getAddress();
  auto *Main = MainExecutorAddr.toPtr<int(int, char**)>();

  (void)Main(argc, argv);
  return Error::success();
}

流程：创建 JIT → 添加 IR 模块 → 查找 main → 转为函数指针并调用。

6. 设计决策与扩展

6.1 惰性编译 vs 全量编译

策略 优点 缺点
惰性编译 启动快、按需编译 过程间优化受限（如内联）
全量编译 + 优化 代码质量高 启动较慢

本章自定义 JIT 选择全量编译 + O2 优化。
