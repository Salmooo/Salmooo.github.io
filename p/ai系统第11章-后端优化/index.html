AI系统：第11章 后端优化

《AI系统：原理与架构》学习笔记 — 深入AI编译器的后端优化：从计算与调度分离，到循环优化、指令优化、存储优化，再到Auto-Tuning自动调优

5 minute read

本章概览

前面几章我们讨论了 AI 编译器前端优化（计算图级别的图算融合、死代码消除等）。但问题是：优化后的计算图到了底层，算子的每一个循环、每一次内存访问，谁来优化？ 答案就是本章的主角——后端优化。

如果说前端优化是"宏观调度"（整个计算图的拓扑结构优化），那后端优化就是"微观精雕"——深入到单个算子的循环嵌套、内存布局、指令选择，把每一滴硬件性能都榨出来。

本章将从 计算与调度的分离哲学 出发，依次讲解循环优化五大法宝、指令和存储优化、DSL 开发（Triton），最后聊到自动调优——让机器帮我们找到最优的算子实现。

11.1 后端优化引言

11.1.1 前端优化 vs 后端优化

AI 编译器有两层中间表示（IR），对应两类优化：

优化类型 关注范围 核心工作
前端优化 整个计算图拓扑结构 算子融合、消除、化简，降低计算和存储开销（第10章内容）
后端优化 单个算子的内部实现 循环内存编排、指令选择、存储分配，让算子跑得最快

一个形象的类比：前端优化是城市规划师，决定哪条路该拓宽、哪个路口该合并；后端优化是 F1 赛车技师，对每一台引擎的进气和点火时机做微调。

11.1.2 后端优化的三步流程

后端优化有标准的"三部曲"：

  1. 生成低级 IR：将 Graph IR 转换为 Tensor IR。不同 AI 编译器内部低级 IR 形式不同，但同一算子的算法原理一致。
  2. 后端优化：针对不同硬件架构（如 Conv1x1、Conv3x3、Conv7x7 都需要不同的循环优化方法），找到算子的最优实现。为什么不能直接丢给 GCC/LLVM？ 两个原因：①数据形式不同——深度学习以张量为主，传统编译器擅长标量；②缺乏 DSL 支持——传统编译器不懂神经网络的领域特定语言。
  3. 代码生成：将优化后的低级 IR 转化为机器指令（通常是借助 LLVM/NVCC 等成熟编译工具）。

11.1.3 算子优化的两大挑战

算子按计算形式分为两类：

  • 访存密集型（Memory-Bound）：频繁读取写入内存，性能受限于内存带宽。如 RNN 训练——计算密度低，瓶颈在 Host 端 Op Launch 上。
  • 计算密集型（Compute-Bound）：大量计算、相对少的内存访问。如矩阵乘法、卷积——性能和算力强相关。

算子优化的三大挑战：

  • 优化手段多样：不同算子、不同参数适用不同优化，组合优化甚至是 NP 问题
  • 通用性与移植性：不同硬件架构差异巨大
  • 不同优化间相互影响：各种优化会相互制约

两大解决思路：

  • 算子库：硬件厂商预置高度优化的计算核心（cuBLAS、cuDNN、oneDNN 等）。问题是迭代慢、跨平台一致性差、算子组合爆炸。
  • 自动生成：Auto Tuning（启发式搜索参数组合）和 Polyhedral（多面体编译优化）方法。

11.2 计算与调度

11.2.1 高斯的启示：计算与调度分离

还记得高斯求和的故事吗？计算 1+2+…+100，逐个加需要 99 次运算，但高斯发现两两配对（1+100=101, 2+99=101…），只需一次乘法 50×101=5050。

对于计算机运算，同样存在这样的"捷径"。一个计算可以有多种不同的实现方式——用最朴素的循环，或用向量化、循环分块、并行化。这就是计算与调度分离的核心思想：

  • 计算（Computation）：描述实现算法的逻辑，不关心具体怎么执行
  • 调度（Schedule）：对计算进行优化控制——执行顺序、内存布局、并行策略

这种思想的最佳践行者就是 Halide——一个用 C++ 作为宿主语言的图像处理 DSL。它让你先定义"要算什么"，再独立地指定"怎么算"，两者解耦。

以矩阵乘法为例，最朴素的实现：

1
2
3
4
5
6

for (int i = 0; i < size; ++i)
    for (int j = 0; j < size; ++j) {
        result[i][j] = 0;
        for (int k = 0; k < size; ++k)
            result[i][j] += A[i][k] * B[k][j];
    }

同样一个矩阵乘法，可以用循环分块、向量化、甚至把它们组合起来——算法功能完全一样，但速度天差地别。所有可能的调度方式构成了"调度空间"，AI 编译器的目标就是从中找到最优解。

11.2.2 调度树：把调度抽象成一棵树

对于一个算子的计算形式（多层嵌套循环 + 多维张量输入），可以通过三个要素来刻画：

  • 循环节点：表示沿某个维度如何遍历计算（顺序、并行、向量化、展开）
  • 存储节点：表示中间结果的存储位置
  • 计算节点：叶子节点，表示正在执行的计算

这就是调度树。给定一棵调度树，通过深度优先遍历就能生成对应的程序代码。

理解调度树的好处在于：只要树是合法的，你就可以随意变换它，生成不同性能的程序——这就是"计算与调度分离"的工程实现。

11.2.3 Halide 调度变换实战

以 Halide 为例，一个简单的梯度计算：

1

4

Var x("x"), y("y");
Func gradient("gradient");
gradient(x, y) = x + y;
Buffer<int> output = gradient.realize(4, 4);

等价于：

1
2
3

for (int y = 0; y < 4; y++)
    for (int x = 0; x < 4; x++)
        printf("Evaluating at x = %d, y = %d: %d\n", x, y, x + y);

现在用 Halide 调度 API 进行优化——融合（Fuse）+ 分块（Tile）+ 并行（Parallel）：

1
2
3
4
5
6
7

9

Var fused;
gradient.fuse(x, y, fused);
// → 将两层循环合并为一个 16 次迭代的循环

Var x_outer, y_outer, x_inner, y_inner, tile_index;
gradient.tile(x, y, x_outer, y_outer, x_inner, y_inner, 4, 4);
gradient.fuse(x_outer, y_outer, tile_index);
gradient.parallel(tile_index);
// → 外层 tile 融合后并行执行，内层保持原始循环

在调度树中，这些操作对应着：

  • Fuse：将两个相邻循环节点合并为一个
  • Parallel：改变循环类型为并行化
  • Vectorize、Unrolling 同理，只是更改循环节点上的属性

Halide 提供的调度优化方式包括：Reorder（交换）、Split（拆分）、Fuse（融合）、Tile（平铺）、Vectorize（向量化）、Unroll（展开）、Parallel（并行）。以 Halide 思想为指导的 AI 编译器 TVM 继承了所有这些优化方式。

11.3 算子手工优化

11.3.1 用 Roofline 模型找到瓶颈

在动手优化前，先搞清楚：当前程序的瓶颈是算力不够，还是内存带宽不够？

Roofline 模型定义了四个关键指标：

指标 含义 公式/说明
计算量 一次完整计算的浮点运算数 如卷积层：\(M^2 \times K^2 \times C_{in} \times C_{out}\)
访存量 一次完整计算的内存交换总量 参数 + 输出特征图（Byte）
计算强度 每字节内存交换对应的浮点计算次数 计算量 ÷ 访存量，越大内存效率越高
理论性能 硬件算力上限 Roofline 模型给出的天花板

从图中可以清晰看到：MobileNet 处于带宽受限区域（在 1080T 上理论性能只有 3.3 TFLOPS），而 VGG 处于算力受限区域（能充分利用 1080T 的算力）。知道自己卡在哪里，才知道优化方向。

11.3.2 三大优化策略

① 循环优化：AI 算子是高度规则的多层嵌套循环，通过调整循环结构可以显著减少开销。常见技术包括：

  • 循环分块（Loop Blocking）——提高 Cache 命中率
  • 循环展开（Loop Unrolling）——减少分支跳转
  • 循环重排（Loop Reordering）——改善数据访问模式
  • 循环融合（Loop Fusion）——减少内存访问次数
  • 循环拆分（Loop Splitting）——创造更多优化机会

（这些将在 11.4 节详细展开）

② 指令优化：利用处理器的向量指令（Intel AVX-512、ARM SVE）和张量指令（NVIDIA Tensor Core、Intel VNNI），让一条指令同时操作多个数据。

③ 存储优化：精细调整数据访问模式和内存层次结构——预取（Prefetching）、双缓冲（Double Buffering）、内存池等，隐藏内存访问时延。

11.3.3 Triton：面向"块"编程的 DSL

手写 CUDA Kernel 是出了名的"劝退"——你需要同时考虑 DRAM、SRAM、计算单元的调度。Triton（OpenAI 研发的开源 DSL）改变了这个局面：

对比维度 CUDA Triton
编程模型 面向 Thread（线程级细粒度） 面向 Thread Block（块级编程）
开发者关注点 内存管理、线程管理、指令选择 Kernel Launch 参数、分块大小、块间交互
性能 vs 效率 极致性能，极低开发效率 90%+ 极致性能，数十倍开发效率

在矩阵乘法中，CUDA 在细粒度线程上编程，而 Triton 是逐块计算的——这种块状迭代空间给了程序员更多灵活性，同时让编译器有更大的优化空间。

Triton 已被集成进 JAX、PyTorch Inductor 等主流框架，实现了从"实验室"到"生产环境"的跃迁。

11.3.4 Triton 实现原理

Triton 基于 MLIR 构建，架构如下：

Frontend：@triton.jit 注解 → 解析 Python AST → 生成 Triton IR (Triton Dialect)

Optimizer 的工作流：

分为三阶段：①Triton IR 优化 → ②Triton IR 到 TritonGPU IR 的转换 → ③TritonGPU IR 优化。贯穿整个流程的核心数据结构是 TritonGPU IR。

Triton 的两大优化思想：

  • Layout 抽象：描述计算资源和输入/输出元素的坐标映射关系——块编码、共享内存编码、切片编码，以 Attribute 附着在 Tensor 上。不兼容时插入中转 Layout。
  • 优化 Pass：包括 Coalescing（向量化访存）、Pipeline/Prefetch（缓解计算访存差异）、Swizzling（避免 Shared Memory Bank-Conflict）。用户只需关注业务逻辑。

11.4 算子循环优化

11.4.1 两个核心矛盾：数据局部性 vs 计算并行性

数据局部性源于计算机的存储层次——从寄存器（最快最小）到 Cache（L1/L2/L3）到主存（慢但大）。Cache 里应该放 CPU 最可能再次访问的数据，这依赖两个局部性原理：

  • 时间局部性：刚处理过的数据很可能被再次处理
  • 空间局部性：刚访问过的数据附近的数据很可能马上被访问

以一个简单的两层嵌套循环为例：

1
2
3

for i in range(n):
    for j in range(m):
        A[i] += B[j]

A[i] 在内层循环完整结束后才被重用（时间局部性），每个 A 元素被重用 m 次；B[j] 每次内层循环都改变（空间局部性），每个 B 元素被重用 n 次。两者呈现不同的局部性模式——这直接决定了优化策略。

计算并行性则来自多核 CPU（线程级并行）和向量寄存器（数据级并行 SIMD）——一次操作多个数据元素。一个 512bit 的向量寄存器可以容纳 64 个 64bit 元素，或 128 个 32bit 元素，或 512 个 8bit 元素。

11.4.2 循环优化五大法宝

① 循环分块（Loop Blocking/Tiling）

这是利用 Cache 局部性最核心的优化。大数据集无法一次放入 Cache，循环按顺序访问时后面数据会把前面的顶出去（Cache Miss），导致大量时间花在加载数据上。

做法：将大数据集切分成小块，一次加载进 Cache，执行完尽可能多的计算才换出。以之前的代码为例，对 j 层做分块（tile size = T）并提到最外层：

1
2
3
4

for j_o in range(0, m, T):
    for i in range(n):
        for j_i in range(j_o, min(j_o + T, m)):
            A[i] += B[j_i]

当 T 个 B 元素能放进 Cache 时，i 循环切换时 B 仍然在 Cache 中——Cache Miss 从 \(n \times m/b\) 降到了 \(m/b\)。更进一步，对 i 也做分块（tile size = W），A 的 Cache Miss 也从 \(n/b\) 变成了 \(n/b\)（前提是 W 个 A 也能放进 Cache）。

分块大小（T、W）的选择很微妙：太大 → Cache 装不下，未充分利用就被替换；太小 → 额外开销掩盖收益。目前主流方法有基于分析的静态方法和基于经验搜索的 Auto Tuning。

② 循环展开（Loop Unrolling）

将循环体复制 n 次（展开因子），减少循环计数、条件判断和分支跳转的开销：

 1
 2
 3

# 原始循环
for i in range(m):
    a[i] += b[i]

# 展开因子=4
for i in range(0, m-3, 4):
    a[i] += b[i]
    a[i+1] += b[i+1]
    a[i+2] += b[i+2]
    a[i+3] += b[i+3]

好处：减少循环开销、提高指令并发度、减少流水线停顿。风险：循环体超过指令缓存容量会导致 Cache Miss，展开过多也会增加寄存器压力。

③ 循环重排（Loop Reordering）

矩阵乘法中，如果 B 是逐列访问而数据是行优先存储的，访问模式就很"反人类"。交换内层循环顺序可以让所有元素按顺序读写：

一次计算输出的一行（中间结果），全部累加后得到结果矩阵的一行——利用的是空间局部性。

④ 循环融合（Loop Fusion）

将相邻循环合并，减少内存访问次数：

 1
 2
 3
 4
 5
 6
 7

 9
10

# 融合前：两个独立循环，a 写入后被清出 Cache，第二个循环又要重新读取
for i in range(len(a)):
    a[i] = b[i] + x
for i in range(len(b)):
    d[i] = a[i] + y

# 融合后：a 刚写入就读取，还在 Cache 里
for i in range(len(a)):
    a[i] = b[i] + x
    d[i] = a[i] + y

⚠️ 注意：循环融合需要检查数据依赖！如果前后循环存在写后读（RAW）依赖，融合可能产生错误结果。

⑤ 循环拆分（Loop Splitting）

将有条件分支的循环拆分为纯计算部分（可上加速器）和带控制流部分（退回 CPU）：

# 拆分前
for i in range(n):
    A[i] = a[i] + b[i]
    c[i] = 2 * a[i]
    if temp[i] > data:
        d[i] = a[i]

# 拆分后：纯计算部分 → GPU/NPU；条件部分 → CPU

    A[i] = a[i] + b[i]
    c[i] = 2 * a[i]
for i in range(n):
    if temp[i] > data:
        d[i] = a[i]

循环拆分还能和融合结合——比如把两个有依赖的循环中的一个拆开，将无依赖的部分融合进去，提升局部性。

11.5 指令和存储优化

11.5.1 指令优化：向量化与张量化

向量化（Vectorization）：SIMD 的工程实现——将多个连续数据批量加载进向量寄存器，对整个寄存器一次性操作：

普通循环 C[i] = A[i] + B[i] 需要 n 次迭代。向量化后 C[i:i+3] = A[i:i+3] + B[i:i+3]，迭代次数直接除以 4。对应硬件指令：

指令集 加法指令 乘法指令
Intel SSE _mm_add_ps _mm_mul_ps
Intel AVX/AVX2 _mm256_add_ps _mm256_mul_ps
ARM NEON vaddq_f32 vmulq_f32

张量化（Tensorization）：向量化的"升级版"。神经网络中的张量是 4 维的 [N, C, H, W]，传统多层循环逐个计算效率极低。NVIDIA Tensor Core 就是为此而生——Volta 架构中一个 SM 包含 128 个 Tensor Core，专门加速矩阵乘法和累加操作。Intel 也有对应的 VNNI 技术。

当模型中出现新算子时，依赖硬件厂商算子库就不够用了——这就回到了前面聊的 Triton 和 Auto Tuning 方案。

11.5.2 存储优化：时延隐藏与内存分配

访存时延隐藏

在纯串行架构中，执行流程是"加载数据 → 执行计算 → 写回结果"，每一步的时间都累加。现代硬件通过并行来打破这个限制：

  • CPU：多线程 + 硬件隐式数据预取。一个线程等待内存时，CPU 切换到另一个线程继续算。
  • GPU：Warp Schedule——在等待内存操作完成时动态调度其他 Warp 执行计算，通过上下文切换隐藏内存时延。

  • NPU：解耦访问/执行（DAE）架构——内存访问操作和计算操作分开执行，并行进行。通常配合双缓冲机制缓存不同 LOAD 指令的数据。

存储分配

传统编译器的内存模型（栈、静态区、堆）在 AI 系统中远远不够。专用硬件有各自的内存管理：

硬件 内存层次
GPU 全局内存（显存，大但慢）→ 共享内存（线程块级，快但小）→ 寄存器（线程级，最快）
NPU 片上内存（权重和激活数据）→ 高并发内存访问 → 专用内存控制器

11.6 Auto-Tuning 原理

11.6.1 为什么需要自动调优？

手工编写高性能算子库面临三个"越来越"：

  • 优化手段越来越多：循环优化、存储优化、指令优化……每种又有自己的参数（如分块因子），软硬件的组合让手工调优耗费巨大精力
  • 硬件平台越来越多样：CPU、GPU、FPGA、ASIC，每种都有不同的优化策略；以卷积为例，不同硬件上的实现差异极大
  • 软硬件迭代越来越快：新处理器架构层出不穷，新算子（几十种卷积变体、各种激活函数）不断涌现

11.6.2 自动调优的四步法

自动调优的核心思路：让编译器在一个巨大的优化空间中自动搜索，找到最佳配置。

四步流程：

  1. 性能分析：分析程序运行情况，识别瓶颈和优化机会
  2. 参数搜索：系统性地探索不同编译选项和运行参数
  3. 性能评估：对不同配置进行测试，评估性能影响
  4. 反馈学习：根据评估结果调整搜索策略，进一步优化参数选择

这个过程本质上是在一个极其庞大的搜索空间（优化组合 × 参数配置 × 硬件特性）中寻找最优解。虽然自动调优还处于活跃研究阶段，但它已经是生产级 AI 编译器的标配能力。

小结与思考

这一章从后端优化的概念出发，一路讲到循环优化、指令/存储优化，再到自动调优，信息密度相当高。但我们真正要记住的核心逻辑其实只有一条：

后端优化的本质，是在"计算定义"和"硬件执行"之间架一座最优的桥梁。

  • 计算与调度分离（Halide/TVM）让我们可以独立地定义算法和优化策略——换个调度方案，同样的计算性能可能差几十倍
  • 循环优化五大法宝（分块、展开、重排、融合、拆分）本质上都是在和数据局部性与计算并行性较劲——目的就是让 Cache 命中率更高、流水线更满、SIMD 单元更忙
  • Triton 证明了"面向块编程"比"面向线程编程"更高效、更友好——90% 的极致性能换取数十倍的开发效率，这笔交易是划算的
  • 自动调优是终极方案——人工不可能遍历所有优化组合，但机器可以

前端优化让计算图"瘦身"，后端优化让每个算子"跑更快"。两者合在一起，才构成 AI 编译器的完整能力。
