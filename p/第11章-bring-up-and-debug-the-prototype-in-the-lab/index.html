第11章: Bring Up and Debug — The Prototype in the Lab

Jun 06, 2026
6 minute read

章节概览

前面十一章的最终目的是把设计跑起来并找到 bug。第11章讲述的就是这最后一个环节：如何在实验室中将原型 bring-up 并从"不工作"走到"完全功能验证"。

章节的两大主题如标题所示：Bring-Up（使原型启动并运行）和 Debug（当它不按预期工作时找到原因）。作者反复强调两者必须分开——板子和工具链的问题必须先行排除，剩余的功能故障才是真正的设计 bug。从空板→参考设计→设计子集→全设计，这是一个不容跳步的递进验证过程。

本章也是全书中"实战色彩"最强的章节——从四抽头 FIR 滤波器测试代码到 UMRBus 高速调试总线的架构，再到"检查安全锁单元的 ID 寄存器"这种极端场景——全是实验室角落实战经验的浓缩。

核心知识点

  • Bring-Up 七步递进流程：测试基板 → 测试基板+子板 → 单 FPGA 参考设计 → 多 FPGA 参考设计 → 检查 SoC 设计实现问题 → 分功能子集加载真实设计 → 加载完整设计
  • 三类设计 Bug（Table 29）：逻辑 bug（RTL 中未发现的错误）→ 接口 bug（外部接口未预见的交互问题）→ 软件 bug（软硬件首次集成时暴露）——第三类最有价值
  • 四抽头 FIR 滤波器测试（Figure 135）：全同步单时钟设计，每抽头分区到不同 FPGA —— 验证复位时序+时钟偏斜+跨 FPGA 连接+分区+综合+P&R+加载的完整流程
  • IOB FF 使用不当的微妙时序问题：三态信号的数据路径用 IOB FF 而控制路径没用 → 数据早到但必须等三态控制 → 净时序优势被抵消。较坏情况：三态控制路径和数据路径到达时间偏斜→ PAD 输出产生毛刺
  • JTAG 是调试的基础通道：可捕获 FPGA 引脚静态快照——但带宽极低，上传一个典型软件映像需 ~30 分钟
  • UMRBus 高速调试总线：USB 2.0 通道→FPGA 内总线协议→连接各 CAPIM 模块→集高速配置、存储器直接访问、软件快速加载、数据流注入于一体——30 分钟缩短到不足 1 分钟
  • Compile Point 增量综合：将设计分为子模块（CP）→ 仅改动过的 CP 重新综合 → 其他 CP 重用历史结果。跨 CP 边界优化可控（soft/hard/locked）
  • Design Preservation 增量 P&R：分区状态设为 “implement”（已变更）或 “import”（保留前次）→ 仅变更分区重新 P&R
  • 组合增量流程：综合 CP = “locked, partition” → 自动标记为 P&R 分区 → Synplify 自动写入 “implement”/“import”——端到端径流时间大幅缩减
  • 10 步 Bring-Up 检查清单（Table 30）：从电源轨→输入时钟→FPGA 配置和复位→小 RTL 设计→加插桩→自运行程序→外部 RAM→Hello World→全 SoC→真实软件映像

详细笔记

11.1 Bring-Up 与 Debug：两个必须分离的步骤

核心原则：板子故障 vs 设计故障——二者必须被分开调试。

“为了有效的 ‘调试-修复’ 循环，关键是确保我们看到的 bug 是真正的设计问题，而非故障的 FPGA 板或在原型方法论中的错误。”

七步递进流程：

  1. 测试基板
  2. 测试基板 + 子板
  3. 单 FPGA 加载小型参考设计
  4. 多 FPGA 加载参考设计
  5. 检查 SoC 设计的实现问题
  6. 按功能子集依次应用真实设计
  7. 应用完整设计

注意：即使在实验室环境中，经验丰富的原型团队完成这些步骤也只需要几小时到几天——但这个投入是"防止浪费数天追踪假 bug"的保险。

11.2 起点：零故障的板子

商业板的优势：已经过供应商工厂的完整生产测试和签核。好的供应商应提供用户可执行的测试工具——如自检终端或自动测试程序。

关键建议：

“花几分钟跑自检，可能节省数小时方向错误的调试分析。”

HAPS 的例子：每块板附带 STB2_1x1 终端块（图133），可插到每个 HapsTrak 连接器上依次测试开路、短路和信号连续性。

**自建

板的需要**：创建和维护一个小型测试设计库来验证板子各部分。

11.3 运行测试设计

典型测试设计清单

测试类型 目的
Signs-of-Life 从外部端口读写寄存器 → 确认 FPGA 自身配置正确
Counter Test 简单计数器 + 时钟/复位输入 → 输出到 LED 或测试点 → 验证时钟和复位正常
子板参考设计 验证子板功能及其与主板的接口
高速 IO 测试 使用与实际相同的物理连线测试 LVDS/HSTDM——环境相关性需真实路径
多 FPGA 测试 简单设计的跨 FPGA 分区 → 验证跨 FPGA 连接和同步

11.3.1 四抽头 FIR 滤波器测试设计

FIR 为什么是理想的多 FPGA 测试（图134+135+136+137）：

  • 全同步单时钟→ 验证复位时序+时钟偏斜
  • 每抽头（乘法器+系数+加法器）分到不同 FPGA
  • 脉冲输入（01h 一个时钟 → 00h 此后所有时钟）→ 预期输出 = 滤波器系数顺序出现

附带收益：运行这个设计 = 完整走通"分区→综合→P&R→位流→配置→调试"全流程。对第一次做原型的团队来说，这是为后续 SoC 设计流程"疏通管道"的最好方式。

11.3.2 构建 Bring-Up 测试设计库

“创建一个可复用的测试基础设施——在不同板子上的已知良好测试库——是一项将惠及多个原型项目的投资。”

11.4 准备上板了吗？

11.4.1 复用 SoC 验证环境

在将 FPGA 就绪版的设计加载到板上之前，最好先用原来的 SoC 验证环境（“Hello World"测试平台）对其再做一次仿真。只有预期中的或故意引入的差异才能被接受；意外的功能差异必须在进实验室前调查清楚。

11.4.2 常见 FPGA 实现问题

三大初次实现问题：

  1. 时序违规：综合/P&R 仅分析单个 FPGA——跨 FPGA 路径和跨时钟域路径不会在标准流程中被高亮
  2. 意外逻辑移除：综合的优化导致模块和 IO “消失”——常见原因是不当的互连或核实例化导致无驱动逻辑。“IO 突然无法解释的下降 = 警示信号”
  3. 跨 FPGA 互连错误：引脚位置约束丢失或被覆盖 → P&R 随机分配引脚 → 源和目的不在同一走线上

11.4.3 时序违规的 8 种类型及对策

类型 要点
内部保持时间 极低时钟速率下看似不可能有时序违规——但保持时间违规与时钟频率无关！必须使用最小延时模型跑全时序分析
跨 FPGA 延迟 使用 X/Y 常量（HAPS-54: X=0.44ns, Y=1.45ns）加入延迟计算
TDM 时序复杂度 复用连接上的多路信号——传输和设计时钟间的延迟约束是关键
IO 时序 尽可能利用 IOB FF——消除内部 FF 到 IOB 的长布线延迟
IOB FF 的不对称使用 微妙问题：三态信号——数据路径用 IOB FF + 三态控制路径不用 → 数据早到但需等三态控制 → 优势被抵消。最坏：到达时间偏斜产生 PAD 输出毛刺
跨时钟域时序 异步时钟间→亚稳态。SoC 中已有的双重锁存等防护可照搬
门控时钟时序 未转化的门控时钟→偏斜导致保持时间违规+可能的毛刺
复用互连上的时序 P&R 后时序分析确认传输和设计时钟延迟在限制内、飞行时间未超预期、异步 MUX 的同步已包含

11.4.4 跨 FPGA 互连错误

数千条信号穿越 FPGA 边界（图139）。一个引脚错位 = 和芯片中一段金属断裂一样破坏性的故障——且更难发现（虽然更容易修复）。

**错误

原因**：约束在流程中的某个工具被丢弃了——常见于"只做了一个小改动但设计突然不工作"的场景。

自动化响应：脚本读取 PAD 报告 → 与"金质"引脚位置参考文件比对 → 在每个设计迭代后自动检查。

11.4.5 与外部世界的互连

命名约定驱动的自动走线分配：信号叫 acknowledge、走线叫 ack、子卡引脚叫 ack → 高级分区工具自动识别并分配。

11.4.6 FPGA IO Pad 配置

规模问题：管理数千路信号的电压标准——每路都要正确配置。一个好的分区工具应自动确保互连引脚使用相同 IO 标准，并在不一致时发出警告。

五大检查项：差分信号 → 外设电压需求 → VCCO/VREF 供电 → 终端阻抗（如 DCI）→ 驱动电流（2-24mA）

驱动电流的隐蔽性：在实验室条件下可能因"运气好"而未暴露——直到软件启用了之前未使用的外设功能。不一致的性能表现 = 典型的驱动电流不足症状。

11.4.6.1 用脚本检查 IO 一致性

创建"金质"参考文件 + 脚本 = 每次迭代自动对比 PAD 报告→无需重复手工检查数千行报告→在进入实验室前捕获配置错误。

11.5 将设计引入板上

分阶段引入的根本原因：一些配置错误会掩盖另一些——分阶段暴露减少这种"遮蔽效应”。

关键检查点：

  1. 编程前确认供电正常→编程后检查 DONE 引脚（HAPS 板上连接 LED）
  2. 配置期间的峰值电流可能触发实验室电源的电流限制→电压轨掉电（brown-out）。这是比板级供电故障更常见的场景
  3. 先检查输入→特别是从其他 FPGA 内部派生的时钟和复位。循环依赖是 SoC 设计在 FPGA 中无法启动的常见原因（FPGA A 的复位 = FPGA B 中的逻辑输出，而后者又依赖 FPGA A 的内部信号）
  4. 先调试单 FPGA 内的部分→排除跨 FPGA 连接问题
  5. 先调试 FPGA 内部的部分→排除外部芯片组交互问题

11.5.1 多 FPGA 的启动状态不正确

问题本质：FPGA 成为 operational 的时间各不相同——即使只差一个时钟沿，第一个寄存器到寄存器的传输就丢失了。

根本原因：PLL/DCM 的内部反馈电路 → 锁定时间可变。外部硬 IO（以太网 PHY 等）比 FPGA 早得多就绪。

解决方案：全局复位管理——所有系统级"就绪"条件汇聚到单点 → 在全部条件满足后才释放——已在第 8 章详细讨论。

11.6 调试板上问题

11.6.1 三类设计 Bug

类型 描述 典型场景
逻辑 Bug RTL 中未发现的错误 真实世界数据依赖——仿真从未暴露
接口 Bug 对外部硬件的未预见交互 特定 IP 组合首次在同一平台中使用
软件 Bug 软硬件首次集成 时间依赖+数据依赖+环境依赖——最难建模

任何特定故障可能同时涉及两类甚至全部三类 Bug。

11.6.2 逻辑设计问题的调试方法

第一步：隔离到 FPGA 级别——需要设计的内部可见性。

可见性层级递进：

  1. 物理探针：测试点和连接器 → 仅限顶层 IO 和关键信号
  2. JTAG 边界扫描：取样 FPGA 引脚值 → 静态快照 → 通过智能触发捕捉时间序列
  3. ChipScope / Identify 嵌入式追踪：内部节点 + BlockRAM 捕获 → 数千采样 × 数千节点

DFP 技巧：RTL 编写者创建"你会最先看哪"的关键节点清单——作为默认插桩的起点。

11.6.3 调试可见性

视角转换：

“与纯 RTL 仿真器相比，FPGA 原型的可见性远不如。但这是错误的比较。原型是在替代最终芯片的位置——因此它实际上比测试芯片或芯片本身提供了远更大的电路行为可见性。”

四种调试可见性方法：

  1. 实时信号探测：RTL 修改→内部节点引到 FPGA 引脚→示波器/逻辑分析仪。可以关联信号行为和其他实时事件
  2. 嵌入式追踪提取：EDA 工具插入采样逻辑+BlockRAM→暂存→后期提取分析
  3. 总线基础插桩：在设计中实例化探测块→总线协议读写→类似 UMRBus
  4. 定制调试器：将 MicroBlaze 等小型 CPU 挂到 SoC 内部总线→检测特定数据组合或错误条件

11.6.4-11.6.5 总线基础调试系统（UMRBus）

JTAG 的带宽瓶颈：

  • 上传软件映像：~30 分钟
  • UMRBus 同任务：远低于 1 分钟

高带宽通道的五种关键能力：

能力 价值
高速 FPGA 配置 配置时间从数分钟降到数秒→初期快速迭代时大量节省等待时间
高性能存储器访问 直接读写原型上数 MB 的内部 RAM→避免通过 SoC CPU 和总线中转
软件映像快速上传 30 分钟→不足 1 分钟→软件开发团队的快速迭代方法论得以保留
高性能数据流注入 从主机以接近真实速率注入视频/音频/图像数据→欺骗 SoC 认为数据来自真实世界
远程配置和管理 远程访问和编程→扫描板类型→自动检测设置→远程初始化/配置/监控

UMRBus 的 CAPIM 架构（图141）：多个 CAPIM 模块通过总线协议互联 + 与主机 USB 通信 → 每个 CAPIM 连接到一个设计块 → 统一的访问和控制框架。

**CO

NFPRO 硬件接口模块**：HAPS-60 上 PCIe 到 UMRBus 的桥接——通过 USB 发送大量数据到板载 supervisor → 以 FPGA 的快速并行模式（而非 JTAG）配置 FPGA。

预定义 IP 库的价值：UMRBus-to-SDRAM 组件——直接访问 SDRAM——无需重新综合或 P&R 更改。

11.6.6 使用嵌入式 CPU 的定制调试器

绝大多数有嵌入式处理器的 SoC 已有内置调试器——可直接利用：

1

SoC 内置调试器 → 专用 IO 端口 → FPGA IO 引脚 → 测试点引出 → 笔记本电脑上调试软件（USB 连接）

流程：连接内部调试器 → 读写处理器总线上所有内存空间 → 加载简单程序运行 → 确认基本功能 → 在真实设计上做实际软件调试。

11.7 组合使用多种调试技术

调试更多是直觉而非发明——但"有经验的工程师 + 多样的工具"是最佳组合。

TFT 显示驱动的组合调试示例：

  1. 定制调试器：读写外设寄存器 → 检查基本硬件访问
  2. Identify：在处理器总线上设触发点（=该寄存器地址）→ 追踪信号确认数据正确
  3. 示波器/逻辑分析仪：直接查看 TFT 连线信号 → 确认寄存器读写时信号适当

11.8 12 条实战调试技巧

# 技巧 说明
1 分而治之 逐个接口测试 → 先用可见的（探针外部接口信号）再深入不可见的（ChipScope 探内部状态机）
2 检查外部芯片 问题可能在 FPGA 之外——用 FPGA 内实现一个可综合模型替代外部芯片做对照测试
3 调整 IO 延迟 IODELAY 可编程绝对延迟——调整时钟和数据路径延迟→确保采样正确
4 降低时钟速率 系统不是"死的"但行为异常→降时钟→排除时序边缘问题
5 理解关键设置需求 SoC 内部寄存器可能默认处于睡眠模式需要写入有效值才能激活；外部传感器增益寄存器可能需要非默认配置
6 检查内置安全机制 安全锁单元检查内部 ID 寄存器中的特定值——不存在则原型看似活着但无响应。Boot 代码期望 Flash 中存在密钥才能继续
7 挑战基本假设 大端/小端匹配→发射/接收端口直连或交叉→这些"显然"在跨 FPGA 和外部芯片时都可能错
8 终端电阻 I2C/PCI/SPI 等接口的物理终端 → 连接器是否间歇接触→局部电压轨问题
9 从外向内测试 IP 验证接口而非 IP 功能本身——IP 应是充分测试过的。但如果 IP 是 RTL 形式且不被供应商支持用于 FPGA → 此假设需要被挑战
10 复用板子：从已知良好的开始 从之前项目验证过的 IP 或设计块开始——已知良好的 USB IP + CPU = 验证总线工作 → 以好基础调试新外设
11 先测 GTX 引脚，再测协议 IBERT（ChipScope 内置）→ 评估和监控收发器物理层健康 → 确认后再上协议分析仪
12 使用高级分析仪 实时协议检查、总线性能统计、总线延迟监控。无外部设备→可综合的总线 exerciser + analyzer 自己写

11.9 Bug 修复后的快速周转

调试变成迭代时——大型 SoC 综合+P&R 可能需要 24+ 小时。

调试计划：类似于 SoC 整体的验证计划。每日构建 → 明确每次构建激活哪些部分 → 多份原型拷贝并行使用时特别有用。

小改动的优化：只改动一颗 FPGA → 只需重新合成这颗 FPGA → 前面的分区保持不变 → Pre-synthesis 分区显著加速。只需几个小时甚至几十分钟。

多处理器并行：多个 CP 同时综合 → 即使是全量重构也能缩短总运行时间。

个人总结

第11章是全书中最"接地气"的一章。前10章都在讲怎么正确地做，这一章讲的是"当事情不如预期时怎么找到原因"。

对调试的分类（逻辑/接口/软件三类 Bug）比看上去更深层——它实际上定义了原型验证能发现什么类型的价值。逻辑 bug 应该由 RTL 仿真发现——原型发现它们其实是意外收获。接口 bug 是"多个 IP 首次在同一平台协同"的常见产物。而软件 bug——软硬件首次集成时暴露的问题——才是 FPGA 原型真正的价值所在。正是在这个阶段，原型从一个技术工具变成了一个商业价值引擎。

本章最实用的哲学就是**“先验证平台，再验证设计”**。这看似在 bring-up 阶段花更多时间，但"不要将板子问题和设计 bug 混为一谈"是 debug 的第一铁律——违反了它，你可能花三天追踪一个不存在的 RTL bug，而根因只是一根线缆接触不良。

关于 UMRBus 的讨论也指向了一个重要的行业方向：JTAG 带宽（KB/s 级）→ USB/PCIe 带宽（MB/s-GB/s 级）→ 调试范式的质变。当软件映像加载从 30 分钟降到 30 秒，这不仅节约了时间——它让软件团队保持了他们的"快速迭代─测试─修复"的惯常工作方法论。这比任何单一调试功能都更能提升软件团队对原型的接受度。

实践思考

  • FIR 测试设计是"零投资高回报"的起点——花 2 小时写一个简单的全同步测试设计并成功跑在多 FPGA 上，比直接上 SoC 设计然后花 3 天追踪"是板子还是设计的问题"要划算得多
  • IOB FF 的不对称使用是真实项目中的高频陷阱——在代码审查时专门检查三态信号的数据和控制路径是否一致使用 IOB FF
  • IO 约束验证脚本是"睡得着觉"的保障——在 makefile 中将 PAD 报告与"金质"引脚的自动比对作为 P&R 后的强制步骤。一个引脚错位 = 数天的不必要调试
  • 循环依赖是 SoC 启动失败的常见但隐蔽的原因——检查"FPGA A 的复位是否依赖 FPGA B 中需要 FPGA A 信号的逻辑"——在分区时就应该被识别和打破
  • 不要在 JTAG 上浪费软件团队的时间——如果原型的主要用户是软件团队，投资建立高带宽通道（USB/PCIe+UMRBus 或等效方案）是软件团队接受原型的必要前提
  • Compile Point + Design Preservation 组合是迭代阶段的"生产力放大器"——需要花时间初始设置，但从第三次设计迭代开始就以 2-5× 的速度回报
