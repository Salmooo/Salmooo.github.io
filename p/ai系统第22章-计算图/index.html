AI系统：第22章 计算图

《AI系统：原理与架构》学习笔记

2 minute read

本章概览

第 22 章深入探讨 AI 框架的核心抽象——计算图（Computation Graph）。计算图作为 AI 框架中连接前端表达与后端执行的关键数据结构，贯穿神经网络模型的整个生命周期——从前向计算到自动微分、从编译优化到运行时调度执行。本章从计算图的基本原理出发，逐步深入到自动微分的实现、调度与执行策略、控制流的表达，以及动态图与静态图的转换技术。

22.1 引言

目前主流的 AI 框架都选择使用计算图来抽象神经网络计算表达。神经网络的训练流程主要包括前向计算、计算损失、自动求导、反向传播和更新模型参数 5 个过程。在基于计算图的 AI 框架中，这 5 个阶段统一表示为由基本算子构成的计算图，算子是数据流图中的一个节点，由后端进行高效实现。

在实际程序实现中会遇到的编程性问题（如控制流 if/else/while/for），本章也将深入探讨计算图和控制流之间的关系，以及计算图的调度和执行方式。

22.2 计算图基本原理

22.2.1 AI 系统化问题

在 AI 工程化过程中面临的挑战包括：

  • 如何对复杂神经网络实现自动微分？
  • 如何利用编译期分析 Pass 对计算进行化简、合并、变换？
  • 如何规划 Kernel 在 GPU/TPU/NPU 上高效执行？
  • 如何进行内存预分配和管理？

为了解决这些问题，目前主流的 AI 框架都选择使用计算图来抽象神经网络计算。

计算图定义：被定义为有向图，其中节点对应于数学运算，计算图是表达和评估数学表达式的一种方式。在 AI 框架中，计算图就是一个表示运算的有向无环图（DAG）。

22.2.2 计算图的基本构成

数据表达方式：

  • 标量（Scalar）：只有大小没有方向的量，是零阶张量
  • 向量（Vector）：有大小和方向的量，是一阶张量
  • 矩阵（Matrix）：二维数组，是二阶张量
  • 张量（Tensor）：向量和矩阵的推广，是 AI 框架中数据的统一表示

在 AI 框架中，图像任务通常将图片组织成三维张量 (C, H, W)，NLP 任务中句子组织成二维张量，增加 Batch 维度后变成四维张量 (N, C, H, W)。

张量表示的重要因素：

  • 元素数据类型：整型、浮点型、布尔型等
  • 形状（Shape）：描述维度及每个维度的长度

计算图的两个核心元素：

  • 节点（Node）：在 AI 框架中代表算子（Operator）——具体的计算操作
  • 边（Edge）：代表张量（Tensor）——数据在节点间的流动

22.2.3 PyTorch 计算图

PyTorch 使用的是动态计算图，有两重含义：

  1. 前向传播立即执行：无需等待完整计算图创建，每条语句动态添加节点并立即执行
  2. 反向传播后立即销毁：调用 Backward 后计算图被销毁，下次调用需重新构建

PyTorch 中的 Function 是计算图的另一类节点，同时包含前向计算和反向传播逻辑，通过继承 torch.autograd.Function 来创建自定义算子。

22.3 计算图与自动微分

22.3.1 深度学习训练流程

训练过程分为三步：

  1. 前向计算：输入数据 → 模型输出预测值
  2. 计算损失：真实值与预测值比较，对各参数求偏导
  3. 更新权重：根据优化器策略，沿梯度方向更新参数 \(w = w - \eta \cdot \text{grad}\)

反向传播算法实际上就是自动微分。在 AI 框架中，计算图的根节点为损失函数，节点间的偏导称为误差项。

22.3.2 自动微分基础回顾

自动微分（Automatic Differentiation, AD）按工作模式分为：

  • 前向自动微分
  • 反向自动微分

按实现方式分为：

  • 基于对偶数的前向微分
  • 基于 Tape 的反向微分（如 PyTorch）
  • 基于源代码转换的反向微分

反向微分的优势：在神经网络中，输出是标量损失函数，保留前向计算的所有中间结果，只需运行程序一次即可计算出损失函数对所有参数的导数。时间复杂度为 \(O(m)\)，其中 \(m\) 是输出变量个数。

22.3.3 AI 框架自动微分方式

在反向模式的自动求导中，根据反向计算图的构建时机，分为两种方式：

动态计算图（表达式追踪）

PyTorch 采用基于磁带的自动微分（Tape-Based）：在前向计算时记录操作到 Tape 上，反向时重放 Tape 计算梯度。

优点：易用性高，方便调试 缺点：需保存大量中间结果，内存占用较高

静态计算图

图层 IR 方式：静态生成计算图，在执行前完成构图和编译优化。

优点：可全局编译优化、内存复用、执行效率高 缺点：错误不易发现、调试困难

AI 框架在实现自动微分时，对每个中间层存储的是向量-雅可比乘积而非完整的雅可比矩阵，以减少存储空间。

22.4 计算图的调度与执行

22.4.1 图调度

算子定义：

  • 狭义算子（Kernel）：对张量执行的基本操作集合（四则运算、数学函数等），不感知微分
  • 广义算子（Function）：配合 Autograd 模块支持自动梯度求导

算子间调度（FIFO 队列）

调度策略：通过先进先出（FIFO）队列执行：

  1. 将入度为 0 的节点加入 FIFO 队列
  2. 从队列中选择节点分配给线程执行
  3. 节点执行结束后将其后继节点加入就绪队列
  4. 重复直到队列为空

算子并发调度

找到相互独立的算子进行并发调度，通过后端编译优化 Pass 提供并发执行队列：

策略：低代价节点在当前线程执行，高代价节点派发给线程池中其他线程并发执行。

算子异构调度

在手机端侧，CPU、GPU、NPU 等多种异构计算 IP 共同参与计算：

  • CPU 算子：利用 CPU 多核心能力
  • GPU 算子：为高度并行 Kernel 提供加速
  • NPU 算子：为高维张量提供专用 Kernel

异构调度的挑战：调度逻辑复杂、上下节点依赖重、通信开销常大于计算开销。

22.4.2 图执行

三种执行模式：

单算子执行

将计算图展开为执行序列，按序列逐个 Kernel 执行。PyTorch、TensorFlow Eager、MindSpore PyNative 均采用此方式。

优点：灵活性高，可直接通过 Python 调度，便于调试 缺点：缺乏全局优化信息，无法完成算子融合、代数简化等编译优化

整图下沉执行

将整个计算图或子图一次性下发到 DSA 芯片（如 TPU、昇腾 NPU）执行。避免了 Host 与 Device 频繁交互，获得更好的整体计算性能。

局限：动态 Shape、复杂控制流、副作用等场景面临技术挑战。

图切分到多设备执行

将大模型计算图拆分到不同 AI 加速芯片上执行，兼顾性能和灵活性。

22.4.3 PyTorch 算子执行

PyTorch 调用 torch.add 时经历两次调度：

  1. 第一次调度：根据设备（Device）和布局（Layout）动态选择实现函数（如 CPU/Strided、CUDA/Strided 等）
  2. 第二次调度：根据数据类型通过 Switch 分支静态选择具体的 Kernel 执行

22.5 计算图的控制流实现

计算图在数学上是 DAG，但编程中需要 if/else/while/for 等控制流。引入控制流后可以构造更复杂的神经网络结构。

22.5.1 控制流

引入控制流的两大难点：

  1. 计算图变为动态，分支选择依赖运行时数据
  2. 循环导致计算图出现环，无法有效拓扑排序

三类设计方案：

方案 代表框架 核心思路
复用宿主语言 PyTorch 用 Python 控制流驱动后端执行
支持控制流原语 TensorFlow 计算图中引入控制流原语，运行时原生支持
源码解析 MindSpore 前端代码 → AST → 静态计算图（含控制流子图）

22.5.2 动态图（复用宿主语言）

PyTorch 采用动态图机制，遇到控制流时使用 Python 原生方式执行，遇到 PyTorch API 时才按动态计算图执行。这被称为"模型即代码“的声明式编程方式。

优点：

  • 易用性高，即时输出中间结果
  • 模型定义如同编写普通程序

缺点：

  • 易于滥用前端语言特性，带来性能问题
  • 执行流在语言边界来回跳转，运行开销大
  • 不利于大规模部署
  • 跨语言边界优化困难

22.5.3 静态图

控制流原语（TensorFlow）

TensorFlow 引入控制流原语（Switch、Merge、Enter、Exit、NextIteration），运行时以**执行帧（Execution Frame）**管理。

执行帧可嵌套，对应嵌套控制流。不同执行帧中无数据依赖的算子可并发执行。

优点：编译期可全局优化，执行效率高 缺点：编程习惯差异大，构建步骤相对复杂

源码解析（MindSpore）

通过 AST 解析 Python 源码为计算图，控制流展开为子图。遇到 if/else 分别创建两个子图，运行时动态选择。

优点：解耦宿主语言与执行、继承静态图优化优势 缺点：部分宿主语言控制流不能表示、带有一定约束

22.6 动态图与静态图转换

22.6.1 背景

发展三阶段：

  1. 动静分离（2017-2018）：TensorFlow 默认静态图，PyTorch 默认动态图
  2. 动静结合（2019-2022）：TensorFlow 2.0 支持动静态图转换，PyTorch 2.0 支持 Compiler
  3. 动静统一（目标）：灵活在动态图与静态图间切换，是 AI 框架的终极追求

转换路径：

  • 动态转静态：运行中通过 JIT 自动转换，如 PyTorch Lazy Tensor
  • 静态转静态：编译中发现不支持语法则 Fallback 回 Python

22.6.2 追踪模式（Tracing）

基于追踪的方式直接执行用户代码，记录算子调用序列并保存为静态图。

两个阶段：

  1. 创建并运行动态图代码，捕获所有算子构建静态图
  2. 再次调用相同模型时，自动指向静态图执行

优点：简单易实现，广泛支持动态控制流 缺点：只能保留一次执行轨迹，缺失其他分支；循环无法有效追踪全部迭代

22.6.3 源码解析（Parsing/Script）

三个阶段的转换流程：

  1. 词法分析：将源代码转换为词法单元列表
  2. 语法分析：词法单元 → AST → AI 框架内部 AST
  3. 转换：内部 AST → 别名分析 → SSA 化 → 类型推断 → 静态计算图

PyTorch Dynamo 属于基于源码转换，但在 Python 解释器中转换成 FX Graph，而非对 Python 语言本身进行转换。

优点：能保留完整控制结构，更广泛支持动态控制流 缺点：Python 动态类型导致类型推导复杂；过度灵活的控制流会 Fallback 回 Python

小结

第 22 章系统阐述了计算图作为 AI 框架核心抽象的全貌：

  • 基本原理：计算图 = 算子（节点）+ 张量（边），是有向无环图，统一表达神经网络计算
  • 自动微分：反向传播即为自动微分，分为动态图（Tape-Based）和静态图（图层 IR）两种实现
  • 调度执行：从单算子串行 → 并发调度 → 异构调度 → 整图下沉，性能逐级提升
  • 控制流：三种方案（复用宿主语言/控制流原语/源码解析）各有权衡
  • 动静转换：从 Tracing 到 Script/Parsing 再到 Dynamo，追求易用性与性能的统一

核心启示：计算图的设计决定了 AI 框架的表达能力、优化空间和执行效率。动静统一是 AI 框架技术演进的终极方向。
