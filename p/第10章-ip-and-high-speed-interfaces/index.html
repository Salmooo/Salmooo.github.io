第10章: IP and High Speed Interfaces

Jun 06, 2026
5 minute read

章节概览

现代 SoC 中 IP 块占比已超过 50%（见第1章）。第10章专门讨论如何在 FPGA 原型中处理这些来自"外部"的设计块——无论它们是以 RTL 源码、加密网表、加密位流、还是物理测试芯片的形式交付。

章节结构从抽象到具体：IP 交付格式的五种分类 → 每种格式的 FPGA 适配策略 → 软 IP（DesignWare 为例）的三种替换路径 → 外设 IP 的三种使用模式 → 速率适配器的设计 → 外部硬 IP 的集成考量 → FPGA 原生 IP 作为替代的三种包含方式。

一个贯穿始终的主题：IP 在原型中的目的不是验证 IP 本身（那应该由 IP 供应商保证），而是不在设计中留下空洞——让硬件和软件的其余部分能在 IP 存在的条件下得到验证。

核心知识点

  • 五种 IP 交付格式（Table 25）：RTL 源码 / 加密源码 / 软 IP（中间形式）/ 网表 / 物理 IP（硬 IP，已由晶圆厂完成版图）
  • 非 RTL 交付格式的优劣对比（Table 26）：模型（仿真好但不可用于原型）< 加密 FPGA 网表（灵活性低）< 加密 FPGA 位流（AES-256，非常安全）< 测试芯片（最高性能、最低灵活性）
  • ARM SMM（Software Macro Model）：加密 FPGA 映像的典型代表——故意省略 MMCM 等时钟基础设施 → 允许低于最低锁定频率的慢速时钟 + 单步调试
  • AES-256 双密钥方案：电池后备密钥（最安全，断电即丢失，无已知提取方法）vs 多晶 eFUSE 密钥（安全性较低，破坏性拆解可读——但有 3 倍冗余位混淆）
  • 软 IP 三种包含方式：实例化（最直接）→ 推断（通过函数调用或运算符）→ 合成算子（综合工具将语言运算符映射到优化模块）
  • 外设 IP 三种使用模式（Table 28）：验证 IP 自身（最高速度+周期精确）→ IP 作为 SoC 的组成部分（缩放速度+周期精确）→ 软件验证（“足够快”+功能等价即可）
  • 速率适配器的两种方案：FIFO 吸收全速突发数据→降速转发；移位寄存器 FIFO 缓冲突发→全部接收后再降速传出。连续数据流会溢出，只适用于突发数据
  • FPGA IP 包含的三种方式：Plain Text Netlist（综合读取+优化+含入）→ White Box（仅读取时序信息，不优化逻辑）→ Black Box（占位符，P&R 阶段才填入预生成的网表）
  • PLL/MMCM 的最低频率限制是硬性约束：试图在低于规范的最低频率下运行"即使在实验室似乎能工作也很冒险"

详细笔记

10.1 IP 与原型验证

本书对 IP 的定义：不是项目内部产生的，而是从"外部"引入的设计块。仅讨论数字 IP——模拟或混合信号 IP 显然不能在 FPGA 中工作。

原型验证中 IP 的特殊角色：

  • IP 组件是预测试过的→应该按规范工作 → 原型任务简化
  • 但 SoC 设计很可能是特定 IP 组合首次被放在同一平台中
  • IP 供应商应该愿意并能够支持其 IP 在 FPGA 原型环境中的建模 → SoC 团队在选择 IP 时就应该询问供应商"你们对原型团队有什么支持？"

10.2 IP 的多种形式

Table 25：五种交付格式

格式 描述
RTL 源码 完整的 Verilog/VHDL，供应商授权或开源
加密源码 RTL 受保护，需显式或工具流内自动解密
软 IP 中间形式交付（可能加密），需要后端处理
网表 预综合的 SoC 库单元或通用门级元素（如 GTECH）
物理 IP（硬 IP） 晶圆厂预布局；原型中以模型或测试芯片呈现

10.2.1 RTL 源码 IP

  • 与 SoC 设计中其他 RTL 没有本质区别——但许可证可能禁止修改 RTL
  • 供应商可能已有 FPGA-ready 版本（可能需要额外许可证和费用）
  • 即使功能完全等价，为前沿 SoC 工艺优化过的 RTL 在最快 FPGA 上也会慢得多

10.2.2 RTL 不可得时

IP 污染：在公司内部或向第三方开发者大范围分发 RTL→意外（甚至蓄意）修改→许可和支持纠纷。

IP 防盗：无需解释。

四种替代 RTL 的交付格式对比（Table 26）：

格式 优势 局限 安全性
模型 仿真极好 性能限于仿真器；不能用于原型 “安全”——因为不是真实 IP
加密 FPGA 网表 比 RTL 难逆向 仿真无可见性 仅与加密一样安全
加密 FPGA 位流 比仿真快 灵活性低、可能需妥协 非常安全
测试芯片 最高性能 最低灵活性；开发周期长 非常安全

关于模型的关键说明：模型不是用来综合的——它是用来在失去 RTL/测试芯片可见性时恢复部分可见性的。一个好的 IP 供应商应该随 IP 提供丰富的补充模型支持。CPU IP 还提供指令集仿真器（ISS）——对周期精度几乎无知但极快——最适合早期软件开发（第13章讨论 SCE-MI 接口将 ISS 模型与真实 RTL 桥接）。

10.2.3 加密源码

  • 如果加密仅用于安全运输 → 解密后拥有 RTL → 完全可按正常流程原型化
  • 如果 RTL 在工具流中始终加密（“On-the-Fly"解密）→ 需要每个工具内置支持解密和密钥
  • 两个常见例子：Synopsys 的 synenc 加密流、Xilinx 的加密 .ngc 文件
  • 行业将最终走向 IEEE 标准的 IP 加密和封装

10.2.4 加密 FPGA 网表

  • 交付给公司纳入自己的 FPGA 设计，但安全级别低——因为必然在流程中被解密
  • 最终映像文件会再次加密，确保只能在目标 FPGA 硬件中以确定性方式使用
  • 部分 IP 有限时使用限制 + 需要特定下载/调试硬件

10.2.5 加密 FPGA 位流（最安全的FPGA IP交付方式）

双密钥方案：

方案 机制 安全性
电池后备密钥 密钥存储在电池供电的 key memory 最高——无已知方法提取；断电即永久丢失
多晶 eFUSE 密钥熔断到物理熔丝 较低——破坏性拆解可读，但有 3× 冗余位混淆

工作方式：FPGA 内部完成全部解密，只有加密的 FPGA 映像在外部可见——供应商交付高价值 IP 的极佳方式。

ARM SMM（Software Macro Model）的关键特性：

  • FPGA 映像复制了 IP 块设计的集成（顶层）级别（图123）
  • 故意不包含 MMCM 等时钟基础设施 → 时钟速度可低于通常的最低锁定频率 → 支持时钟停止和单步调试
  • 用户需自行为 FPGA 内部逻辑的不同时钟域提供时钟和引脚复用

10.2.6 测试芯片

最安全但也最昂贵的交付方式。通常仅适用于广泛使用的高价值 IP（如 ARM CPU 核）。

测试芯片的妥协清单：

  • 预定义接口和配置选项 → 限制使用方式
  • 存储映射或中断结构可能与最终 SoC 使用不匹配
  • 选择性遵循周期精度
  • FPGA 和测试芯片间使用异步桥接 → 引入延迟
  • 信号复用以适应连接器引脚限制
  • 功能限制（禁用测试模式、减少中断、片上合并总线）

软件或系统设置必须适应测试芯片的能力，而非相反。

测试芯片的最佳用途：支持 OS 开发、基准测试、通过 AXI 总线连接的扩展 IP 块开发（以 ARM 为例）。

宽总线的挑战：SoC 和测试芯片间需引脚复用 → 引入 IO pad 和板级延迟 → 降低系统速度 → 可用高速串行信令 + HSTDM 缓解（第8章）。

10.3 软 IP（以 DesignWare 为例）

软 IP 定义：物理实现由终端用户决定的任何形式的 IP。包括 RTL、网表、二进制形式、预编译中间库。通常用于性能和面积目标相对容易实现的较低价值 IP。

SoC 设计中使用软 IP 的三种方式

一：实例化（Instantiation）（图127）

1

DW02_mac #(A_width, B_width) U1 (...)

→ FPGA 合成中的黑盒 → 需"填充"黑盒

二：推断（Inference）（图128）

1
2

`include "DW02_mac_function.inc"
assign MAC_func = (func_TC) ? DWF_mac_tc(...) : DWF_mac_uns(...);

→ 综合工具的搜索路径指向等效的功能定义

三：合成算子（Synthetic Operators）（Table 27）

1

c <= a * b;

→ 综合工具识别 * → 绑定到合成算子 → 执行算术和资源共享优化 → 映射到 FPGA 的 DSP 资源

“因为这种自动化和简单性，SoC 团队应尽可能使用合成算子而非直接实例化软 IP。”

10.3.1 替换实例化的软 IP

最佳实践：

  1. 选择 IP 时就确保有已验证的 FPGA 等价物
  2. 在原始设计中用 `ifdef 在两种实例化间分支
  3. 更理想的做法：单一实例化同时服务两个目标——工具在不同流程中提供适当内容 → SoC 设计师无需做任何特殊安排

Synopsys 的内部实践：在班加罗尔，曾将所有 DesignWare 构建块逐一分析并创建功能等价的 RTL——任何正确实例化的 DesignWare 元素都会被自动解释（而非黑盒）。

最新进展：Synopsys 修改了 FPGA 合成工具允许原生使用 DesignWare 块；DesignWare 开发者本身也对部分块做了 FPGA 优化。

10.3.2 替换推断的软 IP

与实例化的类似，但问题不是"填充黑盒"而是让 FPGA 工具从给定的 RTL 中推断出与原始 SoC 综合相同的功能。

关键操作：将 SoC 综合工具（Design Compiler）的搜索路径中指向的 DesignWare 功能集，在 FPGA 工具中替换为等效路径或本地 include 文件。

10.3.3 替换合成软 IP

FPGA 原型中最简单的场景：

SoC 综合中 a * b 通过 pragma map_to_operator 绑定到 dw02_mult → FPGA 综合中相同的 * 被自然地推断为乘法器 → 自动映射到 FPGA 的 DSP48 专用资源。

10.4 外设 IP

三种使用模式（Table 28）

模式 速度需求 精度需求
验证 IP 本身 最高速度（达最终芯片速度） 周期精确
IP 作为 SoC 组成部分 缩放速度 周期精确
软件验证 “足够快” 功能等价

10.4.1 模式一：验证 IP 本身

本质上是小型 SoC 原型项目的子集——许多 IP 设计能放入 单一 FPGA（避免了分区！）。如第2章介绍的 Synopsys 波尔图 IP 团队的 HDMI 原型（图130）：FPGA 内一部分是待验证 IP，另一部分是从 Xilinx IP 库中组装出的验证环境基础设施。

10.4.2 模式二：IP 作为更大 SoC 的组成部分

系统整体速度很可能受限于 SoC RTL 映射到 FPGA 的性能 → 挑战变为将外设 IP 的速度与 SoC 核心的速度进行缩放匹配。

10.4.2.1 核心与外设 IO 之间的速率适配

问题：外设数据到达速度超过 FPGA 能处理的速度。

方案一：FIFO 作为速率适配器

工作方式：以全速（或标准允许的最低速率）吸收外部刺激 → 仅处理子集 → 以降速时钟驱动到 FPGA 内部。

原则：如果原型跑 SoC 真实速度的 1/3 → 应只接收 1/3 的刺激量。

方案二：移位寄存器 FIFO 缓冲（图131）

工作方式

：

  1. 预测突发中会到达的最大数据量
  2. 以全速率接收全量数据到移位寄存器 FIFO
  3. 全部接收后以降速传给 FPGA 原型

相同方法用于反向——从 FPGA 降速 → 加速到外部接口。

速率适配的局限：

  • 对规则的突发数据包效果极好
  • 能处理深数据（只要有足够 FIFO 存储器）
  • 连续数据流会溢出 → 数据包丢失或需在原型准备好前丢弃
  • 但在许多设计中这种方法极其成功

图131 的特定例子：两个外部 USB 2.0 通道 125MHz → 异步 FIFO 缓冲 → “嗅探器"电路识别完整数据包 → 以 1/10 速度读入原型。

“考虑内外数据流间缓冲的最佳时机是SoC 最初被设计时。让原型团队在早期参与 → 速率缩放设计按时完成 → 软件团队预判必要的缩放代码变更。”

10.4.3 模式三：为软件验证原型化 IP

软件不需要周期精度，只要 “足够快” + 功能近似 + 通道同步 即可。

极端情况：完全替换 SoC IP 为 FPGA 版本。甚至可以替换为 transactor（事务处理器）——执行相同功能的更高层抽象（第13章将详细讨论）。

10.5 在原型中使用外部硬 IP

最常见的硬 IP 块：CPU 核心 + 标准总线接口 PHY（Ethernet、USB、PCI/PCI-X/PCIe、DDR2）。

使用外部硬 IP 的副产品优势：释放 FPGA 资源——大型 IP 块（DSP、处理器子系统）很容易消耗整颗甚至多颗 FPGA。如果 IP 需要在多 FPGA 间拆分，成本 + 性能代价可能太高（尤其没有 IP 内部可见性时）→ 此时使用外部测试芯片是最优选择，即使接受其局限性。

10.6 用 FPGA IP 替换缺失的结构

FPGA 供应商（如 Xilinx）提供广泛的 IP 核——但不能在 SoC 中使用（Xilinx 仅许可在其自家器件中使用）。

FPGA IP 的三种包含方式：

方式 综合行为 优化 适用场景
Plain Text Netlist 读取核的 FPGA 级网表 → 根据需要优化 → 包含进输出网表 + 传递时序约束 完全优化 非加密 IP
White Box 仅读取核的边界时序信息 → 不优化核的逻辑 仅边界信息 安全的加密 IP
Black Box 完全忽略核的网表 → 仅留"占位符” → P&R 阶段才用预生成网表填充 完全不优化 最安全的加密 IP

10.6.1 外设 IP 实例：PCIe-to-SATA Bridge

Synopsys 都柏林 IP 实验室在 HAPS-51 + 单颗 Virtex-5 LX330 上实现了完整的 PCIe 和 SATA 接口以及桥接逻辑。

FPGA 特定修改：pipe2v5gtp 和 sata2v5gtp——将标准 Pipe 接口（PCIe PHY 与逻辑间的中间接口）适配到 Virtex-5 GTX 收发器物理层。

“这是对现代 FPGA 器件规模的又一证明。”

10.6.2 速度问题：最低速度限制

PLL/MMCM 的最低工作频率是硬性约束：

“在实验室里低于规范运行似乎能工作——但这是危险的。”

SoC PLL 和 FPGA PLL 可能有不同的抖动、占空比、频率、漂移和精度 → 必须确认 MMCM 或其他时钟电路的精度足够接近 SoC 基础设施以产生有意义的结果。

个人总结

第10章是一章关于"如何应对’不知道里面是什么’的设计块"的操作指南。它与第7章（处理你知道里面是什么的设计块）形成互补——第7章教你修改 RTL，第10章教你处理没有 RTL 的情况。

最有价值的观点是 “IP 存在的目的是不在设计中留洞”。原型团队不需要验证 IP——那是 IP 供应商的职责——但原型团队需要 IP 以某种形式存在，以便周围的硬件和软件能在完整系统上下文中得到验证。这个视角转移意味着我们在选择 IP 替代方案时，功能等价性优先于性能，可用性优先于完美匹配。

速率适配器是连接"高速外设世界"和"降速原型内部世界"的桥梁。它并非 SoC 设计的一部分，而是纯粹的原型工程创造——这正是本章标题中"额外工程"的含义。这种工程的价值需要架构预判（第9章的 DFP 原则）——如果在 SoC 架构设计阶段就考虑速率适配，可以避免后期被迫在缺乏缓冲机制的情况下重新设计接口。

加密位流 + AES-256 是 IP 供应商和原型工程之间建立信任的技术基础——供应商放心交付高价值 IP，原型团队获得接近完整功能的 FPGA 实现。

实践思考

  • “选择 IP 时就应该问供应商对原型有什么支持”——这应该成为 SoC 项目 IP 选型决策的一个正式评估维度。一个有 FPGA 版本或测试芯片的 IP 比"仅 RTL"的 IP 能为项目节省数周甚至数月的原型移植时间
  • 合成算子（Synthetic Operator）是最优的软 IP 包含方式——c <= a * b 自动映射到 FPGA DSP48，而不需要在 RTL 中做任何 IP 实例化。在产品编码标准中应该鼓励使用语言内置运算符而非直接例化 IP 模块
  • 速率适配器的 FIFO 深度计算：最大突发长度 × (接口速率/原型速率) × 数据宽度——如果这个值超出可用 BlockRAM，则需要降速、降突发、或接受数据丢失
  • White Box 是加密 IP 的最佳折中——既保护了 IP 内部逻辑的机密性，又给综合工具足够的边界时序信息来优化周围逻辑
  • PLL 最低频率约束可能导致原型无法做慢速单步调试——这就是 ARM SMM 故意省略 MMCM 的原因。在自己的原型中如果也需要单步调试，可以采用类似架构
  • 测试芯片和 FPGA 之间的宽总线复用几乎总是必要的——在规划平台互联时就要考虑为测试芯片预留足够的 LVDS 差分对或高速串行通道
