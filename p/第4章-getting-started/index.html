FPGA学习笔记

第4章: Getting Started

Jun 04, 2026
5 minute read

章节概览

前三章讲了 Why（为什么做原型）和 What（FPGA 技术是什么），第四章终于进入 How——如何启动一个 FPGA 原型项目。

本章是一份完整的项目启动手册，回答了六个核心问题：(1) RTL 什么时候才算准备好了？(2) 需要多少 FPGA 资源？(3) 原型能跑多快？(4) 处理一遍设计要多久？(5) 工程工作量多大？(6) 团队需要什么技能？

最有实践价值的发现包括：初始利用率不超过 50% 的经验法则、基于综合报告计算 FPGA 数量、四 FPGA 系统首次实现耗时 10-16 人周的估算基准，以及"至少需要一个会看 RTL + 懂 FPGA + 擅长 bench 调试的专职工程师"的团队配方。

核心知识点

  • 启动清单（Table 7）：分 Setup 和 Usage 两大阶段，覆盖选 FPGA、制板/买板、设计移植、Bring-Up、跑软件、实地测试的全流程
  • RTL 准入标准：TI 团队要求覆盖率达到 80% 才放入原型；同步到 ASIC 流程的试实现阶段（“Hello World"测试通过）是实用的交接点
  • FPGA 资源量化（LX760）：~750K LUT + ~950K FF ≈ 400 万 SoC 门（满载）或 250 万门（合理利用率），720 个 36Kb BlockRAM ≈ 26Mb，864 个 DSP48E1，1200 IO，18 个 MMCM
  • 利用率黄金法则：初始 ≤50%（为后期增长和调试逻辑留余地），极限 ≤75%（生产 FPGA 设计的上限）
  • 从综合报告反推 FPGA 数量：总 LUT 需求 / (单器件 LUT × 50%) = 所需 FPGA 数
  • 知名 IP 的 FPGA 消耗量：ARM Cortex-A5（小型配置）≈ 2 颗 LX330 @50%，ARM Mali-400 w/3 pixel processors ≈ 4 颗 LX330 @50%
  • 首次实现耗时：4 FPGA 系统、50% 利用率、宽松时钟 → 10-16 人周（5-8 周 / 双人团队）
  • 后续迭代耗时：小改动 1-4 小时/FPGA（增量模式）；中改动需分区调整 → 最多几天
  • 时钟和 IO 常比逻辑更早成为瓶颈：高互连设计即使逻辑资源够用，仍可能因 FPGA 间连线耗尽而受限
  • 运行时间以小时计而非天：日常应选择"白天改代码、晚上跑脚本"的节奏

详细笔记

4.1 启动清单（Table 7）

两阶段框架：

阶段 任务 考量
Setup 选择 FPGA 容量？速度？IO？
自建 vs 买板 内部能力？时间？预算？
设计移植到 FPGA 原型三定律、最小化设计改动、构建追踪

关键事实：大部分工作在 Setup 阶段，大部分收益在 Usage 阶段。

这个清单的主线 = 本书后续各章的路线图。

4.2 估算所需资源：FPGA

作者给出一个血淋淋的教训：有新手严重低估 FPGA 资源需求，导致内部设计制造的整个平台报废，数周项目进度付诸东流。

4.2.1 RTL 需要多成熟？

核心矛盾：越早做原型越好（给软件团队争取时间） vs 太早做的 RTL 不成熟（浪费时间于仿真器就能轻易发现的初级 bug）。

行业实践两个极端：

  • TI（保守派）：RTL 代码完整、覆盖率达到 ≥80% 后才放入原型
  • 激进派：代码完成 75% 即可提前交付 RTL

实用建议：将 RTL 的首次交付时点同步到 ASIC 流程开始试实现的时间点。

这个时点的特征：

  • 每个块级已通过测试
  • RTL 已组装足够运行基本的早期测试
  • “Hello World” 测试通过——能在仿真器中加载/读写寄存器、看到端口、数据能在 SoC 中流通
  • 如果设计在仿真器中通过这类测试，在原型中也应该至少能通过——这本身就是 prototype 好坏的试金石

关键机制：RTL 原型交付应纳入整体 SoC 验证计划，作为正式里程碑——这是第9章"面向原型验证的设计”（DFP）的一部分。

4.2.3 无法映射到 FPGA 的设计块

两类块不能进 FPGA：

  1. 模拟电路（天然不匹配）
  2. 固定数字 IP（无源码且无 FPGA 网表）

替代方案：

  • IP 供应商的评估板（固定芯片中的 IP）
  • 自建等效功能板
  • 现有/遗留 SoC 芯片挂接到 FPGA 平台

外部模块的五大互通考量：

考量 具体内容
电气连通性 RTL 中提供连接到 FPGA 外部的模块
信令 IO 信令电平兼容性、信号完整性
时钟和时序 提供必要时钟、计算时钟延迟和互连时序
逻辑同步 外部硬件冷启动远快于 FPGA 加载 → 同步问题、复位、脉冲宽度
物理/机械 物理连接方式、供电、线缆管理、反复插拔保护

减小设计规模的三种策略：

  1. 永久移除：有硬件等价物的块 → 物理 SoC 或评估板替代（典型的如处理器 IP）
  2. 临时移除：超量的模块分批原型化（如多个存储接口分时验证）
  3. 规模缩小：存储器缩小到只跑代码子集，多通道设计只保留一个通道验证驱动软件

4.2.4 一颗 FPGA 到底有多大？

不应用 SoC 门数来衡量 FPGA——“一颗 FPGA 有 n 百万门"这种说法往往是误导。

LX760 资源详览（以 SoC 等效门表示）：

资源 数量 等效门数
逻辑 750K LUT + 950K FF 约 400 万门（满载）/ 250 万门（合理利用率）
存储器 720 个 BlockRAM ~26Mb 各配置（ROM/RAM/FIFO）+ 8.3Mb 分布式 RAM
DSP 864 个 DSP48E1 每个等于数百等效门——算术密集型设计可释放大量逻辑
IO 1200 引脚 往往比逻辑和存储更关键
时钟 18 个 MMCM 全局或区域低偏斜时钟——多时钟设计的关键限制
特殊块 以太网 MAC / PCIe / GTX 不能透明映射——需将 SoC 块替换为 FPGA 等价物

最重要的提醒：

  • FPGA 与 SoC 在门数指标上不可比
  • FF/LUT 比率高的流水线设计映射更好，跑得更快
  • IO 引脚是比逻辑容量更常见的瓶颈

4.2.5 SoC 设计到底有多大？（在 FPGA 术语下）

FPGA 高利用率的三宗罪：

  1. 设计本身：FF 丰富的设计映射效率高；FF 稀疏的设计门利用率低
  2. 时钟资源：多时钟设计必须仔细检查时钟资源和区域限制
  3. 互连资源：高互连设计可能局部耗尽布线资源，导致该区域其余逻辑不可用

建议：虽然 90%+ 利用率可能达到，但初始利用率目标应 ≤50%。

4.2.6 FPGA 资源估算方法论

手动估算：特殊资源（BlockRAM、乘法器、高速 IO）手工估算不难且较准确；随机逻辑和 FF 的手工估算误差大。

推荐做法：RTL 可综合时立即跑 Quick Synthesis → 获得精确的资源使用估计。

Total LUTs = 1,919,724
单颗 LX760 LUTs = 758,784
目标利用率 = 50%
所需 FPGA 数 = 1,919,724 / (758,784 × 0.5) ≈ 5 颗

为什么限制利用率？：

  • P&R 质量随利用率升高大幅恶化
  • 运行时间急剧增长（90% 利用率 → 24+ 小时跑全程）
  • 没有余量应对中后期设计增长和调试逻辑
  • 原型的目标是降低风险 → 把 FPGA 塞到满不是低风险做法

利用率经验值：

  • 生产 FPGA 设计 → ≤75%
  • 原型项目 → 60% 甚至 50% 都很合理

4.2.6.1 知名 SoC 模块的 FPGA 消耗量

IP / SoC 功能 FPGA 资源需求
ARM Cortex-A5（小型配置） 2 颗 Virtex-5 LX330 @50%
ARM Cortex-A9 1 颗 Virtex-5 LX330 @80%
ARM Mali-400 w/3 pixel processors 4 颗 Virtex-5 LX330 @50%
ARC AS221（音频） 1 颗 Virtex-5 LX330 @10%
ARC AV 417V（视频） 1 颗 Virtex-5 LX330 @60%

即使 FPGA 持续变大，原型第一定律（SoC > FPGA）仍然成立。

4.2.7 原型能跑多快？

六大影响因素：

因素 影响机制
设计类型 高流水线设计比低流水线更快
内部互联复杂度 高扇出 → 布线延迟长 → 慢；且分区 IO 更多
资源利用率 高利用率 → 拥塞 → 长延迟 → 慢
FPGA 原始性能 理论极限（但通常远在设计-工具限制之下才触达）
跨 FPGA 时序 FPGA 标准 IO 速度远低于内部逻辑速度——往往是系统时钟的主导限制
外部接口 如果外部刺激不能减速，原型必须匹配其速率

引脚复用的性能代价（第8章详细展开）：

4 路 20MHz 信号复用到一根物理线 → 至少需 80MHz 的数据率 + 额外开销 → 每路有效数据率 < 物理上限的 1/4。引脚复用严重限制系统时钟，应在早期可行性阶段评估。

4.3 一个原型中能用多少颗 FPGA？

理论无上限，实际受限于：

  1. FPGA 间互联：分割越多，跨 FPGA 连线通常越长 → 引脚复用可缓解但降低速度
  2. 信号传播延迟：IO pad 的延迟大于芯片内部——跨 FPGA 路径通常是时序关键路径。多板连接经多个连接器/线缆时，接地回流和参考电平可能退化
  3. 时钟分布：跨 FPGA 的时钟偏斜必须最小化——系统变大后物理时钟分配越来越难
  4. 手工分区的复杂度：FPGA 越多，手工分区越不现实
  5. 管理成本的线性增长：需并行处理的多颗 FPGA 要求多个工具 License

建议：早期发现分区和实现问题是关键——使用 Certify 等分区工具可显著简化和加速。

4.4 估算所需资源（工具和硬件）

工作站配置：

  • 几乎全部跑在 Linux 工作站上
  • 每个进程至少 16GB RAM
  • 驱动因素：FPGA 数据库有数百万个实例

并行化：分区后的多颗 FPGA 可以同时跑综合和 P&R→ 总运行时间 = 最慢那颗 FPGA 的时间。

4.5 跑一遍设计要多久？

工具链各阶段耗时比例

任务 传统流程（Flow 1） 物理综合流程（Flow 2）
综合 10% 10%
映射 20% 35%（前后端合并）
布局 30% 35%
布线 35% 15%
位流生成 5% 5%

布局+布线占总时间的 65-70%（传统流程）——这就是为什么利用率对大 FPGA 如此关键。

实际耗时量级：大 FPGA 90% 利用率 → 24+ 小时跑完全流程，其中 3/4 在 P&R。

4.5.1 缩短运行时间的六大策略

策略 方法
加工作站和 License 更多并行处理
降利用率 重新分区到更多 FPGA——宁愿 6 颗 @50% 不要 4 颗 @75%
放宽时序约束 非关键路径放低目标→显著降低 P&R 时间
增量流程 未改动部分不复用——第11章详述
快速通道 关掉部分优化步骤（如 Synplify Fast Synthesis）
放宽时序模型 极端情况下利用原型只在实验室运行的事实

日常节奏：“白天改设计、晚上跑 build 脚本、第二天看结果"是成熟的实践模式。自动脚本应包含错误捕获和通知功能。

4.5.2 分区运行时间

分区工具的运行时间在整个流程中占比小，但自动化和容错性比速度更重要：

如果某个新门出现在 RTL 中，分区器应该能自动假设"源和目标都在某 FPGA 中 → 该门也自动放入该 FPGA”，而不是半夜弹出一个对话框问你要放哪——然后脚本就卡住了。

4.6 工作量预算是多少？

4.6.1 首次实现的工作量

活动 内容
SoC 设计修改 流水线优化、裁剪不需要原型化的部分、SoC 结构→FPGA 结构替换
分区 把设计划分到多颗 FPGA
设计约束生成 引脚布局、时序约束、综合指令
调试功能配置 插入调试插桩（可选但常用）
物理设计实现 综合→映射→P&R→时序分析→位流生成

首次实现耗时基准：

4 FPGA 系统，每颗 50% 利用率，宽松时钟目标 → 10-16 人周。 即 双人团队 5-8 周。

这不包括平台和工具评估培训时间。首次团队还需加上学习新技术、工具和获取平台组件的时间。

4.6.2 后续迭代的工作量

改动规模 耗时
小改动（仅改调试探针/小逻辑改动） 1-4 小时/FPGA（增量模式下仅变更模块重新综合+P&R）
中等改动（设计增长超出 50% 利用率 + 时序更难 + 可能需重新分区） 最多几天

4.6.3 工程人力资源

这不是给实习生做的暑期项目。

成功的原型工程师需要同时具备：

  1. 对 RTL 设计有深刻理解
  2. 熟悉 FPGA 技术
  3. 掌握 EDA 工具
  4. 优秀的 bench 硬件调试能力

“这些技能在 SoC 验证团队中不常见——需要刻意建设和维持这样的内部专业力量。”

大公司的实践：维护一个专职原型验证团队。这些专家的投资回报率很高。

推荐策略：

  • 首次项目：雇佣顾问——既能交付当前项目，又能把团队带上轨道
  • 某些任务可以让 SoC 工程师分担（如识别多周期路径、RTL 时序改进）

4.7 FPGA 平台（预商用平台示例）

以 Synopsys HAPS-64 为例：

  • 4 颗 Virtex-6 LX760 + 1 颗小 FPGA（板级配置 + Supervisor + 通信）

  • 不同互联选项 + 独立子板（硬 IP、存储、IO）
  • 灵活性 = 跨项目复用的前提

个人总结

这一章没有讲任何硬件细节，却是全书对首次实践者最有价值的一章。它把前面三章的理论变成了可执行的项目计划。

对我最大的启发是 “50% 初始利用率"这个经验法则。乍看很浪费——为什么要多买一倍的 FPGA？但算算账：

  • 90% 利用率：24+ 小时跑一遍，且没有余量容纳中后期设计增长和调试插桩
  • 50% 利用率：几小时跑一遍，多个白天迭代一次，且有足够的空间应对项目不确定性

FPGA 本身可能比工程师的时间便宜。 浪费几天等待 P&R、因利用率爆满而重新分区——这些人力成本很快就超过多买几颗 FPGA 的费用。

另一个深刻印象是工程团队的技能配方。RTL 知识 + FPGA 技术 + 工具掌握 + bench 调试能力——这确实是罕见的 T 型人才。难怪作者建议首次项目雇顾问带队。

最后，“Hello World"测试作为 RTL-FPGA 交接信号这个思路极其实用——它提供了一个客观的、可自动化的、不依赖主观判断的准入标准。

实践思考

  • “早上改设计、晚上跑 build"的节奏需要自动脚本支撑——花一周写好可靠的构建脚本 + 错误通知机制，比整个项目期间每天手动启动各个步骤要划算得多
  • 引脚复用必须在早期评估——等到分区做了几轮才发现跨 FPGA 连线不够、必须降速来引脚复用，代价巨大
  • 不要低估外部模块的同步问题——FPGA 加载需要时间，外面芯片早就在跑了。冷启动、热复位、异步复位/同步复位的场景要在设计阶段就想清楚
  • 首次实现的 10-16 人周基准可以作为项目规划的上限参考（不含学习曲线）
  • 利用率 50% 不是浪费——它买的是迭代速度、调试余量和项目抗风险能力
