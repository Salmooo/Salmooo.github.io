第13章: Prototyping + Verification = The Best of Both Worlds

Jun 11, 2026
5 minute read

章节概览

前十二章都在讲如何独立地构建和使用 FPGA 原型。第13章回答一个关键问题：如何将 FPGA 原型与其他验证方法（RTL 仿真器、虚拟平台、仿真加速器）连接起来，构建一个统一的混合验证环境？

这是一个重要的话题，因为没有任何一种验证方法能同时满足"最早可用 + 最高速度 + 最佳可见性"三个需求。本章展示了如何让它们互补：虚拟平台提供最早可用性，FPGA 原型提供速度和 RTL 精度，仿真器提供不可替代的可控性和可见性——而连接它们的桥梁就是协同仿真接口（HDL Bridge）、事务级接口（SCE-MI/TLM-2.0）和物理接口（UMRBus）。

本章以一个完整的 USB OTG 混合验证案例作为教材级示例贯穿始终，并给出了基于 JPEG 图像处理的四种接口技术的量化性能对比（RTL 仿真 700s → 协同仿真 50s → SCE-MI 160ms → UMRBus 70ms = 性能跨越 10,000×）。

核心知识点

  • 三种原型链接方式的性能-自主性谱系（图149）：协同仿真（仿真器控制硬件，最低性能最高可控性）< 事务级验证（消息通道，中间地带）< 在线原型（自由运行，最高性能最低可控性）
  • HDL Bridge 协同仿真：双向周期精确链接→仿真器和硬件 lock-step 运行。不需要任何设计修改——工具自动生成硬件/软件两侧的 wrapper。但性能限于仿真器速度（KHz 级）→ 无法使用 PLL/DCM/PHY
  • SCE-MI 2.0 四函数模型：Import function（硬件→软件）、Export function（软件→硬件）、Send pipe（硬件→软件流式数据）、Receive pipe（软件→硬件流式数据）——仅四个函数即可实现完整的事务级双向通信
  • TLM-2.0 两种编码风格：Loosely Timed（LT——仅内存读写，开发快）→ Approximately Timed（AT——带时序注释，可估算软件性能和架构分析）→ Cycle-Accurate Transactor（信号级精确，连接 RTL）
  • VMM HAL：允许相同的约束随机测试平台在 “纯仿真” 和 “硬件加速” 两种模式间仅通过命令行开关切换——最大化了验证代码的复用
  • JPEG 图像处理性能对比（Table 32）：RTL 仿真 700s → 协同仿真 50s (14×) → SCE-MI 160ms (4,375×) → UMRBus 70ms (10,000×)。SCE-MI 基于 UMRBus 实现——证明了事务抽象的价值
  • 5 种混合验证使用模式：虚拟平台复用已有 RTL → 虚拟 CPU + FPGA 外设 → 虚拟平台作为 FPGA 测试平台 → 虚拟+物理 IO → Virtual ICE 远程访问
  • Amdahl 定律在系统分区中的应用（Table 33）：DUT 占仿真 90% → 硬件加速最多 10×。“如果目标是仿真加速，不断将更多组件（包括测试平台）移到 DUT 中”
  • USB OTG 案例的关键教训：中断频率是性能杀手——USB 在高速模式下每 125µs 产生一个 Start-of-Frame 中断→处理器的中断服务淹没了 SCE-MI 带宽→虚拟平台反而比系统原型更快。分区不当 → 硬件加速可能比纯软件更慢

详细笔记

13.1 系统原型的概念

系统原型 = 硬件（FPGA 系统）+ 软件（虚拟）模型 + 传统 EDA 仿真器的混合体。

两种核心技术：

技术 链接方式 同步方式 典型接口
协同仿真（Co-simulation） 周期精确、信号级 仿真器和硬件 lock-step HDL Bridge
事务级验证（Transaction-based） 抽象消息、非信号级 独立时钟域、消息通道 SCE-MI, TLM-2.0

图149：三种链接方式的性能-自主性谱系

  • 协同仿真：性能最低——但测试平台可以控制硬件
  • 事务级验证：中间地带——通过抽象消息通信
  • 在线原型（自由运行）：性能最高——但完全自主、外部不可控

13.2 所需工作量

“不幸的是，我们不能只是把原型插到仿真器上就期望它工作。”

关键数据：为事务级验证准备好 FPGA 需要额外增加 10-15% 的"time-to-prototype"工作量。

影响因素：

  • 协同仿真：仅需在工具环境中设置几个约束——额外工作极少
  • 事务级验证：取决于是否已有现成的 transactor → 有 = 很少额外工作；需要新建 = 显著投资

13.3 混合验证场景

作者强调这些场景"不是假设——Synopsys 工程师与许多设计团队交谈后确认了他们已经在做或计划在即将到来的项目中做的事情"。

六个核心场景：

  1. 重用已有（遗留/收购）RTL → 将其放入 FPGA + 与虚拟模型结合 → 快速创建完整系统模型
  2. 复杂处理器用已有虚拟模型 + FPGA 建模仿真的外设 → CPU 运行快于 FPGA 等效实现
  3. 从完整的虚拟模型开始 → 随 RTL 逐渐成熟替换虚拟部分 → 同一套测试重复运行
  4. FPGA + 子板实现真实 IO + 虚拟 IO 替代真实 IO → 回归测试绝对重复之前的刺激条件
  5. FPGA 原型放置在远端 → 软件开发者通过桌面访问 → 真实 IO 的好处 + 熟悉的开发环境
  6. 以上部分或全部模式的组合

13.4 验证接口

13.4.1 协同仿真接口

HDL Bridge 是 Synopsys 的协同仿真实现：VCS 仿真器 ↔ FPGA 原型（CHIPit/HAPS）之间的双向周期精确链接。

关键特性：

  • 不需要修改设计——工具自动生成硬件和软件两侧的 wrapper
  • CHIPit 工具基于用户提供的 RTL 自动准备通信基础设施
  • 设置极快：仅需提供顶层设计描述 + 定义时钟和复位极性
  • 允许仿真器监视 FPGA 内 DUT 的所有内部寄存器（图150）

**设计

流程**（图151）：用户定义输入（RTL + 约束 + 测试平台）→ 工具自动生成 wrapper + 通信基础设施 → 编译 → 运行

**协同

仿真的根本局限**：

“仿真器只能跑几 KHz → 硬件在 lock-step 模式 → 无法使用需要最低频率的组件：PLL、DCM、IP PHY。”

调试与性能的跷跷板：HDL Bridge 允许监视 FPGA 内部寄存器——但监视的信号越多，接口性能越低。

“提前规划时钟结构以支持低频时钟——这会让协同仿真的设置更简单。”

13.4.2 事务级验证接口

核心差异：协同仿真 = 仿真器控制硬件；事务级验证 = DUT（硬件）与仿真器或主机软件通过传递消息/事务通信。

优势：比协同仿真更快 + 更容易调试。设计团队可以在关注实现之前先专注于系统和行为——因为他们在处理器上运行软件来定义验证场景。

13.4.3 TLM 与 Transactor

SystemC TLM-2.0 两种编码风格：

风格 特点 开发速度 适用场景
Loosely Timed (LT) 仅处理内存位置的读写——无时序 快 高层次系统探索
Approximately Timed (AT) 带时序注释 稍慢 软件性能估算、架构分析

从高层到硬件的精度桥接：需要一个能将事务 → 信号、信号 → 事务双向转换的 Transactor。

Transactor 的位置选择：

  • 软件（主机）：开发快但性能受限于主机-硬件通信
  • 硬件（FPGA 内）：将计算密集部分在硬件中处理——编为可综合状态机 / 总线功能模型（BFM）。接收消息 → 转换为连接到设计端口或内部总线的信号

可综合 Transactor 的开发成本：对复杂接口（如 AXI）可能极其耗时 → 复用极其可取。

13.4.4 SCE-MI 标准

Accellera 2007 年批准的 SCE-MI 2.0：为事务级验证提供多通道消息传递环境。

SCE-MI 1.1 的消息通道模型：

  • 软件侧：消息端口代理（C++ 对象，API 访问通道）
  • 硬件侧：消息端口宏（在 transactor 内部实例化）
  • 单向：输入通道（软件→硬件）或输出通道（硬件→软件）
  • 不是硬件总线——类似于网络套接字 + 消息传递协议

Transactor 的双重职责*：

  • 分解（Decompose）：输入通道消息 → 周期精确事件序列 → 时钟进入 DUT
  • 重组（Recompose）：DUT 输出的事件序列 → 消息 → 通过输出通道发送到软件侧

SCE-MI 2.0 的简化*：采用 SystemVerilog DPI 函数调用模型：

方向 数据类型 硬件调用（H→S） 软件调用（S→H）
数据传输 控制信号/小数据量 Import function Export function
流式传输 大量数据 Send pipe Receive pipe

“如果数据量巨大 → 必须使用 pipe。如果只有控制信号或小量数据 → 使用 function。如果对大量数据使用 function→握手开销可能超过数据包本身。”

SCE-MI 的时钟控制：交付给 SCE-MI transactor 的时钟由基础设施自动控制——用户无需手动停止和启动时钟。数据在 SCE-MI 时钟域（独立于 DUT 时钟域）上传输 → 传输期间 DUT 时钟被暂停 → 传输完成后自动释放。

13.4.5 SCE-MI 2.0 实现示例

CHIPit/HAPS 的实现架构（图153）：UMRBus → SCE-MI 基础设施 → Transactor（用户定义）→ DUT。CHIPit 工具同时创建硬件基础设施和软件 C/C++ 文件。

**与

SystemVerilog 测试平台的桥接**（图154）：仿真器中的 SV Testbench → DPI → C 环境 → SCE-MI 2.0 DPI → FPGA 硬件。同一个 C 代码同时通过标准 DPI 与仿真器通信 + 通过 SCE-MI DPI 函数与硬件通信。

13.4.6 VMM HAL

VMM HAL 的核心价值：相同的约束随机测试平台——一条命令行开关 = “纯仿真"模式还是"硬件加速"模式。

工作原理：将 VMM 环境中的交易层驱动和监视器 → 替换为 SCE-MI 2.0 transactor → DUT = 包含 SoC 设计的 FPGA 原型。

关键收益：测试平台和测试用例可以完全在纯仿真中开发和调试——无需硬件。开发完成后 → 硬件加速模式仅需命令行切换。

13.4.7 物理层接口

UMRBus 的独特定位：专为实现主机与 FPGA 原型之间高性能、低延迟通信而设计的物理层。不同于通用总线（如 PCIe）——后者可能因为共享非协同仿真数据流量而性能较低。

13.5 接口技术性能对比

基于 JPEG 图像处理的量化性能（Table 32）：

接口 是否需要改 RTL Wrapper 生成 时钟模式 信号可见性 JPEG 运行时间 相对倍速
RTL 仿真 No No 主机控制 全部 700s 1×
协同仿真（HDL Bridge） No Yes 主机控制 寄存器+端口 50s 14×
SCE-MI Yes Yes（工具） SCE-MI 控制 全部 160ms 4,375×
UMRBus Yes No 自由运行 全部 70ms 10,000×

关键结论：

  • 协同仿真比纯 RTL 仿真快 14×——但仍是秒级，因为 lock-step 于仿真器速度
  • SCE-MI 比协同仿真快约 300×——这来自事务抽象的威力
  • UMRBus 最快——因为是专用通道且自由运行
  • “性能取决于设计——不能一概而论。取决于多少计算在硬件上、多少在仿真器中、有多少数据在两侧之间传输、使用什么传输机制。”

13.6 使用模式详细展开

13.6.1 虚拟平台复用已有 RTL

场景（图156）：新架构需要尽早开始软件开发——但新 RTL 尚不可得。

**方案

**：虚拟平台中最新的 ARM 核心 + 已有 RTL（如视频编解码器）实例化到 FPGA。混合系统比纯虚拟更快获得周期精度，比等待全新 RTL 快得多。

投资回报：新 RTL 从零到原型 = 2-3 个月；全新虚拟平台模型时间更长但比等待新 RTL 快 → 混合模型 = 两者优势结合。

13.7 为软件的虚拟平台（图157）

**问题

**：复杂处理器（如 ARM Cortex-A8）在 FPGA 中实例化困难——速度远慢于专用 ISS。

方案：将处理器的 ISS 留在虚拟平台中 → 大部分事务在软件中处理 → 仅访问外设时走 SCE-MI 到 FPGA。关键细节：本地存储器（L3）也必须留在虚拟平台——否则从 FPGA 侧通过 SCE-MI 取指令和数据会抹杀所有性能收益。

“在工作站上用虚拟模型处理器执行嵌入式软件比在 FPGA 原型中更快。FPGA 用来维持外设和加速器的精度。”

13.7.1 虚拟平台作为测试平台（图158）

**时序

**：项目前期用虚拟平台启动了预 RTL 软件开发 + 编写了测试用例。

当 RTL 可用时：将 TLM 替换为 FPGA 中的 RTL → 同一套软件验证测试重新运行。

多重价值：

  • 复用系统级开发阶段的投入
  • 对比纯软件仿真结果 vs 硬件中的结果 → 验证硬件流程
  • 通过重复一组已知刺激验证流程
  • 为验证计划提供反馈
  • 高效运行回归测试

13.7.2 虚拟与物理 IO（图159）

**哲学

**：虚拟 IO 达不到真实物理接口的验证完整度——我们需要在系统上下文中验证设计，将其连接到真实世界。

手机摄像头的例子：将基带设计连接到工作站上的摄像头 → 软件相信信息来自手机中的真实摄像头 → 允许在适当时使用真实世界刺激。

GUI 场景：“手机在打游戏和下载邮件的同时能接电话吗？"——这类涉及并发真实场景的验证通过 GUI 交互来覆盖。

13.7.3 Virtual ICE（图160）

**解决

的问题**：软件开发者不想要/不需要看到桌上的板子——但确实需要访问 DUT 的功能来写代码。

架构：FPGA 模型中包含 LCD 和 UART 接口 → 通过 SCE-MI 将这些接口重定向回 PC 上的虚拟平台 → LCD 和 UART 以虚拟形式呈现在开发者桌面上。

核心优势：软件工程师留在他们熟悉的开发环境中（键盘+屏幕）——而不是实验室中。大型或脆弱的原型可以安全地留在实验室机柜中。

代价：网络传输的仿真性能比直接的实验室高速连接差。

13.8 系统分区——Amdahl 定律的应用

固定约束：

  • 测试平台的不可综合部分不能放硬件
  • 设计大小可能溢出目标 FPGA
  • 高利用率 → 更长的实现时间

灵活选择：

  • 只能在已有 transactor 的地方做切割（或可以轻松获得 transactor 的地方）
  • 倾向于在明确定义的行业标准接口处切割（如 AHB 总线接口），而非在任意点——后者需要自己建模
  • 计算必须主导通信——如果软件/硬件接口上的通信量过大，硬件加速的优势被抹杀

Amdahl 定律的验证加速预测（Table 33）：

场景 测试平台占% DUT 占% 最大加速因子
测试平台密集型 90% 10% 10%
DUT 密集型 10% 90% 900%

“如果目标是仿真加速，考虑计算在哪里发生。不断将更多组件移入 DUT——如果可能，将测试平台也综合到 DUT 中。”

迭代式分区策略：随着 RTL 成熟逐步将更多组件移入 DUT——不将未经测试的代码放入 FPGA 有助于风险控制。

“如果 DUT 使用多颗 FPGA，将仿真器接口仅分配给其中一颗 FPGA——这会提升性能。”

13.9 案例研究：USB OTG（全书最完整的混合验证案例）

系统架构（图161）：

  • 虚拟平台侧：Innovator → Samsung SoC 模型（LCD、触摸屏、DMA、物理以太网）→ 运行未经修改的 Linux 映像
  • FPGA 原型侧：USB 2.0 OTG 核心 RTL
  • 桥梁：AHB 总线 transactor → SCE-MI 2.0
  • 物理接口：子卡 → USB 存储棒

Table 34 的完整性能数据：

操作 RTL 仿真 RTL+虚拟平台 虚拟平台 系统原型 纯硬件
精度 周期精确 取决于分区 事务级功能精确 取决于分区 周期精确
Linux boot - 86s 29s 43s 12s
Modprobe - 702s 23s (31×) 61s (12×) 10s (70×)
Mount - - 15s 22s <1s
Copy - - 4s 3s <1s
Umount - - <1s 9s <1s
缩放性能 0.03125× 1× 225× ~30× 625×

最反直觉的结果：纯虚拟平台在多个测试中比系统原型（FPGA 硬件加速）更快！

原因分析：USB 在高速模式下每 125µs 产生一个 Start-of-Frame 中断——处理器需要服务每一个中断 → SCE-MI 带宽被大量中断淹没 → 硬件加速的优势被大量跨接口通信消耗殆尽。

“USB 不是演示硬件加速性能的最佳例子——因为核心产生中断的数量以及处理器需要服务的中断数量。经过大量工作优化设计以阻止中断淹没带宽后，性能才有所提升。分区正确的话——视频编解码器之类的东西在系统原型上的性能会比虚拟平台好得多。”

这个失败案例的价值：它为 13.8 节的 Amdahl 定律提供了最生动的反证——当计算收益（硬件中运行 RTL）被通信代价（处理大量中断的跨接口往返）超越时，硬件加速的结果可能比纯软件更差。

关键图示解读

图149：三种原型链接方式的性能-自主性谱系

  • 横轴 = 性能，纵轴 = 自主性——两个维度相互排斥
  • 在线原型自由运行最大性能 → 耦合仿真受仿真器控制最小性能 → 事务级验证居中间

个人总结

第13章可能是全书中最被低估的一章。它回答了一个在项目规划会上经常被问但很难回答的问题：“我们已经有了虚拟平台（或仿真器环境），为什么还需要 FPGA 原型？”——以及反过来：“我们已经有了 FPGA 原型，为什么还需要虚拟平台？”

本章的回答是：两者不是竞争关系——它们是同一个统一验证生态中互补的节点。虚拟平台给最早期的软件开发和测试平台验证提供了速度和可用时机；FPGA 原型给 RTL 成熟后的高精度、高速验证提供了平台；而 SCE-MI 和协同仿真接口将它们桥接起来——让团队可以在同一个验证计划中，根据各阶段的精度/速度需求灵活调配。

但本章最诚实的部分不是成功案例——而是 USB OTG 的"失败"案例。纯虚拟平台在很多测试中比花了大量努力构建的系统原型更快。这个反直觉的事实揭示了一个残酷的工程真相：如果分区的接口上通信量大于计算加速收益，硬件加速投入可能得不偿失。不是所有设计都适合混合验证——正确的设计（如视频编解码器——计算密集型、接口简单）可以获得巨大收益；不正确的设计（如 USB——中断密集、接口复杂）可能比纯软件更慢。

Amdahl 定律在验证中的含义不仅是"预测加速上限”——更重要的是它告诉我们应该把测试平台本身也综合进 FPGA。这是很多团队容易忽视的一步——他们自然地认为测试平台应该留在仿真器侧，但这样做就把仿真器变成了 Amdahl 定律中的串行瓶颈。

实践思考

  • “易于获得 transactor 的地方” 是决定系统分区的实际约束——在做架构选型时就确认哪些接口有现成的 transactor（AHB 总线的 transactor 容易获得；自定义接口的 transactor 需要自己开发——投入数周甚至数月）
  • 中断频率是事务级验证的潜在性能杀手——在设计规划阶段就评估 DUT 的中断频率 × 每次中断的数据传输量 × SCE-MI 通道带宽。如果中断开销超过计算加速，考虑另一种分区方案
  • VMM HAL 的"命令行切换"是最低风险的混合验证引入方式——先用纯仿真环境开发和调试所有测试用例 → 待一切稳定后再切换到硬件加速模式 → 无需任何代码变更
  • 协同仿真要求 PLL/DCM 在低频下工作——这可能与第8章讨论的最低锁定频率限制冲突。在设计阶段就为协同仿真预留一个低频直通时钟路径
  • SCE-MI 2.0 的 pipe vs function 选择不是性能问题——是功能问题。大型数据用 function 会因为握手开销而退化→ 在设计 transactor 时就必须决定正确的数据传输机制
  • “将测试平台本身综合进 DUT"是 Amdahl 定律在验证中最激进的推论——但这需要从项目开始就以"可综合测试平台"为前提设计验证环境
