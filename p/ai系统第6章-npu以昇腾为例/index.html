AI系统：第6章 NPU——以昇腾为例

《AI系统：原理与架构》学习笔记 — 深入华为昇腾NPU，从达芬奇架构到AI Core核心单元，从数据布局到编译器优化

4 minute read

本章概览

前面几章我们聊了 CPU、GPU、TPU 这些"舶来"的 AI 芯片，这一章终于轮到中国力量登场了——华为昇腾（Ascend）。

如果说 GPU 的思路是"把通用计算做到极致再拿去跑 AI"，TPU 的思路是"只做 AI 计算，别的啥也不管"，那昇腾走的是第三条路：全栈自研 + 开放生态。从底层的达芬奇架构、到中间的 CANN 异构计算框架、再到上层的 MindSpore 和 MindX，昇腾试图打通从芯片到应用的完整链路。

这一章是整个 AI 硬件篇的"终局之战"——你会看到不同于英伟达 CUDA Core 和谷歌脉动阵列的第三种 AI 计算范式：

  • 达芬奇架构：同时拥有矩阵、向量、标量三种计算单元，既有 ASIC 的效率，又有通用芯片的灵活性
  • AI Core 设计哲学：存储转换单元（MTU）硬件化 Im2Col、“多进单出"的数据通路、“大Z小z"的分形排布
  • 数据布局转换：从 NCHW/NHWC 到 NC1HWC0 五维格式，再到 NZ 分形格式——这一切都是为了喂饱那个 16×16 的 Cube

一句话总结：昇腾的核心竞争力不在于单卡算力多大，而在于"端边云全场景"的统一架构——同样的代码可以跑在毫瓦级的嵌入式设备上，也可以跑在几百瓦的数据中心集群里。

6.1 昇腾 AI 系统概述

6.1.1 昇腾 AI 产业生态

华为的昇腾生态包含了从底层硬件到上层应用的全栈体系：

层级 内容
硬件系统 基于达芬奇内核的昇腾处理器、模组、板卡、小站、服务器、集群
基础软件 CANN 异构计算架构、驱动、运行时、加速库、编译器
AI 框架 MindSpore（开源），同时支持 PyTorch、TensorFlow、PaddlePaddle
应用使能 MindX（含 MindX DL、MindX Edge）、ModelArts、HiAI

昇腾 AI 产品覆盖了边缘侧、云侧推理、云侧训练三大场景：

昇腾处理器按 PCIe 工作模式分为 RC（Root Complex）模式和 EP（Endpoint）模式：

PCIe 模式 说明 典型产品
RC 模式 CPU 直接运行 AI 业务软件，接入外挂设备 Atlas 200 AI 加速模块、Atlas 200 DK
EP 模式 Host 侧运行 AI 业务，Device 侧通过 PCIe 接入 Atlas 300 系列推理卡、训练卡

EP 模式下，Host 系统和 Device 系统通过 PCIe 通道交互，将 AI 任务加载到 Device 侧的昇腾处理器中运行。这种"主从架构"在 AI 加速卡中非常普遍。

6.1.2 昇腾 AI 系统架构

昇腾 AI 全栈架构的层次与本书介绍的整体 AI 系统架构高度吻合：

昇腾架构的设计理念非常清晰：

  • 模块化：支持端边云场景下的独立部署
  • 协同性：模块之间具备相互协同能力
  • 独立演进：各层之间支持独立升级，互不绑定

基于这套架构，昇腾提供了四大核心优势：

1. 超强算力

场景 产品 算力
数据中心训练 Atlas 训练卡 320 TFLOPS FP16
数据中心集群 Atlas 集群 256 ~ 1024 PFLOPS FP16
边缘推理 Atlas 推理卡 88 TOPS INT8

更惊艳的是能效比——Atlas 集群采用全液冷设计，PUE < 1.1，E 级算力集群 5 年可节省电费超 1 亿元。

2. 全栈开放

按能力分层开放，面向不同开发者提供不同工具：

  • 业务应用开发者：MindX SDK，极少代码即可实现 AI 功能
  • AI 模型开发者：MindSpore + 主流框架适配 + 模型转换工具
  • 算子开发者：CANN + MindStudio + Ascend C 统一编程接口

3. 使能应用：MindX DL / MindX Edge 将设备资源和算力统一抽象，让应用开发者无需关心底层硬件细节。

4. 端边云协同：硬件层面统一采用达芬奇架构，CANN 使能各类形态硬件；软件层面一次开发、全场景部署；端侧增量数据回传云侧训练优化，优化模型直接下发端侧。

6.1.3 昇腾未来展望

昇腾的未来方向可以归纳为几个关键词：

  • 多样化异构算力：由应用驱动的计算架构，SoC 成为主要形态
  • 智能化编译系统：用 AI 来开发 AI，基础软件的智能化程度将极大提升
  • 生态持续丰富：更多算法、加速库、SDK、开发者、合作伙伴
  • 安全可信：将安全和可信作为基本功，确保 AI 可靠地服务于人类

6.2 昇腾 AI 处理器

6.2.1 昇腾 AI 处理器概述

2018 年，华为推出了基于达芬奇架构的昇腾 AI 处理器。这不是一个"追赶式"的设计，而是从零开始，专为 AI 计算量身打造的架构。

达芬奇架构的核心武器是 3D Cube 矩阵计算单元——一条指令即可完成 4096 次乘加运算：

一条指令 = 16×16×16 = 4096 次乘加。GPU 需要用 256 个线程、16 个时钟周期才能完成的事，昇腾一条指令就搞定了。

达芬奇架构的几个关键设计特点：

  • CISC 指令集：高度灵活的专用指令，能应对日新月异的新算法和新模型
  • 多维计算模式：标量、矢量、矩阵三种计算模式，打破了 ASIC 的局限性
  • 混合精度支持：推理用 INT8/INT4，训练用 FP16/FP32
  • 全场景适配：从几十毫瓦到几百瓦的芯片，统一架构、一次开发多场景部署

6.2.2 AI 处理器架构

昇腾 AI 处理器本质上是一个 SoC（片上系统），主要包含：

  • AI 计算引擎：AI Core（达芬奇架构）+ AI CPU
  • 多层级缓存/缓冲区：L1/L2/L3 Cache + 片上 Buffer
  • DVPP（数字视觉预处理模块）：视频编解码、JPEG/PNG 编解码、图像预处理
  • 任务调度器（TS）：专门服务于 AI Core 和 AI CPU 的调度
  • 总线接口：PCIe 接口与 Host 系统交互

早期的昇腾处理器逻辑架构：

当前基于达芬奇架构的昇腾 AI 处理器架构更加成熟：

所有模块通过基于 CHI 协议的片上环形总线相连，确保数据共享和一致性。处理器集成了多个 ARM CPU 核心，按功能分为主控 CPU（控制芯片整体运行）和 AI CPU（承担非矩阵类复杂计算），两者的核数可由软件动态分配。

昇腾 910 —— 云侧训练利器

昇腾 910 采用了先进的 Chiplet（芯粒）技术，包含 6 个 Die：

  • 1 个计算芯粒：32 个 DaVinci Core + 16 个 CPU Core + 4 个 DVPP
  • 1 个 IO 芯粒
  • 4 个 HBM 芯粒（总计 1.2 TB/s 带宽）

云侧训练场景的针对性优化：

  • 最高规格 Ascend-Max：每个 Core 每周期完成 4096 次 FP16 乘累加
  • HBM + DDR：高 Load/Store 带宽应对反向 SGD 的大量缓存访问
  • 100G NIC（RoCE v2）：支持万卡集群互联
  • 高吞吐率 DVPP：最高支持 128 路 1080P 视频解码

昇腾 310 —— 边缘推理专家

相比 910，昇腾 310 的定制化 IP 更少，但提供了更多外设接口。由于边缘推理场景 Batch Size 通常只有 1，昇腾 310 选择了**较小的矩阵计算维度（4×16×16）**以快速实现矩阵乘——不追求峰值吞吐，而是追求低延迟。

6.2.3 计算加速原理

在神经网络中，卷积计算的计算量往往是决定性的。昇腾 AI 处理器从软硬件架构上都对卷积计算进行了深度优化。

卷积 → 矩阵乘法的转换

卷积计算在底层被拆解为几步：

  1. Im2Col：将输入特征矩阵 X 和权重矩阵 W 展开为 X_I2C 和 W_I2C
  2. GEMM：矩阵相乘得到中间输出 Y₀
  3. 加偏置：累加偏置项 b，得到最终输出 Y

AI Core 卷积计算数据通路

整个数据流的核心路径：

  1. 数据搬运：矩阵 X 和 W 从 DDR/HBM → 输入缓冲区 → 存储转换单元（MTU）
  2. 格式转换：MTU 对 X 和 W 进行补零和 Im2Col 重组，得到 X_I2C 和 W_I2C
  3. 矩阵计算：X_I2C 和 W_I2C 送入矩阵计算单元，分块为 16×16 矩阵进行乘法
  4. 偏置累加：累加器将偏置项与中间结果 Y₀ 累加得到 Y
  5. 激活/池化：向量计算单元对 Y 进行激活和池化处理

关键优化：如果权重 W_I2C 需要重复使用，可以将其固定在输入缓冲区中反复使用，避免重复从 DDR/HBM 读取——这就是"数据复用"在硬件层面的体现。

6.3 昇腾 AI 核心单元

6.3.1 达芬奇架构

达芬奇架构的本质是特定域架构（DSA）——既不像 CPU/GPU 那样追求通用，也不像 ASIC 那样只做一件事。它专门针对深度学习领域的常见计算模式进行了优化。

AI Core 包含三种基础计算资源，对应三种常见的计算模式：

计算单元 对应模式 支持的精度 功能定位
矩阵计算单元（Cube Unit） 张量计算 INT8、INT4、FP16 “主力军”，负责矩阵乘法
向量计算单元（Vector Unit） 向量计算 FP16、FP32 “辅助军”，负责激活、池化等
标量计算单元（Scalar Unit） 标量计算 - “指挥部”，负责控制和地址计算

这三种计算单元形成了三条独立的执行流水线，在系统软件的统一调度下互相配合。

特别值得注意的是存储转换单元（MTU）——这是达芬奇架构的一大特色。在 GPU 中，Im2Col 这样的格式转换是通过软件实现的，效率低下。达芬奇架构直接把这一步固化在硬件电路中，能够在极短时间内完成整个转置过程，实现不间断的卷积计算。

6.3.2 计算单元

矩阵计算单元 —— 一指令定乾坤

矩阵计算单元是 AI Core 的算力核心。它能用一条指令完成两个 16×16 矩阵的相乘（这也是"Cube"名称的来历——16³ = 4096）：

矩阵计算单元内部有 256 个矩阵计算子电路，可以并行完成结果矩阵的 256 个元素计算。处理大型矩阵时，采用**分块平铺（Tiling）**策略——将大矩阵切成多个 16×16 子矩阵，分批计算、累加结果。

“大Z小z"数据排列

为了匹配矩阵计算单元的工作方式，数据需要按照特定的分形格式排列：

  • “大Z小z”（图左）：分块之间按行排序，块内按行排序 → 常用于特征图
  • “大Z小n”（图中）：分块之间按行排序，块内按列排序 → 常用于权重
  • “大N小z”（图右）：分块之间按列排序，块内按行排序 → 常用于卷积输出

分步计算大型矩阵

计算 C₁ 结果子矩阵需要两步：先用 A₁×B₁ 算中间结果，再用 A₂×B₂ 计算并累加——这种"分步累加"的策略充分利用了片上缓存的容量和数据局部性。

向量计算单元

向量单元是矩阵计算单元和输出缓冲区之间的"桥梁”。矩阵运算完成后，向量单元可以顺便完成 ReLU 激活、池化等操作，支持连续寻址和固定间隔寻址等多种模式。

6.3.3 存储系统

达芬奇架构通过精心设计的分布式缓冲区保证数据的及时供应，消除数据流传输瓶颈。

存储系统的关键组件：

组件 功能
总线接口单元（BIU） AI Core 与外部交互的"大门”，负责读写请求转换
存储转换单元（MTU） 数据格式转换（补零、Im2Col、转置、解压缩）
输入缓冲区 暂存需要频繁重复使用的数据，减少外部读取
输出缓冲区 暂存中间结果和输出数据，必须经过它才能写回核外
寄存器 矩阵计算单元的 16×16 输入/结果寄存器

数据通路——多进单出

达芬奇架构的数据通路设计非常独特——多进单出：

  • 数据流入：可以通过多条通路，从外部直接流入矩阵计算单元、输入缓冲区和输出缓冲区中的任何一个
  • 数据流出：必须通过输出缓冲区，最终才能写回核外存储系统

这样设计的逻辑是：神经网络的输入数据种类繁多（多通道特征图、多个卷积核的权重和偏置），但输出只生成特征矩阵——用并行输入提高供给效率，用统一输出简化控制。

与谷歌 TPU 的统一缓冲区理念相似，AI Core 也采用了大容量片上缓冲区设计，通过增大片上缓存来减少片外访问频次，从而降低数据搬运功耗。

6.3.4 控制单元

控制单元相当于 AI Core 的"司令部”，负责整个计算过程的指令调度。

指令执行的核心流程：

指令缓存 → 标量指令处理队列（解码） → 指令发射模块 → 分发到对应队列
                                                      ├── 矩阵运算队列 → 矩阵计算单元
                                                      ├── 向量运算队列 → 向量计算单元
                                                      └── 存储转换队列 → 存储转换单元

关键设计要点：

  • 多流水并行执行：矩阵、向量、存储转换三条流水线可以同时运行，提升整体执行效率
  • 事件同步模块：通过软件插入同步符来控制流水线之间的依赖关系——当矩阵运算需要等待向量计算的结果时，事件同步模块会暂停矩阵流水线直到数据就绪
  • 系统控制模块：负责任务块的配置、启动、状态报告和中断处理

在达芬奇架构中，无论是流水线内部还是流水线之间的同步，都是通过事件同步模块 + 软件控制来实现的——这给了程序员极大的灵活性，同时也对编程能力提出了更高要求。

6.4 昇腾数据布局转换

6.4.1 昇腾数据排布

数据排布格式的选择直接影响矩阵乘法和访存的效率。昇腾既不选择 NHWC，也不选择 NCHW，而是采用了一种自有的五维数据格式。

NC1HWC0 格式

昇腾将特征图的 C 维度进行分割，统一采用 NC1HWC0 的五维格式存储：

  • 将 C 维度分割为 C₁ 份 C₀（C₁ = C / C₀，向上取整）
  • 对于 FP16 类型，C₀ = 16；对于 INT8 类型，C₀ = 32
  • 这恰好对齐了矩阵计算单元"一条指令处理 16×16 矩阵"的能力

转换代码一目了然：

3
4

# NHWC → NC1HWC0
Tensor.reshape([N, H, W, C1, C0]).transpose([0, 3, 1, 2, 4])

# NCHW → NC1HWC0
Tensor.reshape([N, C1, C0, H, W]).transpose([0, 1, 3, 4, 2])

Fractal Z 与 NZ 格式

为了更高效地搬运和进行矩阵计算，昇腾引入了分形数据格式：

NZ 格式的核心规律：

  • 分形之间按列主序排布（大 N）
  • 分形内部按行主序排布（小 z）
  • 存储优先级：W₀ 方向 → H₀ 方向 → H₁ 方向 → W₁ 方向 → N 方向

将 ND 格式转换为 NZ 格式的过程：

1

(N, H, W) → pad → (N, H₁×H₀, W₁×W₀) → reshape → (N, H₁, H₀, W₁, W₀) → transpose → (N, W₁, H₁, H₀, W₀)

6.4.2 AI 编译器布局转换算法

数据布局转换的目的是将内部数据布局转化为后端硬件友好的形式。但这需要权衡——布局转换本身也有开销。

AI 编译器通过插入 CASTDATA 算子来实现数据格式转换。但编译器还要做优化——消除多余的转换：

三种典型场景：

  • 场景一（左）：OP1 和 OP2 都使用 NCHW，无需插入转换
  • 场景二（中）：OP1 需要 NHWC 但输入是 NCHW，插入两次转换（输入 NCHW→NHWC、输出 NHWC→NCHW）
  • 场景三（右）：OP1 用 NHWC、OP2 用 NCHW，中间需要一次转换

如果三个连续的 1×1 卷积算子都是 NHWC 格式，编译器会自动消除它们之间多余的转换算子——这就是"上下文感知"的优化能力。

训练场景与推理场景的一个关键区别是：推理时需要额外处理权重数据布局的转换，因为训练通常用 GPU（偏 NCHW），而推理可能在手机 CPU 上执行。

小结与思考

这一章从昇腾的产业生态讲到 AI Core 的微观架构，信息密度非常大。让我用几个问题来收尾：

  • 达芬奇架构和 CUDA Core / TPU 脉动阵列的本质区别是什么？ 达芬奇有三条独立流水线（矩阵+向量+标量），既有 TPU 的专用效率（硬件化 Im2Col），又有 GPU 的灵活性（可编程标量单元控制）。这是一种"DSA + 可编程"的折衷设计。

  • 存储转换单元（MTU）为什么如此重要？ 在 GPU 中，Im2Col 用软件做、效率低；在 TPU 中，数据靠脉动阵列在 PE 之间"流动"。达芬奇用一个硬件模块来专门完成数据格式转换——这是对"频繁操作硬件化"这一理念的极致贯彻。

  • “多进单出"数据通路的设计智慧：神经网络输入"多”（多通道特征图、多权重矩阵），输出"一"（最终特征图）。用多条路径同时拉数据进来，用一条路径统一管输出——以最小的硬件成本适配算法特征。

  • NC1HWC0 五维格式的巧妙之处：C₀ 取 16（FP16）正好匹配矩阵计算单元的 16×16——这不是巧合，是软硬件协同设计的"对齐"。从芯片设计到数据格式到编译器，昇腾的全栈自研优势在这里体现得淋漓尽致。

如果说英伟达的护城河是 CUDA 生态 + Tensor Core，谷歌的护城河是脉动阵列 + TPU Pod，那华为昇腾的护城河就是全栈自研 + 端边云统一架构 + 中国市场深度绑定。从达芬奇架构的硬件设计，到 CANN 的软件生态，再到 MindSpore 的框架层——昇腾正在构建一个完整的、不受制于人的 AI 计算体系。

下一篇预告：AI 硬件的旅程到这里暂告一段落。从 CPU → GPU → TPU → NPU，我们看到了四种截然不同的设计哲学。接下来，让我们进入 AI 编译与计算架构的世界，看看编译器是如何在"上层算法"和"底层硬件"之间架起桥梁的。
