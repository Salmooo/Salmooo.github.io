第8章: Partitioning and Reconnecting

Jun 05, 2026
8 minute read

章节概览

第8章解决"原型第一定律"——SoC 比 FPGA 大，必须分到多颗 FPGA 中。它不仅讨论"怎么分"，还讨论"分完后怎么重新连起来并保持同步"。

章节结构是一个从粗到细的完整工作流：Board Description → 面积估算 → 分区步骤（10 步手动流程） → 走线分配 → 自动化分区 → 性能优化（时序预算 + IO FF 利用） → 跨 FPGA 三同步（时钟/复位/启动） → TDM 信号复用（三种方案对比 + 完整量化时序计算） → HSTDM（LVDS + IOSERDES）。

本章最具工程价值的内容是分段式计算那个 4:1 同步 MUX 方案最终的 14.28MHz（而非直观的 25MHz）设计频率——这个详细的逐周期数据传输分析（Table 20）展示了 TDM 性能估算不能靠直觉，而必须逐级拉时序表格。

核心知识点

  • 分区前必须准备 Board Description File（Verilog 格式）：列出 FPGA 间所有走线、复位和时钟走线、FPGA 例化与引脚映射、不影响信号连接的部分（电源轨、上拉电阻、去耦电容）不需要描述
  • 手动分区十步法（Table 17）：描述板资源 → 估算面积 → 分配顶层 IO → 分配高互连块 → 分配最大块 → 分配剩余块 → 复制共享资源 → 复用超标互联 → 分配走线 → 迭代改进
  • 影响分析（Impact Analysis）：在分配前预览"如果选 uA → 面积 +672 LE，IO +137 / 如果选 uB → 面积 +672 LE，IO -83"——基于数据的决策而非猜测
  • 复制（Replication）是 IO 优化最有效的技巧：一个块需要同时驱动两个不同 FPGA 中的逻辑 → 在每个 FPGA 中各复制一份 → 省掉所有跨 FPGA 的连线
  • 走线分配三优先原则：(1) 完全匹配端点（同一走线经过所有所需 FPGA）→ (2) 匹配端点 + 最少冗余端点 → (3) 切分信号到两条走线（源 FPGA 多用一根输出引脚）
  • 时钟同步四步法：板级 PLL 生成基时钟 → 每 FPGA 内 MMCM 重新同步并分发 → 基时钟作为反馈时钟输入保持 MMCM 输出相位一致 → 只在所有 PLL/MMCM 锁定后才释放全局复位
  • 三 FF 流水线级实现跨 FPGA 复位分发：第一和第三 FF 可放入 IO FF → 中间 FF 在内部 → 整个时钟周期供复位信号传播——极大放宽 P&R 约束
  • TDM 复用比 ≠ 设计频率反比：4:1 同步 MUX 实际设计频率仅 14.28MHz（100MHz/7），而非 25MHz（100MHz/4）——额外 3 个传输周期用于跨域时序开销
  • 三种复用方案对比：MUX 方案（最简单、性能最低、IO 约束复杂）→ Shift-Register（灵活、中等性能、适合长飞行时间板）→ IOSERDES/LVDS（性能最高、128:2 复用比、仅限差分 IO 板）
  • LVDS + IOSERDES = 复用比的质变：单端 8:1 需要 1 个走线 → 差分 8:2 需要 2 个走线 → 但 LVDS 速率可达 1GHz → 128:2 成为可能 → 有效互联带宽远超单端方案

详细笔记

8.1 我们是否总需要跨 FPGA 分区？

不需要的场景：设计足够小、或原型范围被有意限制到单 FPGA。FPGA 容量沿 Moore’s Law 持续增长，但 SoC 也同时变大（多 CPU + 视频图形 + DSP）——第一定律仍将长期成立。

DFP 的预分区（第9章预告）：通过提前规划，使每个主要功能块自然适配单颗大 FPGA，或能在边界清晰信号很少的地方轻松拆分。

不总是需要 EDA 分区工具，但没有工具很难成功。工具三大优势：

  1. 全局实现约束 → 工具透明传播到各 FPGA 的 P&R
  2. 可选插入引脚复用和逻辑复制
  3. 与综合工具紧密集成 → 资源和时序信息互通

8.2 分区总体方法

8.2.1 推荐的交互式手动分区十步法（Table 17）

步骤 任务 核心原因
1 描述目标资源 所有 EDA 分区工具的基本要求
2 估算面积 防止分区时过度使用资源
3 分配顶层 IO 确保与外部资源的连接；引导 IO 电压区域选择
4 分配高互连块 为宽总线最小化跨 FPGA 互联
5 分配最大块 早期反馈资源平衡状况；留小块填补间隙
6 分配剩余块 可控地使用复制和修剪；在较低的层级工作
7 复制关键资源 减少 RTL 修改量；同步各 FPGA 的时钟和复位
8 复用超标互联 释放 FPGA IO 给关键路径
9 分配走线 分配跨 FPGA 信号 = 固定引脚位置
10 迭代改进 初始分区通常可以改善；成熟后可提升目标速度

8.2.2 Board Description File

Certify 使用的是 Verilog 格式的板描述文件（图104），包含：

  • 板级走线（如 wire [44:1] AB; wire [29:1] AC; wire [62:1] AD）
  • FPGA 例化及引脚映射（如 XC6VLX760FF1760 uA (.pin_AR8(A1_A[1]), ...)）
  • 复位和时钟走线

不需要描述的：电源轨、上拉电阻、去耦电容、FPGA 配置引脚。

关键：连接性信息必须绝对正确。最好直接从板布局工具导出 → 脚本过滤。可配置互联（延迟或交换互联，第6章）的板 → 板描述需反映项目的实际配置。

8.2.3 估算各子块面积

Certify 三种估算模式（Table 18）：

Effort 模式 描述
Low Model based 网表操纵器写 .est 文件；最快
Medium Model based 改进寄存器估算；更长运行时间
High（默认） Architecture based FPGA 映射器写估计文件；最慢但最精确

好的副产品：面积估算通常偏高 → 鼓励我们在 FPGA 中留更多空间。

8.2.4 分配顶层 IO

  • 被强制的分配优先做（如哪个 FPGA 连到外部 RAM 子卡、外部 PHY、逻辑分析仪探头）
  • 灵活平台可通过重新排列子卡位置改变哪些 FPGA 引脚连接哪些外部资源
  • 部分团队的策略：让一个 FPGA 承担尽可能多的外部 SoC IO → 虽然该 FPGA 看似被过度约束，但给其余 FPGA 的分配留下了最大自由度
8.2.4.1 IO 电压区域注意
  • SoC 中不同端口需要不同电压连接外部资源 → FPGA 引脚必须按 Bank（每 Bank 40 引脚，共享同一电压）配置
  • 没有完全自由的引脚布局：分区工具应在分区时提供电压区域指导并警告不正确的 IO 放置

8.2.5 分配高互连块

第一步最关键——从与其他块共享最多互联的块开始。

Certify 的互连矩阵（图106）：一眼看出 b2v_inst3 有最多顶层 IO 连接（106 条），b2v_inst1 和 b2v_inst2 彼此有 82 条互连但与其他块几乎无连接 → 将 1 和 2 放在同一 FPGA，3 放在另一 FPGA。

**高互

连块无法放同一 FPGA 时的策略**：向下一级层级寻找连接较少的块 → 提取到不同 FPGA → 仍会增加 IO，但比上一级放入不同 FPGA 的增加量少。如果一直往下找 → 越来越细的粒度 → 分区对设计迭代越来越脆弱（优化导致重命名等）→ 不如回到顶层换一个完全不同的粗分区策略。

“尝试不同的起点并对分配做部分完成，以’感受’适配到 FPGA 和互联中的难易度。如果一个分区方向很快变得难以平衡，它也不太可能最终令人满意。”

Certify 的"温度计"指示器（图105）：每颗 FPGA 三个字段——逻辑资源使用率 + IO 使用率。分配块时实时上下变动。

8.2.6 分配最大块

  • 从最大的块开始 → 留小块最后填补
  • 平衡资源利用率 → 保持在 <70%
  • 如果某块 RTL 会频繁更新或插桩 → 保持其 FPGA 的低利用率 → 加速设计迭代

8.2.7 分配剩余块 → 8.2.8 复制以降低 IO → 8.2.9 复用超标互联

复制（Replication）：当一个块需要同时在两个不同 FPGA 中驱动宽总线 → 在每个 FPGA 中各放一份复制 → 消除所有跨 FPGA 连接。特别适用于时钟和复位等芯片支持元素。

MUX 补充：当复制仍不够时，只能用时域复用——在一条物理走线上传输多路信号。详见 8.6。

8.2.10 分配走线

重要概念纠正：不是"给 FPGA 分配信号到引脚"，而是**“给信号分配走线”**——一条走线可能经过多颗 FPGA 的引脚，所有引脚同时被固定。

推荐分配顺序：

  1. 时钟、复位和其他全局输入
  2. 测试点或探针输出（匹配板上连接）
  3. 连接专用资源或连接器的顶层端口
  4. 其余跨器件信号（注意电压电平要求）

三优先原则：

  1. 有完全相同端点的走线 → 首选
  2. 端点匹配 + 最少冗余端点 → 冗余端点的 FPGA 引脚被浪费
  3. 无满足 1 或 2 的走线 → 将信号拆分到两条走线（源 FPGA 多用一根输出引脚，图112）

自动化走线分配：EDA 工具可遵循相同规则自动完成。半自动化：自己做关键的连接 + 工具补完不关键的。命名约定：信号名 + 资源引脚名使用相同命名约定 → 工具识别 addr1 信号连接 ram addr1 引脚。

8.2.10.2 无 EDA 工具时的替代方案（“电子表格方法”）

某团队的多项目经验：从虚拟综合的 log 提取信号名 → 导入电子表格 → 文本搜索匹配 → 手工分配到 FPGA 引脚 → 生成 UCF。极少见，不推荐——不仅需开发和维护脚本/电子表格，且引入额外错误机会。“Make vs Buy"决策中绝大多数人选择商业工具。

8.2.11 迭代改进

  • 多人并行尝试不同的初始分区→ 起始点对最终结果影响巨大
  • 与 SoC 后端布局团队讨论分区策略 → 他们可能分享降低路由拥塞的思路
  • 非显而易见的特殊策略：按时钟域分区（特定时钟驱动的所有逻辑放入同一 FPGA）→ 关键路径完全在 FPGA 内部 → 更高性能
  • 实验室中 bring-up 当前版本的同时，并行探索改进分区

8.3 自动化分区

自动分区工具的六个辅助任务：

  1. 固定需要连接外部资源的引脚组（否则分区器可能把它们分散到所有 FPGA）
  2. 约束每器件的资源使用率：50-70%
  3. 为黑盒填入尺寸信息（否则自动分区器无法为它们预留空间；黑盒不能拆分）
  4. 手动分配时钟和复位→ 在所有 FPGA 中复制
  5. 为峰值性能手动分组关键块
  6. 允许自动分区器执行复用

“自动分区工具不能替代有经验的原型工程师找到最优解，但它们可以尝试大量策略直到找到可行的。最佳状态：人做初始分配，机器完成剩余。”

针对 FPGA 原型（高性能是目标），完全自动的分区不现实——“人类始终需要也值得参与其中”。

8.4 改善原型性能

核心问题：跨 FPGA 信号路径时间通常 > FPGA 内部路径时间 → 跨 FPGA 时序是系统时钟频率的限制因素。

时序预算（Time Budgeting）：所有 FPGA IO 引脚的边界条件已知（板走线性能、确切路径、源 FPGA 中驱动逻辑）→ 可自动转换为每个 FPGA 的 IO 时序约束。

8.4.1 顺序边界（Sequential Boundaries）的时序预算

IO FF 的关键价值：

  • 每个 FPGA IO Pad 内嵌多个 FF——“免费"可用
  • SoC 顶层约束自动适用于所有 FF → 映射到 IO FF 后简化了各 FPGA 的隔离约束
  • 时钟到输出延迟 + 目标建立时间 → 最小化

但不允许随意插入流水线 FF：插入 FF 改变系统级调度 → 必须与原始设计师确认 → 可能需要在其他地方插入补偿 FF → 建议重新运行系统级仿真。

8.4.2 组合边界（Combinatorial Boundaries）的时序预算

为什么默认约束不对：P&R 工具默认假设整个时钟周期都可用——但实际上信号从另一 FPGA 的内部 FF 出发，经历了 FPGA 内布线 + 输出 Pad + 板走线 → 到达时只剩小于时钟周期的时间完成输入 Pad 到内部 FF 的路径。

Certify 的时序预算算法（图113）：

  • Quick-Pass 综合估算各 FPGA 内路径占总路径的比例
  • 如 FPGA A 内部 40ns（占 100ns 总约束的 40%），FPGA B 内部 10ns → A 获得更宽松的 IO 时序约束
  • 多 FPGA 边界 + 不同时钟域也被纳入计算
  • 为每条多 FPGA 路径生成 slack 值

8.5 跨多 FPGA 的设计同步

原型中时钟、复位和启动必须表现得好像"硬件的跨 FPGA 边界不存在”。

8.5.1 多 FPGA 时钟同步

两个潜在问题：

  1. 时钟偏斜和不确定性：板级 PLL → 各 FPGA 时钟输入的走线和缓冲器引入偏斜 → 可能导致相关时钟间短路径上的保持时间违规。解决：用 FPGA 内部 MCMM 重新生成时钟，后端工具理解 MCMM 的偏斜和不确定性。

  2. 时钟同步：当相关时钟在各 FPGA 本地重新生成 → 分频时钟（如 divide-by-N）可能因不同复位或初始条件而失步。解决：确保所有分频器在同一时钟沿释放（见下一节的全局复位机制）。

8.6-8.7 信号复用（TDM）

8.6.1 TDM 的基本要求和约束

核心要求：在一个设计时钟周期内传输所有 IO 数据值到另一个 FPGA。

4:1 MUX 的简单推演：设计时钟 20MHz → 传输时钟 ≥ 80MHz。但实际上要 >80MHz——因为需要满足传输时钟上的 setup/hold 和数据被设计时钟锁存时的时序余量。

在大多数使用复用的情况下，复用降低了设计的整体速度，且往往是系统速度的主导限制因素。

Certify 的两种复用方案：CPM（Certify Pin Multiplier）+ HSTDM（High Speed TDM）。对应同步/异步两种模式。

8.7.1 基于 MUX 的方案

简单 MUX 方案（图119）：MUX + 发送 FF (FFMO) → 板走线 → 接收 FF (FFMI) → DMUX → 捕获 FF (FFMC) → 设计 FF (FFR)。

**候选

信号选择准则**：直接由 FF 驱动的信号最优（可映射到 IO FF → 时钟周期上最大比例可用于传播）。正 slack 良好的路径才适合复用。

8.7.3 基于移位寄存器的方案（图120）

数据并行

加载到移位寄存器 → 传输时钟移位输出 → 接收端移位输入 → 并行输出。优势：无额外组合延迟 → 全周期用于传输 → 特别适合较长飞行时间的板子。风险：前一个采样被接收之前不能发送新数据 → 传输线反射或不连续可能导致不可靠。

8.7.5 LVDS + IOSERDES 方案（HSTDM）

源同步接口：时钟与数据同一源并行发送 → 时钟具有与数据相同的 off-chip/on-chip 偏斜 → 匹配延迟走线的板上时序变得简单。

差分信令的代价与收益：

  • 代价：每路串行数据需要 2 根引脚（差分对）→ 8:2 而非单端的 8:1
  • 收益：LVDS 速率可达 1GHz → 128:2 的复用比 → 远比单端 8:1 的方案更大的有效互联带宽

8.8 复用方案的时序约束

必须添加四类约束：

  1. 传输时钟的时钟周期约束——基于 IO 延迟 + 板延迟 + 容限计算
  2. 设计时钟到传输时钟的最大延迟约束——确保复用连接在定义延迟内完成
  3. 时钟到时钟比率——通知 P&R 工具 mux 时钟与设计时钟的关系
  4. 如果这些约束缺失 → 功能完整性的风险

8.9 分区与重连：总结

“有一个老笑话：迷路的司机问路人怎么去伦敦，路人回答说’哦，你不会想从这里出发的！’“这个故事是说——有些 SoC 设计本身就是最难分区的起点。我们不想从复杂、高度互连、多宽总线的设计开始。第9章（Design-for-Prototyping）将讨论如何从更好的起点出发——让原型工程师参与上游决策，打破第三定律的源头。

个人总结

第8章在全书中的位置非常特别——它既是第7章（设计移植）的延续（已经 FPGA-ready 的设计，现在要分到多颗 FPGA 中），也是所有后续章节（Bring-Up、调试、现场部署）的前提。

本章最重要的思想跃迁是从"把设计放进 FPGA"到"把设计分配到一个互联受约束的物理系统中”。Board Description File 这个看似纯粹格式性的东西，实际上代表了这个跃迁——它把抽象的"更多 FPGA"变成了具体的"AB 有 44 条走线、AC 有 29 条、AD 有 62 条"的物理约束。从此分区的每一步都有硬性数字约束——不是"这样分好不好”，而是"这样分有没有足够的物理连线”。

复用方案的量化时序计算（Table 20）是全书最具工程学科气质的部分。4:1 MUX 实际频率是 14.28MHz 而非 25MHz——这个差异来自对四个捕获 FF 的顺序填充、对设计时钟到传输时钟延迟（最大 1 传输周期）的纳入、以及对 mux/dmux 流水线级的完整建模。忽略这些细节就意味着一块板子标称 25MHz 但只能稳定跑 14MHz——在真实项目中，这就是两周的调试时间。

实践思考

  • 不要用直觉估算 MUX 后的设计频率。画一个像 Table 20 一样的逐周期数据流表——它能暴露 MUX 流水线的真实延迟，往往比直觉大得多
  • 复制是 IO 优化的"免费午餐"——只要一个块在两个 FPGA 都需要，复制它完全消除了该块所有跨 FPGA 互联。在分区早期就应主动寻找复制机会
  • Board Description 应从布局工具自动导出而非手工编写。一个引脚位置错误可能需要数天才能定位；在大规模原型中上千引脚的手工编写不可能可靠
  • 三 FF 复位流水线是解决跨 FPGA 同步复位的"银弹"——代价仅仅是两三个时钟周期的复位延迟（对全局复位信号完全可以忽略），但必须记得 综合器的 SRL 优化需要被显式禁止
  • 异步 vs 同步 MUX：7:1(同步) vs 9:1(异步) → 只有差别 2 个传输周期的差异 → 如果板级有现成的高质量时钟分布方案，选同步可获得更高性能；如果板级时钟分布不成熟或有意用独立的低频源，选异步获得更大的自由度
  • HSTDM 需要 Link Training → 如果自己实现（而非使用商业分区工具的 IP），训练状态机 + IDELAY 的自动校准是一个显著的设计工作量
  • 不要等到分区完成后才考虑性能——从一开始就将 IO FF 的使用、时序约束和复用方案作为分区决策的一部分来评估
